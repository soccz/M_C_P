# Ch.5 클로드의 기억

> **이 챕터를 마치면**: 클로드가 왜 매번 같은 말을 잊는지 이해하고, CLAUDE.md와 Memory 시스템을 설정해서 "한 번 말하면 영원히 기억하는" 환경을 만들 수 있다. .claude/ 폴더 전체 구조를 파악한다.

---

## 배경: "아까 말했잖아"의 좌절

클로드를 처음 쓰는 사람이 가장 답답해하는 순간이 있다.

```
You: 이 프로젝트는 React + TypeScript야. 테스트는 Vitest를 써.
Claude: 네, 알겠습니다.

... (30분 후) ...

You: 테스트 추가해줘.
Claude: Jest로 테스트를 작성하겠습니다.
You: ???  아까 Vitest라고 했잖아!
```

이건 클로드가 멍청해서가 아니다. **기억의 구조가 다르기 때문이다.**

사람은 며칠 전 대화도 기억한다. 클로드는 **이번 대화 안에서만** 기억하고, 그것도 대화가 길어지면 앞부분을 잊기 시작한다. 마치 칠판에 글을 쓰는데 칠판 크기가 정해져 있어서, 새 글을 쓰면 앞의 글이 지워지는 것과 같다.

이 "칠판"의 크기를 **컨텍스트 윈도우(context window)**라고 한다. 아무리 큰 모델이라도 한계가 있다.

> **💭 생각의 흐름**
>
> **문제** — 매번 "이 프로젝트는 React야, TypeScript야, Vitest 써" 반복하는 게 지겹다
> **질문** — "한 번만 말하면 기억하게 할 수 없나?"
> **발견** — CLAUDE.md에 적으면 매 대화 시작 시 자동으로 읽는다
> **결과** — 프로젝트 규칙을 CLAUDE.md에 적은 뒤, 두 번 다시 반복할 일이 없어졌다

---

## 기억의 3가지 레이어

클로드의 기억은 세 단계로 나뉜다:

### 1. 세션 기억 (대화 기억) — 이번 대화만

지금 이 대화에서 주고받은 모든 내용. `/clear`를 하거나 대화를 끝내면 사라진다.

- **장점**: 별도 설정 없이 바로 사용
- **한계**: 대화가 길어지면 앞부분을 잊음 (컨텍스트 윈도우 한계)
- **관리법**: `/compact`로 압축, 중요한 건 다른 레이어로 옮기기

### 2. 파일 기억 — CLAUDE.md와 rules/

파일에 적어놓은 규칙과 맥락. 매 대화가 시작될 때 자동으로 읽힌다.

- **CLAUDE.md**: 프로젝트의 대헌장. 항상 읽힘
- **.claude/rules/**: 경로별 세부 규칙. 해당 파일 작업할 때 읽힘
- **장점**: 영구적, 모든 세션에서 적용
- **한계**: 너무 길면 컨텍스트를 낭비

### 3. Memory 시스템 — 대화 간 기억

클로드가 대화 사이에 기억하는 것. `~/.claude/` 안에 저장된다.

- **user 메모리**: 사용자의 역할, 선호도
- **project 메모리**: 프로젝트 진행 상황
- **feedback 메모리**: "이렇게 해라/하지 마라" 피드백
- **reference 메모리**: 외부 자료 위치
- **장점**: 대화를 끝내도 유지됨
- **한계**: 자동 저장이 아니라 클로드가 판단해서 저장

```
기억의 수명:

세션 기억     ████░░░░░░  이번 대화만
파일 기억     ██████████  영구 (파일 삭제 전까지)
Memory 시스템 ████████░░  영구 (직접 삭제 전까지)
```

## CLAUDE.md — 프로젝트의 대헌장

CLAUDE.md는 **프로젝트 루트에 놓는 규칙 파일**이다. 클로드가 대화를 시작할 때 가장 먼저 읽는다.

### 왜 중요한가

CLAUDE.md가 없으면 클로드는 매번 "이 프로젝트가 뭔지" 몰라서 물어보거나 추측한다. CLAUDE.md가 있으면 "아, 이 프로젝트는 이런 거구나. 이 규칙을 따르면 되겠다"고 바로 시작한다.

마치 새 직원에게 주는 **업무 매뉴얼**과 같다. 매뉴얼 없이 입사하면 매번 물어봐야 한다. 매뉴얼이 있으면 알아서 움직인다.

### 최소 CLAUDE.md

처음 만들 때는 이것만 있어도 충분하다:

```markdown
# 프로젝트 작업 계약서

## 명령
- install: npm install
- dev: npm run dev
- test: npm test
- lint: npm run lint
- build: npm run build

## 경계
- api: src/api/
- ui: src/components/
- shared types: src/types/

## 규칙
- '.env' 파일은 수정하지 않는다
- 파일 삭제 전에는 먼저 묻는다
- 변경은 작고 검토 가능하게 유지한다
- 테스트가 통과하기 전에는 끝났다고 하지 않는다

## 검증
- unit: npm test
- integration: npm run test:e2e
- visual: npm run storybook
```

### CLAUDE.md에 넣으면 좋은 것

| 항목 | 예시 |
|------|------|
| 프로젝트 설명 | "React + TypeScript 기반 할일 관리 앱" |
| 실행 명령어 | dev, test, lint, build |
| 폴더 구조 설명 | "api/는 백엔드, components/는 프론트" |
| 코딩 규칙 | "세미콜론 없이, 탭 대신 스페이스 2칸" |
| 금지 사항 | ".env 수정 금지, node_modules 건드리지 말 것" |
| 검증 방법 | "npm test 통과해야 완료" |

### CLAUDE.md에 넣지 말아야 할 것

| 넣지 말 것 | 이유 |
|----------|------|
| 전체 코드 복사 | 컨텍스트 낭비. 클로드가 직접 파일을 읽을 수 있다 |
| API 키, 비밀번호 | 보안 위험 |
| 50줄 넘는 설명 | 너무 길면 오히려 집중력 분산. 핵심만 |
| 다른 프로젝트 규칙 | 이 프로젝트에 해당하는 것만 |

### `/init`으로 자동 생성

처음이라면 Claude Code에서:

```
/init
```

클로드가 현재 프로젝트를 분석해서 CLAUDE.md 초안을 자동으로 만들어준다. 이걸 수정해서 쓰면 된다.

## .claude/ 폴더 전체 구조

CLAUDE.md만으로도 충분히 시작할 수 있지만, 프로젝트가 커지면 더 많은 설정이 필요하다. 이때 `.claude/` 폴더가 등장한다.

```
my-project/
├── CLAUDE.md                 ← 프로젝트 대헌장 (항상 읽힘)
├── .claude/
│   ├── settings.json         ← 권한과 차단 경계 (Ch.6)
│   ├── settings.local.json   ← 개인 설정 (git에 안 올림)
│   ├── rules/                ← 경로별 세부 규칙 (Ch.7)
│   │   ├── api.md
│   │   ├── testing.md
│   │   └── frontend.md
│   │                         ← settings.json 안에 hooks 설정도 포함 (Ch.8)
│   └── skills/               ← 프로젝트 전용 스킬 (Ch.12)
├── tasks/                    ← 작업 관리 파일들 (Ch.17)
│   ├── plan.md
│   ├── verify.md
│   ├── decision-log.md
│   └── handoff.md
└── src/                      ← 실제 코드
```

### 각 파일의 역할

| 파일 | 한 줄 설명 | 언제 읽히나 |
|------|----------|----------|
| **CLAUDE.md** | 프로젝트 운영 계약 | **항상** (매 세션 시작) |
| **settings.json** | 허용/금지/차단 경계 | 항상 |
| **settings.local.json** | 개인 환경 설정 (git 제외) | 항상 |
| **rules/*.md** | 경로별 세부 규칙 | 해당 경로 작업 시 |
| **hooks (settings.json 내)** | 자동 실행 규칙 | 이벤트 발생 시 |
| **plan.md** | 앞으로 할 일 | 작업 시작 시 |
| **verify.md** | 무엇이 통과해야 완료인지 | 검증 시 |
| **decision-log.md** | 왜 이 경로를 택했는지 | 판단 필요 시 |
| **handoff.md** | 지금 어디까지 했는지 | 세션 전환 시 |

지금 당장 알아야 할 것은 **CLAUDE.md**와 **settings.json** 두 개다. 나머지는 해당 챕터에서 배운다.

> **💭 생각의 흐름**
>
> **문제** — CLAUDE.md에 모든 규칙을 다 넣었더니 50줄이 넘어감
> **질문** — "이걸 다 읽으면 컨텍스트 낭비 아닌가?"
> **발견** — API 규칙은 api.md로, 테스트 규칙은 testing.md로 분리하면 해당 작업할 때만 읽힌다
> **결과** — CLAUDE.md는 10줄 이내 공통 규칙만, 나머지는 rules/로 분리 → Ch.7에서 상세

## Memory 시스템

CLAUDE.md는 **프로젝트 단위** 기억이다. 모든 프로젝트에서 적용되는 **개인 단위** 기억은 Memory 시스템이 담당한다.

### Memory가 저장되는 곳

```
~/.claude/
├── projects/
│   └── [프로젝트경로]/
│       └── memory/
│           ├── MEMORY.md         ← 인덱스
│           ├── user_role.md      ← 사용자 정보
│           ├── feedback_xxx.md   ← 피드백
│           └── project_xxx.md    ← 프로젝트 상태
└── ...
```

### Memory 4가지 타입

| 타입 | 저장하는 것 | 예시 |
|------|----------|------|
| **user** | 사용자 역할, 선호도, 지식 | "데이터 사이언티스트, 시각화 선호" |
| **feedback** | 행동 교정 피드백 | "테스트에서 mock 쓰지 마" |
| **project** | 프로젝트 진행 상황 | "인증 모듈 리팩터링 중, 목요일까지" |
| **reference** | 외부 자료 위치 | "버그 트래커는 Linear의 INGEST 프로젝트" |

### Memory는 어떻게 저장되나

클로드가 **자동으로 판단**해서 저장한다. 또는 직접 요청할 수 있다:

```
"이거 기억해줘: 나는 Python보다 TypeScript를 선호해."
```

확인하려면:

```
/memory
```

### Memory vs CLAUDE.md

| | Memory | CLAUDE.md |
|---|--------|-----------|
| **범위** | 글로벌 (모든 프로젝트) | 이 프로젝트만 |
| **내용** | 개인 선호, 피드백 | 프로젝트 규칙 |
| **저장** | 클로드가 판단 또는 요청 | 사용자가 직접 파일 작성 |
| **읽히는 때** | 관련 있다고 판단될 때 | 매 세션 시작 시 항상 |

**핵심**: 개인적인 건 Memory에, 프로젝트 규칙은 CLAUDE.md에.

## 🐍 문자열과 f-string — 프롬프트도 결국 텍스트 조합

CLAUDE.md도, Memory도, 결국 **텍스트 파일**이다. 클로드가 읽는 모든 것은 텍스트. 프롬프트도 텍스트. 프로그래밍에서 텍스트를 다루는 법을 알면 이 구조가 더 잘 보인다.

### 문자열 기초

```python
# 따옴표로 감싸면 문자열
greeting = "안녕하세요"
name = '클로드'

# 여러 줄은 세 따옴표
bio = """이름: 클로드
역할: AI 어시스턴트
특기: 코드 작성"""
```

### 문자열 연결

```python
first = "안녕"
second = "하세요"
result = first + second    # "안녕하세요"
```

### f-string — 가장 많이 쓰는 문자열 조합법

```python
name = "클로드"
age = 2
model = "Opus"

# f-string: f"..." 안에 {변수}를 넣으면 자동 치환
message = f"{name}은 {age}살이고 모델은 {model}입니다."
print(message)
# → 클로드는 2살이고 모델은 Opus입니다.
```

### 실전: CLAUDE.md를 파이썬으로 생성하기

> 아래 코드에서 `with open(...) as f:` 부분은 "파일을 열고 쓰는" 문법이다. 지금은 따라만 해도 충분하다. 파일 입출력은 나중에 Ch.23에서 자세히 배운다.

```python
# claude_md_generator.py
project_name = input("프로젝트 이름: ")
language = input("주 언어: ")
test_command = input("테스트 명령어: ")

claude_md = f"""# {project_name}

## 명령
- test: {test_command}

## 규칙
- 주 언어는 {language}
- 테스트 통과 전에는 끝났다고 하지 않는다
- .env 파일 수정 금지
"""

# 파일로 저장
with open("CLAUDE.md", "w") as f:
    f.write(claude_md)

print("CLAUDE.md가 생성되었습니다!")
```

이건 실용적인 예시다. 프로젝트마다 CLAUDE.md를 만들 때 이런 스크립트를 활용할 수 있다. 물론 Claude Code에게 "이 프로젝트의 CLAUDE.md를 만들어줘"라고 시키는 게 더 빠르다.

### 문자열 메서드

```python
text = "  Hello, Claude!  "

text.strip()          # "Hello, Claude!" (양쪽 공백 제거)
text.lower()          # "  hello, claude!  "
text.upper()          # "  HELLO, CLAUDE!  "
text.replace("Claude", "World")  # "  Hello, World!  "
text.split(",")       # ["  Hello", " Claude!  "]

# 확인
"Claude" in text      # True
text.startswith("  H") # True
len(text)             # 20
```

## 잘 안 된다면

| 증상 | 원인 | 해결 |
|------|------|------|
| CLAUDE.md가 적용 안 됨 | 파일 위치가 프로젝트 루트가 아님 | 프로젝트 최상위 폴더에 놓기 |
| 규칙을 무시함 | CLAUDE.md가 너무 길어서 집중력 분산 | 핵심만 10줄 이내로, 나머지는 rules/로 |
| Memory가 저장 안 됨 | 클로드가 저장할 필요 없다고 판단 | 명시적으로 "이거 기억해줘" 요청 |
| 이전 대화 맥락이 없음 | 세션이 끊김 | handoff.md에 상태 저장 (Ch.17) |

## 왜 이렇게 동작하는가

클로드의 기억 구조는 **인간의 기억과 비슷한 설계**를 따른다:

- **세션 기억** = 단기 기억: 지금 하고 있는 대화. 용량 제한 있음
- **파일 기억** = 업무 매뉴얼: 책상 위에 항상 펼쳐놓은 규칙
- **Memory** = 장기 기억: 오래된 대화에서 배운 것들

사람도 중요한 건 메모장에 적는다. 클로드도 마찬가지다. 중요한 규칙은 CLAUDE.md(메모장)에 적어야 잊지 않는다. 머릿속(세션 기억)만 믿으면 까먹는다.

**컨텍스트 윈도우의 크기:**
현재 Claude의 컨텍스트 윈도우는 약 200K 토큰 (opus 기준). 대략 소설 한 권 분량이다. 많아 보이지만, CLAUDE.md + rules + 대화 내용 + 도구 출력이 합쳐지면 금방 찬다. 그래서 **무엇을 넣고 무엇을 빼는가**가 중요하다 — 이게 Ch.9(Context Engineering)의 핵심이다.
