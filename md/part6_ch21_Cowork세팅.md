# Ch.21 Cowork 세팅

> **이 챕터를 마치면**: Cowork의 7일 세팅 로드맵을 따라 실전 환경을 구성할 수 있다. about-me.md, working-rules.md, brief.md 세 가지 시작 템플릿을 작성할 수 있고, 글로벌 지침과 개별 브리프의 역할 차이를 설명할 수 있다.

---

## 배경: 위임은 준비가 전부다

Ch.20에서 Cowork이 "위임 시스템"이라는 것을 배웠다. 하지만 위임은 **준비 없이** 되지 않는다.

상사에게 보고서를 부탁받았을 때를 떠올려보자:
- "대충 알아서 해줘" → 높은 확률로 다시 작업
- "이 자료 보고, 이 형식으로, 금요일까지" → 한 번에 통과

클로드도 마찬가지다. Cowork에서 좋은 결과를 받으려면 **클로드가 읽을 수 있는 형태로 맥락을 준비**해야 한다. 이 챕터는 그 준비를 7일에 걸쳐 하나씩 쌓는 로드맵이다.

> **💭 생각의 흐름**
>
> **문제** — Cowork에서 "경쟁사 분석해줘"라고 했는데, 결과가 너무 일반적이었다. 우리 업계 용어도 모르고, 내가 원하는 형식도 아니었다
> **질문** — "어떻게 하면 내 의도대로 결과가 나올까?"
> **시도 1** — 프롬프트를 더 길게 썼다 → 매번 같은 내용을 반복 입력
> **시도 2** — about-me.md에 내 역할과 선호를 적어두었다 → 결과가 훨씬 나아짐
> **발견** — 매번 프롬프트에 쓰는 대신, **파일로 한 번 적어두면** 모든 세션에 자동 적용된다
> **결론** — Cowork 세팅 = **"나를 설명하는 파일"을 한 번 잘 만드는 것**

---

## 7일 세팅 로드맵

한 번에 완벽하게 만들려 하지 마라. 7일에 걸쳐 **조금씩 다듬는 것**이 핵심이다.

### Day 1: 폴더와 파일 4개 만든다

가장 먼저 할 일은 폴더 구조를 만드는 것이다.

```
CLAUDE-COWORK/
├── ABOUT-ME/
│   ├── about-me.md       ← 오늘 만든다
│   └── working-rules.md  ← 오늘 만든다
├── PROJECTS/
├── TEMPLATES/
└── CLAUDE-OUTPUTS/
```

**about-me.md** — 가장 중요한 파일. 다음 섹션의 템플릿을 참고해서 작성한다.

**working-rules.md** — 빈 파일로 시작해도 된다. Day 3에서 채운다.

그 다음 Claude Desktop 앱에서:
1. **Cowork 탭** 클릭
2. CLAUDE-COWORK/ 폴더에 **접근 권한 부여**
3. **Settings → Cowork → Edit (Global Instructions)** 에서 기본 지침 입력

Global Instructions에 넣을 최소 내용:

```
모든 작업 시작 전에 ABOUT-ME/ 폴더의 파일을 먼저 읽어주세요.
결과물은 CLAUDE-OUTPUTS/ 폴더에 저장해주세요.
ABOUT-ME/, PROJECTS/, TEMPLATES/ 폴더의 파일은 수정하거나 삭제하지 마세요.
```

### Day 2: 작은 과제를 한 번 맡겨 본다

세팅만 하고 안 써보면 의미가 없다. **작고 구체적인 작업** 하나를 맡겨본다.

좋은 첫 과제 예시:
- "이 PDF를 읽고 핵심 5가지를 정리해줘"
- "이 폴더의 텍스트 파일들을 읽고 공통점을 요약해줘"
- "이 데이터를 표로 정리해줘"

**나쁜 첫 과제**: "우리 사업 전략을 수립해줘" — 너무 넓고 맥락이 부족하다.

첫 결과를 받으면 두 가지를 확인한다:
1. **형식이 내 기대와 맞는가?** → 안 맞으면 Day 3에서 working-rules.md에 추가
2. **톤이 내 기대와 맞는가?** → 안 맞으면 about-me.md에 선호 스타일 추가

### Day 3: working-rules.md를 수정한다

Day 2의 결과를 보고 **규칙을 추가**한다.

```markdown
# working-rules.md (Day 3 버전)

## 형식 규칙
- 글머리 기호 중심으로 작성 (긴 문단 지양)
- 표를 적극 활용
- 파일명에 날짜 포함: YYYY-MM-DD_제목.md

## 톤 규칙
- 비즈니스 톤 (캐주얼 금지)
- 수치에는 출처 표기
- 불확실한 정보는 "확인 필요"로 표시

## 금지사항
- ABOUT-ME, PROJECTS, TEMPLATES 폴더 파일 수정 금지
- 추측 기반 결론 금지
```

이것은 Claude Code에서 Ch.7의 rules/ 파일을 작성하는 것과 같은 과정이다. 결과를 보고 규칙을 다듬는 **피드백 루프**다.

### Day 4: 템플릿을 만든다

반복되는 작업이 있다면, 그 형식을 **템플릿**으로 추출한다.

```markdown
# TEMPLATES/weekly-brief-template.md

## 주간 브리프 — {{날짜}}

### 이번 주 핵심 (3줄)
1.
2.
3.

### 상세

#### 카테고리 A
- 주요 변화:
- 수치:
- 우리에게 의미:

#### 카테고리 B
...

### 다음 주 주시 포인트
-
```

템플릿이 있으면 매번 형식을 설명하지 않아도 된다. "weekly-brief-template.md 형식으로 만들어줘"라고만 하면 된다. 이것이 Ch.9에서 배운 **Progressive Disclosure** — 필요한 정보를 파일로 분리해두는 것.

### Day 5: 프로젝트 폴더를 실제 업무 기준으로 나눈다

```
PROJECTS/
├── 경쟁사-분석/
│   ├── brief.md             ← 이 프로젝트의 목표와 범위
│   ├── 경쟁사-목록.md        ← 참고 자료
│   └── 이전-분석-결과.md     ← 과거 결과 (참고용)
│
├── 주간-보고서/
│   ├── brief.md
│   └── KPI-기준.md
│
└── 고객-피드백-정리/
    ├── brief.md
    └── 분류-기준.md
```

각 프로젝트 폴더에 **brief.md**를 넣는다. brief.md 템플릿은 이 챕터 뒤에서 다룬다.

### Day 6: output 파일 규칙을 정한다

CLAUDE-OUTPUTS/ 폴더가 쌓이면 금방 지저분해진다. **파일명 규칙**을 정한다.

```
파일명 형식: YYYY-MM-DD_프로젝트명_내용.확장자

예시:
  2026-04-14_경쟁사분석_주간브리프.md
  2026-04-14_고객피드백_4월정리.md
  2026-04-15_주간보고서_15주차.md
```

이 규칙을 **working-rules.md**에 추가한다:

```markdown
## 출력 파일 규칙
- 파일명: YYYY-MM-DD_프로젝트명_내용.확장자
- 기존 파일을 덮어쓰지 말고, 새 버전으로 생성 (v1, v2)
- 프로젝트별 하위 폴더 사용 가능
```

### Day 7: 반복 작업 하나를 Scheduled Task로 올린다

7일째에 첫 자동화를 설정한다. Ch.23에서 자세히 다루지만, 기본 설정은 간단하다.

Claude Desktop 앱에서:
1. 왼쪽 사이드바의 **"Scheduled"** 클릭
2. **"+ New task"** 선택
3. 태스크 이름, 프롬프트, 빈도(매일/매주), 작업 폴더 설정
4. **"Save"**

첫 Scheduled Task로 좋은 예시:
- 매주 월요일 아침: "PROJECTS/경쟁사-분석/ 폴더를 보고 주간 브리프를 만들어줘"
- 매일 아침: "오늘 할 일을 정리해줘"

### 7일 요약

```
Day 1: 폴더 4개 + about-me.md 작성 + Global Instructions
Day 2: 작은 과제 하나 맡겨보기
Day 3: 결과 보고 working-rules.md 다듬기
Day 4: 반복 형식을 TEMPLATES/에 추출
Day 5: PROJECTS/ 폴더를 실제 업무 기준으로 구성
Day 6: CLAUDE-OUTPUTS/ 파일명 규칙 정하기
Day 7: 첫 Scheduled Task 설정

핵심: 한 번에 완벽하게 ❌ → 조금씩 다듬기 ✅
```

이것은 Claude Code에서 CLAUDE.md를 처음 쓰고, 작업하면서 점점 다듬어가는 과정과 같다.

## Cowork 글로벌 지침 예시

Global Instructions(글로벌 지침)는 **모든 Cowork 세션에 자동 적용**되는 규칙이다. Settings → Cowork → Edit에서 설정한다.

### 최소 글로벌 지침

```
# 기본 행동
1. 모든 작업 시작 전에 ABOUT-ME/ 폴더의 파일을 읽는다.
2. 프로젝트 관련 작업이면 해당 PROJECTS/ 하위 폴더의 brief.md를 읽는다.
3. 결과물은 CLAUDE-OUTPUTS/ 폴더에 저장한다.

# 폴더 규칙
- ABOUT-ME/, PROJECTS/, TEMPLATES/ — 읽기 전용 (수정·삭제 금지)
- CLAUDE-OUTPUTS/ — 쓰기 전용

# 실행 규칙
- 불확실하면 실행 전에 질문한다.
- 파일명 형식: YYYY-MM-DD_프로젝트명_내용.확장자
```

### 확장 글로벌 지침

실무에서 사용하는 더 구체적인 예시:

```
# 기본 행동
1. 모든 작업 시작 전에 ABOUT-ME/의 파일을 전부 읽는다.
2. 프로젝트 작업이면 해당 PROJECTS/ 폴더의 brief.md를 읽고, 
   참고 자료가 있으면 함께 읽는다.
3. 템플릿이 있으면 TEMPLATES/에서 해당 형식을 따른다.
4. 결과물은 CLAUDE-OUTPUTS/에 저장한다.

# 언어와 톤
- 한국어로 작성
- 비즈니스 톤 (캐주얼 금지)
- 수치에는 출처 표기
- 불확실한 정보는 [확인 필요]로 표시

# 폴더 규칙
- 읽기 전용: ABOUT-ME/, PROJECTS/, TEMPLATES/
- 쓰기: CLAUDE-OUTPUTS/ 만

# 실행 규칙
- 불확실하면 실행 전에 질문한다
- 계획을 먼저 보여주고 승인 후 실행한다
- 기존 파일을 덮어쓰지 않는다 (버전 증가: v1 → v2)
- 파일 삭제 절대 금지
```

### Claude Code의 CLAUDE.md와의 대응

```
Claude Code                    Cowork
──────────────────────────────────────────
CLAUDE.md 첫 줄                Global Instructions (Settings에서)
"이 프로젝트는 ~이고           about-me.md
 내 역할은 ~이다"              
rules/ 파일들                  working-rules.md
"테스트 후 커밋"같은 규칙      "불확실하면 질문" 같은 규칙
```

## 시작 템플릿 3종

### 1. about-me.md

```markdown
# about-me.md

## 나는 누구인가
- [직함/역할]: 예) 스타트업 마케팅 리드
- [담당]: 예) B2B SaaS 제품의 콘텐츠 마케팅
- [팀]: 예) 마케팅팀 3명, 주간 회의 월요일

## 현재 우선순위
1. [가장 중요한 업무]
2. [두 번째 업무]
3. [세 번째 업무]

## 클로드에게 바라는 것
- [언어]: 한국어
- [톤]: 비즈니스 (캐주얼 금지)
- [형식]: 글머리 기호 중심, 표 적극 활용
- [길이]: 간결하게 (불필요한 서론·결론 생략)

## 내가 자주 하는 실수
- [예: 마감을 잊어서 급하게 처리함 → 미리 알려줘]
- [예: 수치 검증을 빠뜨림 → 출처 반드시 포함]

## 이것만은 하지 마
- 추측으로 결론 내리기
- 이모지 사용
- "~할 수 있습니다" 같은 챗봇 말투
```

**작성 팁**: 직접 쓰기 어려우면, 클로드에게 "나를 인터뷰해줘"라고 요청하라. 질문에 답하면서 자연스럽게 about-me.md가 완성된다.

### 2. working-rules.md

```markdown
# working-rules.md

## 형식 규칙
- 글머리 기호 중심 (긴 문단 지양)
- 표를 적극 활용
- 제목은 명확하고 구체적으로
- 파일명: YYYY-MM-DD_프로젝트명_내용.확장자

## 실행 규칙
- 불확실하면 실행 전에 질문한다
- 계획을 먼저 보여주고 승인 후 실행한다
- 참고 자료가 있으면 반드시 읽고 시작한다

## 금지사항
- ABOUT-ME/, PROJECTS/, TEMPLATES/ 파일 수정·삭제 금지
- 추측 기반 결론 금지
- 기존 파일 덮어쓰기 금지 (버전 증가)

## 품질 기준
- 수치에는 출처 표기
- 불확실한 정보는 [확인 필요]로 표시
- 결과물에 "다음 단계 제안" 포함
```

### 3. project-brief.md (프로젝트 브리프)

실제 파일명은 **brief.md**로 줄여 쓰는 것이 일반적이다. 프로젝트 폴더 안에 넣으면 이름만으로 역할이 명확하기 때문이다.

```markdown
# brief.md — [프로젝트명]

## 목표
[이 프로젝트에서 달성하려는 것. 한 문장.]

## 배경
[왜 이 작업이 필요한지. 2~3줄.]

## 범위
- 포함: [포함할 것]
- 제외: [제외할 것]

## 입력 자료
- [이 폴더의 어떤 파일을 참고해야 하는가]
- [외부에서 가져올 데이터가 있는가]

## 출력 형식
- 형식: [md/표/pptx/...]
- 길이: [대략적인 분량]
- 템플릿: [TEMPLATES/ 폴더의 어떤 파일을 따를 것인가]

## 주의사항
- [특별히 조심할 것]
- [피해야 할 접근법]

## 완료 기준
[이 결과를 보면 "됐다"고 판단할 수 있는 기준]
```

이 brief.md는 Claude Code의 **plan.md**와 같은 역할이다. 목표·범위·완료 기준을 명확히 하면 "다시 해줘"를 줄일 수 있다.

## 글로벌 지침과 개별 브리프의 차이

### 왜 둘 다 필요한가

```
글로벌 지침 (Global Instructions + about-me.md + working-rules.md):
  → 모든 작업에 공통으로 적용
  → "나는 누구이고, 어떤 규칙으로 일하는가"
  → 한 번 쓰면 거의 안 바뀜

개별 브리프 (PROJECTS/X/brief.md):
  → 특정 프로젝트에만 적용
  → "이번 작업의 목표와 범위는 무엇인가"
  → 프로젝트마다 다름
```

이것은 Claude Code에서의 구분과 정확히 같다:

```
Claude Code:
  글로벌 CLAUDE.md (모든 프로젝트 공통)
  프로젝트 .claude/CLAUDE.md (프로젝트별)
  ↓
Cowork:
  Global Instructions + ABOUT-ME/ (모든 세션 공통)
  PROJECTS/X/brief.md (프로젝트별)
```

### 무엇을 어디에 넣는가

| 내용 | 어디에 | 이유 |
|---|---|---|
| 내 역할, 선호 스타일 | about-me.md | 모든 작업에 적용 |
| 형식 규칙, 금지사항 | working-rules.md | 모든 작업에 적용 |
| 폴더 접근 규칙 | Global Instructions | 최우선 적용 |
| 특정 프로젝트의 목표 | brief.md | 그 프로젝트만 |
| 특정 프로젝트의 참고 자료 | PROJECTS/X/ 폴더 | 그 프로젝트만 |
| 재사용 형식 틀 | TEMPLATES/ | 여러 프로젝트가 공유 |

### 흔한 실수

**실수 1: 모든 것을 글로벌 지침에 넣는다**

```
❌ Global Instructions에:
   "경쟁사 분석할 때는 CompanyA, CompanyB, CompanyC를 봐"
   "주간 보고서는 KPI 표를 포함해"
   "고객 피드백은 긍정/부정/중립으로 분류해"

→ 경쟁사 분석 안 할 때도 이 규칙이 읽힘 = 토큰 낭비
```

```
✅ 각 프로젝트 brief.md에:
   경쟁사-분석/brief.md → CompanyA, B, C 정보
   주간-보고서/brief.md → KPI 표 형식
   고객-피드백/brief.md → 분류 기준
```

**실수 2: about-me.md가 너무 길다**

```
❌ 3,000 단어의 자기소개
   → 토큰을 많이 소비하고, 핵심이 묻힘

✅ 500 단어 이내
   → 핵심만, 간결하게
   → Claude Code의 "CLAUDE.md는 500줄 이하" 원칙과 같다
```

**실수 3: 브리프 없이 프롬프트만으로 작업을 지시한다**

```
❌ "경쟁사 분석해줘"
   → 범위? 형식? 깊이? 모두 불명확

✅ "PROJECTS/경쟁사-분석/의 brief.md를 읽고 분석해줘"
   → 목표·범위·형식·주의사항이 파일에 다 있음
```

## 왜 이렇게 동작하는가

Cowork 세팅의 원리는 **영속화(persistence)**다.

매 세션 시작 시 같은 맥락을 반복 설명하는 것은 비효율적이다. 파일로 한 번 적어두면:
1. **반복 입력이 없어진다** — 매번 "나는 마케터이고..."라고 쓸 필요 없음
2. **일관성이 유지된다** — 어제와 오늘의 규칙이 같음
3. **점진적 개선이 가능하다** — 파일을 수정하면 다음 세션부터 적용

이것은 Ch.5에서 배운 **Memory 시스템**의 원리와 같다. Memory가 클로드 내부에 맥락을 저장한다면, Cowork의 파일은 **외부에** 맥락을 저장한다. 둘 다 "매번 반복하지 않기 위해 한 번 기록해둔다"는 같은 문제를 푼다.
