# Ch.11 Harness Engineering

> **이 챕터를 마치면**: 프롬프트와 컨텍스트를 넘어 "작업장 전체"를 설계하는 하네스 엔지니어링을 이해한다. 5가지 최소 정의를 직접 만들 수 있고, Claude Code 운영 7원칙을 실천할 수 있다.

---

## 배경: 프롬프트만으로는 부족한 이유

Ch.9에서 컨텍스트 엔지니어링을, Ch.10에서 프롬프트 엔지니어링을 배웠다. 이 둘을 잘 해도 여전히 문제가 생기는 경우가 있다.

예를 들어:
- 좋은 프롬프트, 좋은 컨텍스트로 작업을 시켰는데 → 클로드가 `.env`를 수정했다
- 코드를 잘 수정했는데 → 테스트를 안 돌리고 "완료"라고 했다
- 복잡한 리팩터링을 시켰는데 → 중간에 방향이 틀어져도 멈추지 않고 계속 갔다

이건 프롬프트나 컨텍스트의 문제가 아니다. **작업장의 규칙이 없어서** 생기는 문제다.

마스터 가이드의 표현:

> **프롬프트 엔지니어링** = "지금 무엇을 해달라고 말하는가"
> **컨텍스트 엔지니어링** = "무엇을 읽고 유지하는 컨텍스트를 설계하는 일"
> **하네스 엔지니어링** = "모델이 실제로 작동하는 주변 실행 환경 전체를 설계하는 일"

한 줄로 요약하면: **"프롬프트가 대화의 문장이라면, 하네스는 그 문장이 실제로 작동하는 작업장이다."**

하네스(harness)라는 용어가 낯설 수 있다. **"클로드가 일하는 작업장 설계"**라고 생각하면 된다.

사람도 마찬가지다. 아무리 유능한 직원이라도:
- 작업 규칙이 없으면 → 제멋대로 한다
- 확인 절차가 없으면 → 실수를 못 잡는다
- 위험한 권한이 열려있으면 → 사고가 난다

하네스 엔지니어링은 **이런 작업장 환경을 미리 설계**하는 것이다.

> **💭 생각의 흐름**
>
> **문제** — 프롬프트는 잘 썼는데 클로드가 테스트 없이 "완료"라고 했다
> **질문** — "왜 테스트를 안 돌리지? 분명히 규칙에 적었는데"
> **발견** — CLAUDE.md에 규칙을 적었지만, 실제로 강제하지는 않았다. Hook으로 PostToolUse에 테스트를 넣어야 한다
> **결과** — 규칙(CLAUDE.md) + 강제(Hook) + 확인(plan-before-edit) = 하네스의 3요소가 다 필요하다는 걸 깨달음

## 시작 프롬프트보다 시작 루틴이 더 중요하다

많은 사람이 "좋은 첫 프롬프트"에 집착한다. 하지만 진짜 중요한 건 **첫 프롬프트가 아니라, 매 세션을 시작할 때의 루틴**이다.

시작 루틴이란:
1. CLAUDE.md와 rules/ 확인 — 규칙이 읽히고 있는지
2. plan.md 확인 — 이전에 어디까지 했는지
3. handoff.md 확인 — 막힌 점이 있었는지
4. `/status`와 `/cost` 확인 — 컨텍스트 상태

이 4단계를 거치면, 첫 프롬프트가 "코드 수정해줘" 한 줄이어도 클로드는 올바른 맥락에서 시작한다. 반대로 아무리 정교한 프롬프트를 써도 규칙이 안 읽히고 있으면 소용없다.

## 하네스 = 5가지 최소 정의 묶음

하네스는 거창한 프레임워크가 아니다. **이 5가지만 정해두면** 클로드가 매번 일관성 있게 동작한다.

### 1. CLAUDE.md에 기본 규칙 적기

```markdown
# 프로젝트 작업 계약서
## 규칙
- 테스트 통과 전에는 끝났다고 하지 않는다
- .env 파일 수정 금지
- 변경은 작고 검토 가능하게
```

**역할**: "무엇을 해야 하고 하면 안 되는가"를 선언

### 2. settings.json에 허용/차단 범위 적기

```json
{
  "permissions": {
    "allow": ["Read", "Edit", "Bash(npm test)"],
    "deny": ["Read(./.env)"]
  }
}
```

**역할**: 선언한 규칙을 기술적으로 강제

### 3. plan.md로 큰 작업을 먼저 쓰기

```markdown
# 계획: 회원가입 버그 수정

## 목표
비밀번호 검증 로직 수정

## 완료 기준
- npm test 통과
- 기존 기능 유지

## 단계
- [ ] 현재 로직 확인
- [ ] 원인 파악
- [ ] 수정
- [ ] 테스트 확인
```

**역할**: 바로 코드를 쓰지 않고 먼저 계획을 세운다

### 4. hooks로 테스트나 포맷 자동화하기

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit",
      "hooks": [{"type": "command", "command": "npm run lint"}]
    }]
  }
}
```

**역할**: 규칙을 자동으로 지키게 강제

### 5. handoff.md에 세션 끝 상태를 남기기

```markdown
# 인수인계 메모

## 끝난 것
- 비밀번호 검증 로직 수정 완료
- lint 통과

## 막힌 점
- 리프레시 토큰 쪽은 아직 미확인

## 다음 단계
- src/auth/refresh.ts 확인

## 돌아왔을 때 질문
- 리프레시 토큰도 같은 방식으로 수정할지?
```

**역할**: 세션이 끊겨도 다음 세션에서 이어갈 수 있다

## 하네스의 3가지 격리

하네스가 하는 가장 중요한 일은 **격리**다. 세 가지를 동시에 격리한다.

### 1. 컨텍스트 격리

한 세션에서 조사, 계획, 구현, 리뷰를 다 하면 컨텍스트가 엉킨다. **역할별로 세션을 나누거나 SubAgent를 쓴다.**

```
❌ 한 세션에서 전부:
   조사 → 계획 → 구현 → 리뷰 → 컨텍스트 폭발

✅ 분리:
   세션 1: 조사 + 계획 (→ plan.md에 저장)
   세션 2: 구현 (plan.md 읽고 시작)
   세션 3: 리뷰 (변경분만 확인)
```

### 2. 파일 실체 격리

같은 파일을 직접 수정하면 실수를 되돌리기 어렵다. **worktree나 별도 output 폴더를 쓴다.**

```bash
claude --worktree    # Git 분리 폴더에서 작업
```

worktree에서 작업하면 원본이 안전하다. 결과가 좋으면 병합, 아니면 버린다.

### 3. 승인 격리

자주 쓰는 명령은 자동 허용하되, 위험한 것은 반드시 확인. **permissions와 hook으로 분리한다.**

```
자동 허용: Read, Edit, npm test, npm run lint
사용자 확인: npm install (새 패키지), git push
자동 차단: rm -rf, .env 접근, git push --force
```

## Claude Code 운영 7원칙

마스터 가이드와 실전 경험을 종합한 7가지 운영 원칙이다.

### 원칙 1: plan-before-edit

```
❌ 바로 수정하기 → 방향이 틀리면 처음부터 다시
✅ plan.md에 계획 먼저 → 확인 → 수정 → 검증
```

초보자일수록 plan이 중요하다. 클로드도 계획 없이 바로 코드를 쓰면 방향을 잃는다.

### 원칙 2: 컨텍스트 위치는 항상 확인

```
/status     ← 컨텍스트 사용량, 모델, 권한 등 현재 상태
/cost       ← 토큰을 얼마나 쓰고 있는지
```

모르는 상태에서 작업하면 컨텍스트가 오염된다.

### 원칙 3: 수정 전에 범위, 완료 기준, 검증 명령을 같이

```
"src/auth/login.ts의 validatePassword만 수정해.
완료 기준: npm test -- signup 통과.
다른 파일은 건드리지 마."
```

범위+기준+검증을 한 프롬프트에 넣으면 이탈이 줄어든다.

### 원칙 4: hooks로 테스트나 포맷 자동화

Edit 후 자동 lint, Bash 실행 전 위험 명령 검사. Ch.8에서 배운 것을 실전에 적용.

### 원칙 5: worktree를 분리 작업 폴더로

```bash
claude --worktree    # 병렬 작업, 안전한 실험
```

리팩터링이나 실험적 작업은 worktree에서. 결과가 좋으면 병합.

### 원칙 6: resume을 의식적으로

```bash
claude --continue    # 마지막 세션 이어가기
claude --resume              # 특정 세션 선택
claude --resume --fork-session  # 특정 세션의 맥락으로 새 세션 시작
```

아무 생각 없이 새 세션을 시작하면 이전 맥락을 다 잃는다. `/compact` 전에 handoff.md를 쓰고, 이어갈 때는 `--continue`를 쓴다.

### 원칙 7: 비용은 토큰 총량이 아니라 전체를 같이

비용을 줄이려면 토큰 수만 볼 게 아니라:
- cache read/write 비율
- tool output 크기
- 세션 길이 (긴 세션 = 앞부분 반복 읽기)
- 테스트 로그 크기

전체 구조를 봐야 한다. Part 5에서 자세히 다룬다.

## Compact Instructions 템플릿

`/compact` 실행 시 압축된 내용에 아래 구조가 남도록 하면 좋다:

```markdown
## Compact Instructions

감축할 때는 아래를 반드시 남겨라.
- 마지막으로 수정한 파일과 수정 이유
- 통과한 테스트, 실패한 테스트, 아직 안 돌린 테스트
- 바뀐 방향과 그 이유
- 다음 세션에서 바로 할 한 가지
- 다음 세션 시작 시 수동으로 확인해야 할 항목
```

이 내용을 CLAUDE.md에 넣어두면 `/compact`가 이 구조대로 압축한다.

## Session Memory 템플릿

긴 세션에서 handoff.md 대신 세션 메모리를 남길 때:

```markdown
# Current State
- 지금 바로 하고 있는 일

# Task Specification
- 사용자가 실제로 요청한 것

# Files and Functions
- 중요한 파일과 왜 중요한지

# Workflow
- 자주 쓰는 명령과 순서

# Errors & Corrections
- 실패한 시도와 다시 하지 말아야 할 것

# Key Results
- 이미 확정된 결과들과 답

# Worklog
- 최근 시도의 진행 순서
```

## 3가지 엔지니어링 비교 — 한눈에 보기

| | Context Eng. | Prompt Eng. | Harness Eng. |
|---|---|---|---|
| **질문** | "무엇을 읽게 할까" | "어떻게 시킬까" | "작업장을 어떻게 설계할까" |
| **관심사** | 파일, 메모리, 규칙 | 문장, 형식, 제약 | 권한, 검증, 격리, 인수인계 |
| **비유** | 책상 정리 | 요청서 작성 | 사무실 설계 |
| **도구** | CLAUDE.md, rules/ | 프롬프트 | settings.json, hooks, plan, handoff, worktree |
| **빠지면** | 관련 없는 정보 → 결과 부정확 | 모호한 요청 → 결과 불일치 | 규칙 미준수 → 사고 발생 |

**세 가지를 다 해야 한다.** 하나만 빠져도 결과가 불안정해진다.

실전 예시: 회원가입 버그 수정

```
[Context] CLAUDE.md + rules/api.md + src/features/signup/ 읽기
[Prompt]  "바로 고치지 말고 plan 먼저. 수정 범위와 테스트 계획 세워."
[Harness] settings.json에 .env deny + Hook에 자동 lint + worktree에서 작업
```

## 실습: 직접 해보기

### 실습 1: 5가지 최소 정의 만들기
자신의 프로젝트에서:
1. CLAUDE.md (이미 있으면 확인)
2. settings.json (Ch.6에서 만든 것 확인)
3. plan.md 하나 작성 (다음 할 작업)
4. hooks (settings.json 내, Edit 후 lint 자동화)
5. handoff.md 템플릿 준비

### 실습 2: plan-before-edit 실전
Claude Code에서 아무 수정 작업을 시작하기 전에:
```
"먼저 plan.md에 이번 작업 계획을 써줘.
목표, 수정 범위, 완료 기준, 테스트 계획 포함.
내가 확인한 뒤에 시작해."
```

### 실습 3: 3가지 엔지니어링 체크
자신이 최근에 한 작업을 떠올리고:
- Context: 필요한 파일만 읽었는가?
- Prompt: 구체적이었는가?
- Harness: 권한/검증/인수인계가 있었는가?

## 왜 이렇게 동작하는가

하네스 엔지니어링은 **소프트웨어 엔지니어링의 기본 원칙**과 같다:

- **관심사 분리** (Separation of Concerns) — 컨텍스트, 프롬프트, 하네스를 분리
- **최소 권한** (Least Privilege) — 필요한 권한만 허용
- **실패 안전** (Fail Safe) — Hook으로 자동 검사
- **재현 가능** (Reproducibility) — plan.md와 handoff.md로 작업을 재현

프로그래밍 경험이 없어도 이 원칙은 "좋은 작업 습관"으로 이해할 수 있다. 일을 잘하는 사람은 무의식적으로 이걸 한다. 하네스 엔지니어링은 **이걸 의식적으로, 파일로, 자동으로** 만드는 것이다.
