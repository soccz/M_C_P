# Ch.23 Scheduled Tasks

> **이 챕터를 마치면**: 예약 작업의 3가지 방식(Desktop/Cloud/세션 내 /loop)을 구분할 수 있고, self-contained 프롬프트를 작성할 수 있다. 첫 자동화를 설정하고 실행을 확인할 수 있다.

---

## 배경: "매번 같은 일"을 자동화하다

Ch.22에서 Channel이 "이벤트에 반응하는 것"이라면, Scheduled Tasks는 **"시간에 반응하는 것"**이다.

반복 작업은 어디에나 있다:
- 매주 월요일 아침에 경쟁사 뉴스를 정리
- 매일 아침 캘린더를 보고 미팅 준비물 정리
- 매주 금요일 주간 보고서 초안 작성

이런 작업을 매번 수동으로 시키면 **시간 낭비**이고, 가끔 **빠뜨린다.** 한 번 설정해두면 정해진 시간에 자동으로 실행되는 것이 예약 작업이다.

> **💭 생각의 흐름**
>
> **문제** — 매주 월요일 아침에 Cowork에서 "경쟁사 브리프 만들어줘"라고 입력한다. 3주 연속 잊어버려서 화요일에야 했다
> **질문** — "월요일 아침에 자동으로 실행되게 할 수 없나?"
> **시도** — Scheduled Task로 "매주 월요일 오전 9시"에 같은 프롬프트를 설정
> **발견** — 다음 월요일, 출근하니 브리프가 이미 만들어져 있었다
> **결론** — 반복 작업 = Scheduled Task. **한 번 설정, 매번 자동**

## 예약 작업의 3가지 방식

Claude에서 예약 작업을 설정하는 방법은 세 가지다.

### 전체 비교

| | **Desktop (Cowork)** | **Cloud** | **/loop (세션 내)** |
|---|---|---|---|
| **실행 위치** | 내 컴퓨터 | Anthropic 클라우드 | 내 컴퓨터 |
| **컴퓨터 켜져야** | 예 | 아니오 | 예 |
| **세션 열려야** | 아니오 | 아니오 | 예 |
| **재시작 후 유지** | 예 | 예 | 아니오 |
| **로컬 파일 접근** | 예 | 아니오 (GitHub clone) | 예 |
| **최소 간격** | 1분 | 1시간 | 1분 |
| **GitHub 필요** | 아니오 | 예 | 아니오 |
| **적합 용도** | 로컬 파일 정기 처리 | 코드 리뷰/배포 자동화 | 진행 중 작업 반복 확인 |

### 1. Desktop 예약 작업 (Cowork)

Claude Desktop 앱에서 설정하는 로컬 예약 작업이다. 앱이 열려 있는 동안 작동한다.

**설정 방법 (3가지 중 택 1):**

```
방법 1 — UI에서 직접:
  사이드바 "Scheduled" → "+ New task" → "New local task"
  → 이름, 프롬프트, 빈도, 작업 폴더 입력 → "Save"

방법 2 — /schedule 명령:
  대화창에 /schedule 입력 → Claude가 대화형으로 안내

방법 3 — 자연어:
  "매일 아침 9시에 코드 리뷰를 실행하는 예약 작업을 만들어줘"
```

**빈도 옵션:**

| 옵션 | 설명 |
|---|---|
| Manual | 스케줄 없음, "Run now"로만 실행 |
| Hourly | 매 시간 |
| Daily | 시간 선택 가능 (기본 오전 9시) |
| Weekdays | 평일만 (토·일 제외) |
| Weekly | 요일 + 시간 선택 |

**핵심 제약:**

```
⚠️ 컴퓨터가 켜져 있어야 하고, Claude Desktop 앱이 열려 있어야 한다.

노트북 덮개를 닫으면 → 절전 → 작업 건너뜀
앱을 종료하면 → 작업 실행 안 됨

놓친 작업: 앱 재시작 시 최근 7일치 확인 → 가장 최근 놓친 1회만 실행
```

### 2. Cloud 예약 작업

Anthropic 클라우드에서 실행되므로, **컴퓨터가 꺼져 있어도** 작동한다.

```
설정: claude.ai/code/scheduled 또는 Desktop 앱 → "New remote task"

차이점:
  ✅ 컴퓨터 꺼져도 실행
  ✅ 24시간 안정 실행
  ❌ 로컬 파일 접근 불가 (GitHub 레포를 fresh clone)
  ❌ 최소 간격 1시간
  ❌ GitHub 연동 필수
```

Cloud 작업은 매 실행마다 GitHub 레포를 새로 clone해서 시작한다. 로컬 파일을 읽어야 하는 작업(Cowork 브리프 등)에는 적합하지 않고, **코드 리뷰, PR 관리, 배포 모니터링** 같은 개발 워크플로우에 적합하다.

### 3. /loop (세션 내 반복)

현재 세션 안에서 프롬프트를 주기적으로 반복 실행한다. Ch.16에서 잠깐 다뤘던 기능이다.

```bash
# 5분마다 배포 상태 확인
/loop 5m 배포가 완료됐는지 확인해줘

# Claude가 간격을 자동 결정
/loop 배포 상태를 확인해줘

# 다른 명령을 반복 실행
/loop 20m /review-pr 1234

# 내장 maintenance 프롬프트 (미완성 작업 계속, PR 관리 등)
/loop
```

/loop의 특징:
- **세션이 열려 있는 동안만** 작동 (세션 종료 시 소멸)
- 재시작 후 유지되지 않음
- 간격 생략 시 Claude가 상황에 따라 자동 조절 (빌드 중이면 짧게, 조용하면 길게)

### CronCreate / CronDelete (고급)

Claude Code 세션 안에서 cron expression으로 더 정밀한 스케줄을 설정할 수 있다.

```
"매일 오전 9시에 PR 리뷰를 실행하는 cron 작업을 만들어줘"
→ Claude가 CronCreate 도구로 0 9 * * * 스케줄 생성

"활성 작업 목록을 보여줘"
→ CronList로 확인

"코드 리뷰 작업을 취소해줘"
→ CronDelete로 제거
```

CronCreate 작업은 **세션당 최대 50개**, 생성 후 **7일 뒤 자동 삭제**된다 (안전장치).

### 언제 무엇을 쓰나

```
"로컬 파일을 정기적으로 처리하고 싶다"
  (경쟁사 브리프, 보고서, 정리)
  → Desktop 예약 작업

"컴퓨터 꺼져 있어도 실행돼야 한다"
  (코드 리뷰, PR 관리, 배포 모니터링)
  → Cloud 예약 작업

"지금 하는 작업을 주기적으로 확인하고 싶다"
  (빌드 대기, 테스트 모니터링)
  → /loop
```

## Self-contained 프롬프트 작성법

예약 작업의 프롬프트는 **맥락 없이 독립 실행**되어야 한다. 사용자가 없으니 질문할 수도 없고, 이전 대화도 없다.

### 원칙

```
1. 자율 실행 가정 — 사용자에게 질문할 수 없다. 모호성 없애기
2. 정확한 범위 명시 — "전부"가 아니라 "이 폴더의 이 파일만"
3. 성공 기준 명시 — 무엇이 완료된 상태인지 명확히
4. 출력 위치 지정 — 결과를 어디에 저장할지
5. 실패 시 행동 — 데이터가 없거나 에러가 나면 어떻게 할지
```

### 나쁜 프롬프트 vs 좋은 프롬프트

```
❌ 나쁜 (모호):
  "경쟁사 분석해줘"

⚠️ 보통 (범위는 있지만 불완전):
  "PROJECTS/경쟁사-분석/ 폴더를 보고 분석해줘"

✅ 좋은 (self-contained):
  "PROJECTS/경쟁사-분석/brief.md를 읽고,
   brief.md에 명시된 경쟁사 3곳의 최근 블로그 포스트를 분석해줘.
   TEMPLATES/weekly-brief-template.md 형식으로 작성하고,
   CLAUDE-OUTPUTS/YYYY-MM-DD_경쟁사브리프.md로 저장.
   새 포스트가 없으면 '이번 주 새 포스트 없음'으로 표시."
```

이것은 Ch.10에서 배운 **제약형 프롬프트**와 같은 구조다. 목표·범위·형식·예외 처리를 모두 포함한다.

## Cowork 첫 자동화 7가지

어떤 작업을 자동화할지 고민된다면, 이 7가지 중에서 골라보자.

### 1. 주간 경쟁사 브리프

```
빈도: 매주 월요일 오전 9시
프롬프트:
  "PROJECTS/경쟁사-분석/brief.md를 읽고,
   경쟁사 블로그의 이번 주 포스트를 분석해줘.
   TEMPLATES/weekly-brief-template.md 형식으로,
   CLAUDE-OUTPUTS/ 에 날짜 포함 파일명으로 저장."
```

### 2. 아침 우선순위 정리

```
빈도: 매일 오전 8시 30분
프롬프트:
  "PROJECTS/일일-정리/ 폴더의 할일 목록과 캘린더를 읽고,
   오늘 가장 중요한 3가지를 골라 정리해줘.
   CLAUDE-OUTPUTS/YYYY-MM-DD_아침정리.md로 저장."
```

### 3. 미팅 후 후속 정리 초안

```
빈도: Manual (미팅 후 수동 실행)
프롬프트:
  "PROJECTS/미팅-노트/ 폴더에서 가장 최근 파일을 읽고,
   후속 액션 아이템을 추출해줘. 담당자별로 분류하고,
   CLAUDE-OUTPUTS/YYYY-MM-DD_후속정리.md로 저장."
```

### 4. 주간 KPI 보고서

```
빈도: 매주 금요일 오후 5시
프롬프트:
  "PROJECTS/KPI/ 폴더의 데이터 파일을 읽고,
   이번 주 KPI 변화를 요약해줘.
   전주 대비 증감 표시, 주의 필요 항목 강조.
   TEMPLATES/kpi-report-template.md 형식으로,
   CLAUDE-OUTPUTS/YYYY-MM-DD_KPI주간.md로 저장."
```

### 5. 콘텐츠 아이디어 요약

```
빈도: 매주 수요일 오전 10시
프롬프트:
  "PROJECTS/콘텐츠/ 폴더의 아이디어 메모를 읽고,
   가장 유망한 3개를 골라 간단한 아웃라인을 만들어줘.
   CLAUDE-OUTPUTS/YYYY-MM-DD_콘텐츠아이디어.md로 저장."
```

### 6. 고객 요청 분류

```
빈도: 매일 오후 6시
프롬프트:
  "PROJECTS/고객-요청/ 폴더에 새로 추가된 파일을 읽고,
   긴급/일반/문의로 분류해줘.
   기존 분류 결과 파일에 추가 (덮어쓰기 금지).
   CLAUDE-OUTPUTS/고객요청_분류.md에 날짜별로 누적."
```

### 7. 매일 아침 캘린더 요약

```
빈도: 매일 오전 8시
프롬프트:
  "PROJECTS/일정관리/ 폴더의 이번 주 캘린더 파일을 읽고,
   오늘 일정에서 각 미팅에 필요한 준비물을 정리해줘.
   CLAUDE-OUTPUTS/YYYY-MM-DD_캘린더요약.md로 저장.
   일정이 없으면 '오늘 미팅 없음'으로 저장."
```

(캘린더 커넥터가 연결된 경우, 파일 대신 커넥터를 통해 직접 일정을 읽을 수도 있다.)

## 실행 확인과 실패 대처

### Desktop 작업 모니터링

```
실행 확인:
  사이드바 "Scheduled" 섹션 → 각 작업의 실행 이력 확인
  과거 실행 (건너뛴 것 포함)이 작업 상세 페이지에 기록됨

첫 실행 팁:
  "Run now"로 즉시 테스트 실행
  → 권한 프롬프트가 뜨면 "always allow" 선택
  → 이후 자동 실행 시 권한 문제 없음
```

### Cloud 작업 모니터링

```
실행 확인:
  claude.ai/code/scheduled → 각 run 클릭
  → Claude가 한 일 확인, 변경 검토, PR 생성 가능

"Run now"로 테스트:
  설정 직후 한 번 수동 실행해서 결과 확인
```

### /loop 모니터링

```
활성 작업 확인:
  "활성 cron 작업을 보여줘" 또는 "scheduled tasks 목록"

작업 취소:
  "배포 확인 작업을 취소해줘" (설명으로 취소 가능)
```

### 실패 시 대처

```
문제: 작업이 건너뛰어졌다
  → 컴퓨터가 절전이었나? 앱이 닫혀 있었나?
  → "Keep computer awake" 설정 확인
  → 안정 실행이 필요하면 Cloud 작업으로 전환

문제: 결과물이 기대와 다르다
  → self-contained 프롬프트를 더 구체적으로
  → "Run now"로 테스트 후 프롬프트 수정

문제: 권한 프롬프트에서 멈춤
  → "Run now"로 한 번 실행 후 "always allow" 설정
  → 또는 허용 목록을 넉넉하게 설정
```

## 🐍 파일 I/O — 자동화의 기본은 읽고 쓰기

예약 작업의 핵심은 "파일을 읽고, 처리하고, 파일로 저장"이다. 파이썬에서도 마찬가지다.

```python
# 파일 읽기
with open("input.txt", "r") as f:
    content = f.read()

# 처리
summary = content[:100]  # 앞 100글자만 추출 (간단한 예시)

# 파일 쓰기
with open("output.txt", "w") as f:
    f.write(summary)
```

`open(파일명, 모드)`:
- `"r"` = read (읽기)
- `"w"` = write (쓰기, 새로 만듦)
- `"a"` = append (추가, 기존 내용 유지)

예약 작업이 하는 일도 이것과 같다:
1. 입력 파일을 **읽고** (`open("brief.md", "r")`)
2. 내용을 **처리**하고 (분석, 요약, 분류)
3. 결과를 **저장**한다 (`open("output.md", "w")`)

## 왜 이렇게 동작하는가

예약 작업의 핵심 원리는 **시간 트리거(time trigger)**다.

Ch.8의 Hook이 "도구 사용 이벤트 → 자동 실행"이었다면:
- **Scheduled Task** = "시간 이벤트 → 자동 실행"
- **Channel** = "외부 이벤트 → 자동 실행"
- **Hook** = "내부 이벤트 → 자동 실행"

세 가지 모두 같은 패턴이다: **이벤트 → 조건 확인 → 실행.** 차이는 이벤트의 출처(시간/외부/내부)뿐이다.

self-contained 프롬프트가 중요한 이유도 이 구조에서 나온다. 사용자가 없는 상태에서 실행되므로, **모든 맥락이 프롬프트 안에** 들어 있어야 한다. 이것은 Ch.10의 제약형 프롬프트와 Ch.11의 하네스 원칙이 합쳐진 것이다.
