# Ch.19 토큰과 비용

> **이 챕터를 마치면**: 토큰이 무엇이고 어떻게 비용이 되는지 이해한다. Prompt Caching의 원리를 설명할 수 있고, 비용이 빠지는 5가지 구간을 인식하여 운영 구조로 줄이는 방법을 실천할 수 있다.

---

## 배경: 클로드는 공짜가 아니다

Ch.18까지 클로드의 다양한 기능을 배웠다. 파일을 읽고, 코드를 수정하고, SubAgent를 띄우고, 테스트를 돌린다. 이 모든 동작에는 **토큰**이라는 비용이 따른다.

토큰을 모르면:
- "왜 갑자기 느려졌지?" → 컨텍스트가 꽉 찼다
- "왜 이렇게 비싸지?" → 도구 출력이 토큰을 먹고 있었다
- "왜 앞에서 한 말을 잊지?" → 토큰 한도를 넘어서 오래된 내용이 밀려났다

토큰을 알면 **왜 그런지** 이해하고, **어떻게 줄이는지** 실천할 수 있다.

> **💭 생각의 흐름**
>
> **문제** — 클로드에게 파일 20개를 리뷰시켰더니 /cost가 $3을 넘었다. 같은 작업을 다시 했더니 $1.5밖에 안 나왔다
> **질문** — "같은 작업인데 비용이 절반으로 줄어든 이유가 뭐지?"
> **발견** — 두 번째에는 **Prompt Caching**이 작동했다. 이미 보낸 배경(CLAUDE.md, rules, 시스템 프롬프트)이 캐시되어 할인 적용
> **확장** — 비용은 "얼마나 썼느냐"보다 **"어디서 빠지느냐"**가 중요하다. 구간을 알면 구조로 줄일 수 있다
> **결론** — 토큰 경제학을 이해하면 **같은 결과를 더 적은 비용으로** 얻을 수 있다

## Token Economy — 토큰 경제학

### 토큰이란

**토큰 = 클로드가 읽고 쓰는 최소 단위.** 글자도 아니고 단어도 아닌, 그 사이의 단위다.

```
"안녕하세요, 클로드!" → 약 8-10 토큰
"Hello, Claude!"     → 약 4 토큰

한국어는 영어보다 토큰을 더 많이 쓴다 (같은 의미, 더 많은 토큰)
```

모든 것이 토큰으로 변환된다:
- 사용자가 입력한 질문 → 토큰
- CLAUDE.md → 토큰
- rules/ 파일들 → 토큰
- 클로드가 읽은 파일 내용 → 토큰
- 클로드가 생성한 코드 → 토큰
- grep 결과, 테스트 로그 → 토큰

### input token vs output token

토큰에는 두 종류가 있다.

```
┌──────────────── input token ────────────────┐
│                                              │
│  시스템 프롬프트 (Claude의 기본 지침)         │
│  CLAUDE.md + rules/ + Memory                │
│  사용자의 메시지                              │
│  도구 출력 (파일 내용, grep 결과, 테스트 로그) │
│  이전 대화 내역                              │
│                                              │
│  = Claude가 "읽는" 모든 것                   │
└──────────────────────────────────────────────┘

┌──────────────── output token ───────────────┐
│                                              │
│  Claude의 응답 텍스트                        │
│  생성한 코드                                 │
│  도구 호출 (어떤 도구를, 어떤 인자로)          │
│  Extended Thinking의 사고 과정               │
│                                              │
│  = Claude가 "쓰는" 모든 것                   │
└──────────────────────────────────────────────┘
```

**핵심: output token이 input token보다 비싸다.** 클로드가 긴 코드를 생성하면 비용이 올라간다.

### 비용 구조

Claude Code의 비용은 **4가지 토큰 유형**으로 나뉜다.

| 유형 | 설명 | 상대 비용 |
|---|---|---|
| **input token** | 새로 읽는 입력 | 기준 |
| **output token** | 생성하는 출력 | input의 약 5배 |
| **cache read token** | 캐시에서 읽은 입력 | input의 약 1/10 |
| **cache creation token** | 캐시를 새로 만든 입력 | input의 약 1.25배 |

```
같은 CLAUDE.md를 10번 보내면:
  첫 번째: cache creation (1.25배) — 캐시를 만든다
  2~10번째: cache read (0.1배) — 캐시에서 읽는다

→ 첫 번째만 비싸고, 이후 9번은 매우 싸다
```

이것이 **Prompt Caching**이다.

## Prompt Caching — 같은 배경을 반복 보내면 할인

### 왜 Prompt Caching이 중요한가

Claude Code는 매 턴(turn)마다 **전체 컨텍스트를 다시 보낸다.**

```
턴 1: [시스템 프롬프트 + CLAUDE.md + 질문 1]
턴 2: [시스템 프롬프트 + CLAUDE.md + 질문 1 + 응답 1 + 질문 2]
턴 3: [시스템 프롬프트 + CLAUDE.md + 질문 1 + 응답 1 + 질문 2 + 응답 2 + 질문 3]
```

시스템 프롬프트와 CLAUDE.md는 **매번 똑같이** 보내진다. 이걸 매번 새로 읽으면 낭비다.

### Prompt Caching의 동작 원리

```
캐시 없이:
  턴 1: 시스템(10K) + CLAUDE.md(500) + 질문(100)
        = 10,600 input token 전액 과금

캐시 있을 때:
  턴 1: 시스템(10K) + CLAUDE.md(500)  → cache creation (1.25배)
        + 질문(100)                    → input (1배)
  
  턴 2: 시스템(10K) + CLAUDE.md(500)  → cache read (0.1배!) ← 여기서 절약
        + 질문 1(100) + 응답 1(500)    → cache read (0.1배!) ← 여기도
        + 질문 2(100)                  → input (1배)
```

**앞부분이 바뀌지 않으면** 캐시가 적용된다. 시스템 프롬프트, CLAUDE.md, 이전 대화 — 이 부분은 매번 같으니까 캐시로 읽어서 **비용이 1/10**로 줄어든다.

### 캐시가 깨지는 경우

캐시는 **5분 TTL(Time To Live)**이 있다. 5분 이내에 다음 턴을 보내면 캐시가 유지되고, 5분이 넘으면 캐시가 만료되어 다시 만들어야 한다.

```
빠른 대화 (캐시 유지):
  턴 1 → (30초) → 턴 2 → (1분) → 턴 3
  ✅ 캐시 계속 활용, 비용 낮음

느린 대화 (캐시 만료):
  턴 1 → (10분 생각) → 턴 2
  ❌ 캐시 만료, 다시 cache creation
```

이것이 실무에서 의미하는 것:
- **집중해서 작업하면 비용이 낮다** (캐시 유지)
- **중간에 오래 멈추면 비용이 올라간다** (캐시 만료)
- **새 세션을 시작하면 캐시가 없다** (처음부터 다시)

### 캐시 TTL 제어하기 🆕

5분 TTL이 기본이지만, 2026년에 **환경변수로 TTL을 조정하는 기능**이 추가됐다. 긴 호흡으로 작업할 때 유용하다.

| 환경변수 | 효과 | 언제 쓰나 |
|---|---|---|
| `ENABLE_PROMPT_CACHING_1H=1` | 5분 TTL 대신 **1시간 TTL** 사용 | 긴 호흡 작업, 미팅 중간에도 캐시 유지하고 싶을 때 |
| `FORCE_PROMPT_CACHING_5M=1` | 1시간이 기본이어도 **5분 TTL로 강제** | 짧은 세션, 캐시 오버헤드 줄이고 싶을 때 |

### 사용법

```bash
# .env 파일이나 셸에서
export ENABLE_PROMPT_CACHING_1H=1
claude
```

또는 한 번만 쓰고 싶으면:
```bash
ENABLE_PROMPT_CACHING_1H=1 claude
```

### 1시간 TTL의 트레이드오프

**장점:**
- 미팅/점심/화장실 다녀와도 캐시 유지 → cache read 비용 유지
- 긴 호흡 작업(리팩터링, 설계 회의) 적합

**단점:**
- **cache write 비용이 5분 TTL보다 약 2배 비싸다** (1.25배 → 2배)
- 짧은 세션에서는 손해 — 캐시를 만들었는데 얼마 안 써먹고 세션이 끝남

### 판단 기준

```
짧은 작업(30분 이내)       → 기본 5분 TTL
긴 리팩터링/설계(1~3시간) → ENABLE_PROMPT_CACHING_1H=1
실험/탐색 (정해진 호흡 없음) → 기본 5분
팀 공유 세션(일시 중단 많음) → ENABLE_PROMPT_CACHING_1H=1
```

> 실무 팁: 이 env var를 `~/.claude/settings.json`의 `"env"` 섹션에 넣어두면 전역 적용된다. 하지만 **프로젝트마다 다른 호흡**을 쓴다면 `.claude/settings.json`에 프로젝트별로 넣는 게 낫다.

## 비용이 빠지는 5가지 구간

Ch.9에서 컨텍스트의 구조를, Ch.13에서 MCP의 토큰 오버헤드를, Ch.16에서 /compact의 효과를 배웠다. 여기서는 **비용 관점에서** 5가지 구간을 정리한다.

### 구간 1: 긴 CLAUDE.md와 규칙 파일

```
CLAUDE.md     500 토큰  → 매 턴 읽힘
rules/ 10개   3,000 토큰 → 매 턴 읽힘
Memory        200 토큰  → 매 턴 읽힘
────────────────────────
합계: 3,700 토큰 × 매 턴
```

10턴 대화에서 3,700 × 10 = 37,000 토큰이 규칙 파일에만 소비된다. Prompt Caching이 있어서 실제 비용은 낮지만, **캐시가 만료되면 전액** 과금된다.

**줄이는 법:**
- CLAUDE.md는 500줄 이하로 유지
- rules/는 path-specific으로 필요한 것만 로딩
- 안 쓰는 규칙은 정리

### 구간 2: tool output — 도구 출력

이것이 **가장 큰 비용 요인**이다.

```
grep "TODO" → 결과 500줄 → 약 5,000 토큰
파일 읽기 (큰 파일) → 2,000줄 → 약 15,000 토큰
테스트 실행 → 로그 전체 → 약 20,000 토큰
npm install → 설치 로그 → 약 10,000 토큰
```

도구 출력은 **캐시되지 않는 경우가 많다.** 새로운 도구 호출의 결과는 매번 새 input token이다.

**줄이는 법:**
- grep은 범위를 좁혀서 (`grep "TODO" src/auth/` vs `grep "TODO"`)
- 파일은 필요한 부분만 읽기 (Read 도구의 offset/limit 활용)
- 테스트는 실패한 것만 보기 (`npm test -- --bail`)
- 불필요한 도구 출력을 CLAUDE.md에서 제한 ("테스트 출력은 실패 케이스만")

### 구간 3: 긴 세션의 컨텍스트 누적

세션이 길어질수록 **이전 대화 전체**가 매 턴 함께 보내진다.

```
턴 1: 1,000 토큰
턴 5: 15,000 토큰 (이전 4턴의 대화 포함)
턴 10: 50,000 토큰
턴 20: 120,000 토큰 ← 위험!
턴 25: 컨텍스트 한도 초과 → 자동 compaction
```

앞 턴의 대화는 대부분 **cache read**로 처리되지만, 세션이 매우 길어지면 캐시 적중률이 떨어지고 비용이 급증한다.

**줄이는 법:**
- Ch.16에서 배운 /compact를 토큰 50% 넘으면 실행
- 한 세션 = 한 작업. 작업이 바뀌면 새 세션
- 큰 작업은 plan.md로 계획 → 작은 단위로 나누어 실행

### 구간 4: agent/subagent의 중복 컨텍스트

Ch.18에서 배운 SubAgent는 **각자 독립된 컨텍스트**를 가진다. 이 말은 **각자 시스템 프롬프트를 따로 읽는다**는 뜻이다.

```
메인 Agent:    시스템 프롬프트 10K 토큰
SubAgent A:    시스템 프롬프트 10K 토큰  ← 별도
SubAgent B:    시스템 프롬프트 10K 토큰  ← 별도
SubAgent C:    시스템 프롬프트 10K 토큰  ← 별도
────────────────────────────────────────
합계: 40K 토큰 (시스템 프롬프트만으로)
```

SubAgent를 4개 띄우면 시스템 프롬프트만으로 40K 토큰이 소비된다.

**줄이는 법:**
- 간단한 작업에 SubAgent 쓰지 않기 (Ch.18 운영 원칙)
- SubAgent의 prompt를 간결하게 (필요한 맥락만)
- 가벼운 모델(haiku) 선택지 고려

### 구간 5: 불필요한 MCP 설정

Ch.13에서 다뤘듯이, MCP 서버를 연결하면 **도구 설명이 매 턴 컨텍스트에 포함**된다.

```
MCP 서버 1개: 도구 10개 × 설명 50토큰 = 500 토큰
MCP 서버 5개: 도구 50개 = 2,500 토큰 × 매 턴

+ 사용하지 않는 도구도 설명은 매번 읽힘
```

**줄이는 법:**
- 안 쓰는 MCP 서버는 제거
- 필요할 때만 연결, 끝나면 해제
- 최소 연결 원칙 (Ch.13 참고)

### 5가지 구간 요약

```
비용 영향 크기:

도구 출력 ████████████████████ (가장 큼)
긴 세션   ██████████████████
SubAgent  ████████████████
MCP 설정  ██████████
규칙 파일 ██████  (캐시 덕에 상대적으로 적음)
```

## /cost로 확인하기 — 습관화

### /cost 명령

```
> /cost

Total cost:            $0.55
Total duration (API):  6m 19.7s
Total duration (wall): 6h 33m 10.2s
Total code changes:    42 lines added, 15 lines removed
```

/cost는 **현재 세션의 누적 비용**을 보여준다.

### 언제 /cost를 확인하나

```
✅ /cost를 치는 타이밍:

1. 작업 시작 직후 (기준점 잡기)
2. 큰 도구 출력 후 (grep, 테스트 등)
3. 작업 완료 시 (총 비용 기록)
4. "느려졌다" 느낄 때 (컨텍스트 과부하 확인)
```

### /cost 읽는 법

```
Total cost: $0.55
             │
             └── 이게 높다면?
                 ├── 도구 출력이 많았나? → 범위 좁히기
                 ├── 세션이 너무 길었나? → /compact
                 └── SubAgent를 많이 썼나? → 필요한 것만

Total duration (API): 6m 19.7s    ← Claude가 실제로 일한 시간
Total duration (wall): 6h 33m     ← 내가 대화한 총 시간
                       │
                       └── API << wall이면 대부분 "내가 생각한 시간"
                           → 캐시가 만료됐을 수 있다
```

### 구독자 vs API 사용자

```
Claude Max/Pro 구독:
  → 월 정액 구독료에 포함
  → /cost로 사용량 확인
  → 토큰 한도(rate limit) 있음

Claude API (Console):
  → 사용한 만큼 과금
  → /cost로 정확한 비용 추적
  → 팀 예산 관리 필요
```

## Token Budget — 팀 비용 관리

API를 사용하는 팀에서는 **토큰 예산(Token Budget)**을 관리해야 한다.

### 팀 비용의 현실

```
평균: $13/개발자/활성일
월 범위: $150~250/개발자
90% 사용자: 일일 $30 이하
```

5명 팀이면 월 $750~1,250 정도다. 비싸지 않을 수도, 비쌀 수도 있다. 중요한 건 **관리하지 않으면 예상보다 훨씬 높아진다**는 것이다.

### 비용 관리 도구

| 도구 | 설명 | 위치 |
|---|---|---|
| **/cost** | 세션별 비용 확인 | 터미널에서 직접 |
| **/cost** | 사용 패턴 확인 (구독자/API 공용) | 터미널에서 직접 |
| **Workspace Spend Limits** | 팀 전체 월 예산 한도 설정 | Claude Console (웹) |

### Workspace Spend Limits

Claude Console(웹)에서 팀 전체의 월 예산 한도를 설정할 수 있다. 한도에 도달하면 추가 사용이 차단된다.

```
Claude Console → Settings → Workspace → Spend Limits
  → Monthly limit: $500
  → 한도 도달 시 알림 + 차단
```

이것은 "나도 모르게 비용이 폭주하는 것"을 방지하는 안전장치다. 특히 여러 명이 API를 사용하는 팀에서 필수적이다.

### TPM / RPM — 속도 제한

예산 말고도 "속도"에 제한이 걸린다.

| 이름 | 의미 | 걸리면 |
|------|------|-------|
| **TPM** (Tokens Per Minute) | 분당 처리 가능 토큰 | 429 에러, 잠시 대기 |
| **RPM** (Requests Per Minute) | 분당 요청 횟수 | 429 에러, 잠시 대기 |

```
기본 Tier (Console):
  TPM: 40,000 ~ 80,000 (모델별 다름)
  RPM: 50 ~ 4,000
```

한도는 사용량이 쌓이면 자동으로 올라간다(Tier 1→2→3...). 갑자기 429가 뜨면 그 분만 쉬고 재시도하면 된다. Claude Code는 기본 백오프(지수 대기)를 내장하고 있어 대부분 투명하게 처리된다.

**Tip**: SubAgent를 많이 병렬로 띄우면 RPM에 먼저 걸린다. 팀 단위로 쓸 때는 Tier 확인 필수.

## 🆕 관측과 감사 — 2026 운영 도구

혼자 쓸 때는 `/cost` 하나면 충분하다. 하지만 팀/조직 단위로 쓰기 시작하면 **"누가, 어디에, 얼마나"**를 한눈에 보고 싶어진다. 2026년 Claude Code는 관측을 위한 세 가지 수단을 제공한다.

### 1. Analytics 대시보드 — 어디서 봐야 하나

**중요한 구분**: Analytics 대시보드는 **두 군데**가 있는데 성격이 다르다.

| 대시보드 | 대상 | 경로 | 보여주는 것 |
|---------|------|------|------------|
| **Teams/Enterprise 대시보드** | 구독 팀 관리자 | claude.ai/settings (admin) | 팀원별 사용 요약, 활성 사용자 수, MAU |
| **Console 대시보드** | API 조직 관리자 | console.anthropic.com | 워크스페이스별 토큰·비용, 모델별 분포, Caching 적중률 |

둘은 **같은 사용자의 활동을 다른 관점**으로 본다. Teams가 "얼마나 쓰는가(빈도)", Console이 "얼마나 드는가(비용)"에 가깝다.

### 2. Contribution metrics + GitHub App

Teams/Enterprise 플랜에는 "클로드가 실제로 코드에 얼마나 기여했는가"를 측정하는 **contribution metrics**가 있다. GitHub App을 연결하면 PR 단위로:

- 클로드가 작성한 줄 수 / 사람이 작성한 줄 수
- 리뷰 반영률
- 시간 절감 추정치

**ZDR 환경에서의 제약**: Zero Data Retention(ZDR, Ch.28 참고)을 켠 조직은 contribution metrics의 일부 데이터가 수집되지 않는다. 프라이버시를 강화하면 세부 지표는 포기해야 하는 트레이드오프다.

### 3. OpenTelemetry 연동

자체 관측 스택(Datadog, Grafana, Honeycomb 등)이 있는 조직은 OpenTelemetry로 내보낼 수 있다.

```bash
# .env 또는 settings.json의 env 섹션
export OTEL_EXPORTER_OTLP_ENDPOINT="https://otel.example.com:4318"
export OTEL_EXPORTER_OTLP_HEADERS="api-key=<YOUR_KEY>"
export CLAUDE_CODE_ENABLE_TELEMETRY=1
```

내보내지는 신호:
- **metrics**: 토큰 소비, API 지연, 도구 호출 빈도
- **traces**: 한 세션의 요청→응답→도구 호출 체인
- **logs**: 에러, 거부된 권한 요청

이게 유용한 이유: 내부 SLA/알림 시스템과 **같은 파이프라인**에 붙일 수 있다. Claude Code만의 별도 대시보드를 운영할 필요가 없다.

### 언제 뭘 쓰나

```
나 혼자 / 주니어 단계       → /cost 만
팀 2~5명                   → Teams 대시보드
조직 전체 / 예산 관리       → Console 대시보드 + Spend Limits
보안/감사 요건              → OpenTelemetry + 내부 SIEM
코드 기여 증명 / ROI 보고   → Contribution metrics + GitHub App
```

## AI factory와 토큰 공장

토큰을 이해하는 유용한 비유가 있다: **AI를 공장으로 보는 것.**

### 공장 비유

```
AI Factory (토큰 공장):

원재료 투입 (input token):
  ├── 설계도 = CLAUDE.md, rules/
  ├── 부품 = 코드 파일, 데이터
  ├── 작업 지시서 = 사용자 프롬프트
  └── 검사 결과 = 도구 출력

생산 (output token):
  ├── 완제품 = 수정된 코드
  ├── 보고서 = 분석 결과
  └── 작업 일지 = 도구 호출

원가 = 투입 토큰 비용 + 생산 토큰 비용
```

이 비유가 유용한 이유:

1. **원재료를 줄이면 원가가 내려간다** — 불필요한 input을 줄이면 비용이 줄어든다
2. **같은 설계도를 매번 새로 인쇄하면 낭비다** — Prompt Caching이 이 문제를 해결
3. **불량품이 나오면 재작업 비용이 든다** — 잘못된 결과를 고치려면 추가 턴이 필요
4. **공장 라인을 분리하면 효율적이다** — SubAgent로 작업 분업

### 비용을 줄이는 운영 구조

공장의 효율을 높이듯, Claude Code의 비용도 **구조**로 줄인다.

```
구조 1: 짧은 계획, 긴 실행 분리
  ├── 계획 세션: 토큰 적게 소비 (plan.md 작성)
  └── 실행 세션: 계획 기반으로 효율적 실행
  → 방향 수정 횟수 감소 = 재작업 비용 감소

구조 2: diff 위주로 작업
  ├── ❌ 파일 전체를 읽고 전체를 다시 쓴다
  └── ✅ 변경할 부분만 읽고 해당 부분만 수정
  → 도구 출력 크기 감소 = input token 감소

구조 3: worktree로 실험 격리
  ├── 메인 세션: 확정된 작업만
  └── worktree: 실험적 시도 (실패해도 메인 비용 없음)
  → 실패 비용이 메인 세션에 누적되지 않음

구조 4: SubAgent 활용
  ├── 탐색은 Explore SubAgent에게 (탐색 결과가 메인 오염 방지)
  └── 메인은 판단과 실행에 집중
  → 메인 세션의 컨텍스트 비용 감소

구조 5: 세션 위생
  ├── 한 세션 = 한 작업
  ├── /compact를 적절히 사용
  └── /cost를 습관적으로 확인
  → 전체 토큰 소비 최적화
```

## 실전: 같은 작업, 다른 비용

같은 "코드 리뷰" 작업을 두 가지 방식으로 했을 때의 비용 차이:

### 비효율적 방식

```
1. "전체 프로젝트 코드 리뷰해줘"
2. 클로드가 모든 파일을 하나하나 읽음 (50개 파일)
3. 중간에 10분 쉼 (캐시 만료)
4. 다시 돌아와서 추가 질문
5. 결과가 마음에 안 들어서 다시 요청

예상 비용: $3~5
```

### 효율적 방식

```
1. plan.md에 "리뷰 범위: src/auth/ 폴더, 보안 관점"
2. SubAgent 2개: 보안 검토 + 에러 핸들링 검토 (병렬)
3. 집중해서 작업 (캐시 유지)
4. 결과를 한 번에 정리

예상 비용: $0.5~1
```

**차이: 3~10배.** 같은 결과인데 구조만 다르다.

## 왜 이렇게 동작하는가

토큰 경제학은 **정보 이론**의 실용적 적용이다.

클로드는 입력을 받아 출력을 생성하는 **변환기(transformer)**다. 입력이 많을수록 처리 비용이 올라가고, 출력이 길수록 생성 비용이 올라간다. 이것은 물리적 제약이다 — 더 많은 계산에는 더 많은 에너지와 시간이 필요하다.

Prompt Caching은 **메모이제이션(memoization)**이다. 한 번 계산한 것을 저장해두고 다시 쓴다. 프로그래밍에서 캐시가 작동하는 원리와 정확히 같다.

그래서 비용 최적화의 원리도 프로그래밍과 같다:
- **캐시를 활용하라** (Prompt Caching)
- **불필요한 I/O를 줄여라** (도구 출력 최소화)
- **작업을 분리하라** (세션 분리, SubAgent)
- **일찍 실패하라** (plan 먼저, 방향 확인 후 구현)
