# Ch.18 Agent와 SubAgent

> **이 챕터를 마치면**: agentic이 무엇인지, Agent와 SubAgent의 차이를 이해한다. SubAgent로 작업을 병렬화하고 컨텍스트를 격리하는 방법을 실천할 수 있다. worktree와 SubAgent의 관계를 설명할 수 있다.

---

## 배경: 클로드 혼자서는 한계가 있다

Ch.17에서 plan.md와 handoff.md로 작업의 시작과 끝을 구조화했다. 하지만 작업이 커지면 새로운 문제가 생긴다.

```
❌ 큰 작업을 하나의 세션에서 전부 처리
   → 컨텍스트가 잡다해진다
   → 파일 검색 결과 + 코드 리뷰 + 테스트 로그가 뒤섞인다
   → 앞에서 준 규칙을 뒤에서 잊는다

✅ 조사는 조사 전문가에게, 리뷰는 리뷰 전문가에게
   → 각자 깨끗한 컨텍스트에서 집중
   → 결과만 메인에게 돌려준다
   → 메인 세션은 깔끔하게 유지
```

이것이 **SubAgent** — 클로드가 다른 클로드에게 일을 시키는 방법이다.

> **💭 생각의 흐름**
>
> **문제** — 프로젝트에 파일이 200개가 넘는다. "보안 취약점 찾아줘"를 시키면 클로드가 파일을 하나하나 열면서 컨텍스트가 폭발한다
> **질문** — "검색 결과가 메인 컨텍스트를 오염시키지 않으면서 조사할 방법이 없나?"
> **발견** — SubAgent는 **독립된 컨텍스트**에서 작업한다. 파일 100개를 뒤져도 메인 세션은 깨끗하다
> **확장** — SubAgent 여러 개를 **동시에** 띄우면 병렬 작업도 된다. 보안 검토 + 스타일 검토 + 테스트 검토를 한 번에
> **결론** — **SubAgent = 컨텍스트 격리 + 병렬 작업.** 큰 작업을 쪼개는 핵심 도구

## agentic이란

일반적인 챗봇은 **질문 → 답변** 한 번으로 끝난다. agentic은 다르다.

```
일반 챗봇:
  사용자 → "이 코드 고쳐줘" → 답변 1개 → 끝

agentic:
  사용자 → "이 코드 고쳐줘"
  클로드 → 파일을 읽는다 (도구 사용)
        → 관련 파일을 더 찾는다 (도구 사용)
        → 수정한다 (도구 사용)
        → 테스트를 돌린다 (도구 사용)
        → 실패하면 다시 수정한다 (판단 + 도구 사용)
        → 통과하면 결과를 보고한다
```

**agentic = 모델이 스스로 도구를 부르고, 결과를 보고, 다음 행동을 판단하는 방식.**

Claude Code가 바로 이 agentic 시스템이다. 파일을 읽고, 코드를 수정하고, 테스트를 돌리고, 에러가 나면 다시 수정하는 — 이 모든 것을 사용자 개입 없이 반복한다.

### agentic loop — 핵심 구조

agentic 시스템의 핵심은 **반복 루프**다.

```
┌──────────────────────────────────┐
│                                  │
│   ① 관찰 (Observe)              │
│   → 현재 상태를 파악한다         │
│   → 파일 읽기, 에러 확인         │
│                                  │
│   ② 생각 (Think)                │
│   → 다음 행동을 결정한다         │
│   → "이 에러는 타입 문제니까     │
│      import를 고쳐야겠다"        │
│                                  │
│   ③ 행동 (Act)                  │
│   → 도구를 실행한다              │
│   → 파일 수정, 테스트 실행       │
│                                  │
│   ④ 완료인가? ──── Yes → 보고   │
│       │                          │
│       No                         │
│       │                          │
│       └── ①로 돌아가기           │
│                                  │
└──────────────────────────────────┘
```

이 **관찰 → 생각 → 행동 → 반복** 구조를 **agentic loop**라고 한다. 클로드가 코드를 고치다가 테스트가 실패하면 다시 에러를 읽고, 원인을 판단하고, 수정하고, 테스트를 돌리는 — 이 반복이 바로 agentic loop다.

## Agent vs SubAgent vs Agent Team

### Agent — 메인 작업자

**Agent = 사용자와 직접 대화하는 클로드 인스턴스.** 터미널에서 `claude`를 실행하면 시작되는 그 세션이다.

```
사용자 ──── Agent (메인 클로드)
             │
             ├── 파일 읽기
             ├── 코드 수정
             ├── 테스트 실행
             └── 결과 보고
```

Agent는 사용자의 지시를 받고, 도구를 사용하고, 결과를 보여준다. 지금까지 배운 모든 것이 Agent의 행동이다.

### SubAgent — 보조 작업자

**SubAgent = Agent가 띄우는 독립된 클로드 인스턴스.** 메인 세션과 **컨텍스트가 분리**된다.

```
사용자 ──── Agent (메인)
             │
             ├── SubAgent A: "코드 탐색해줘"
             │    └── (독립 컨텍스트에서 파일 탐색)
             │    └── 결과만 메인에게 반환
             │
             ├── SubAgent B: "보안 검토해줘"
             │    └── (독립 컨텍스트에서 보안 분석)
             │    └── 결과만 메인에게 반환
             │
             └── 메인은 두 결과를 합쳐서 판단
```

핵심 차이:

| | Agent (메인) | SubAgent (보조) |
|---|---|---|
| **컨텍스트** | 사용자와 공유 | 독립 (분리) |
| **대화 상대** | 사용자 | 메인 Agent |
| **수명** | 세션 전체 | 작업 완료까지 |
| **결과** | 사용자에게 직접 | 메인에게 반환 |

### Agent Team — 여러 SubAgent를 팀처럼 운영

복잡한 작업에서는 SubAgent를 **역할별로** 나누어 팀으로 운영한다.

```
사용자 ──── Agent (메인: 총괄)
             │
             ├── SubAgent: Planner (계획)
             │    └── "어떤 파일을 수정해야 하나?"
             │
             ├── SubAgent: Builder (구현)
             │    └── "계획대로 코드 수정"
             │
             └── SubAgent: Reviewer (검토)
                  └── "수정된 코드 품질 검토"
```

**Agent Team = 역할이 다른 여러 SubAgent를 조합하여 복잡한 작업을 분업하는 구조.**

### 정리 비교표

| | Agent | SubAgent | Agent Team |
|---|---|---|---|
| **정의** | 작업을 수행하는 Claude 인스턴스 | 독립 컨텍스트의 보조 에이전트 | 여러 SubAgent를 팀처럼 운영 |
| **컨텍스트** | 메인 (사용자와 공유) | 분리 (각자 독립) | 각자 분리 |
| **용도** | 일반 작업 | 병렬/격리 작업 | 역할 분담 |
| **예시** | "버그 수정해줘" | "이 폴더만 보안 검토해줘" | Planner + Builder + Reviewer |

## SubAgent를 쓰는 3가지 이유

### 1. 컨텍스트 오염 방지

메인 세션에서 파일 50개를 검색하면, 그 결과가 전부 컨텍스트에 쌓인다. 앞에서 준 CLAUDE.md 규칙이 밀려나고, 검색 결과에 묻혀 중요한 지시를 잊을 수 있다.

```
❌ 메인에서 직접 탐색
   → grep 결과 2000줄 → 컨텍스트의 절반을 차지
   → CLAUDE.md 규칙이 밀려남 → 품질 저하

✅ SubAgent에게 탐색 위임
   → SubAgent가 2000줄을 읽고 소화
   → 메인에게는 "관련 파일 3개, 핵심 내용 요약" 5줄만 반환
   → 메인 컨텍스트는 깨끗
```

**SubAgent는 방화벽이다.** 도구 출력(tool output)이 메인 컨텍스트를 오염시키지 않게 막아준다.

### 2. 병렬 작업

SubAgent는 **동시에 여러 개**를 띄울 수 있다.

```
순차 실행 (SubAgent 없이):
  보안 검토 (5분) → 스타일 검토 (3분) → 테스트 검토 (4분) = 12분

병렬 실행 (SubAgent 3개):
  ┌── 보안 검토 (5분) ──┐
  ├── 스타일 검토 (3분) ─┤ = 5분 (가장 긴 것 기준)
  └── 테스트 검토 (4분) ─┘
```

독립적인 작업이라면 병렬로 띄우는 것이 효율적이다.

### 3. 역할 분리

같은 코드를 봐도 **관점에 따라 보는 것이 다르다.**

```
같은 코드를 세 관점으로:

SubAgent A (보안 관점):
  "SQL injection 가능성, 입력 검증 부족"

SubAgent B (성능 관점):
  "N+1 쿼리 문제, 캐시 미사용"

SubAgent C (가독성 관점):
  "변수명 불명확, 함수가 너무 길다"
```

하나의 메인 세션에서 세 관점을 다 가지려면 프롬프트가 길어지고 초점이 흩어진다. SubAgent로 나누면 각자 **자기 역할에만 집중**한다.

## SubAgent의 실제 사용

Claude Code에서 SubAgent는 **Agent 도구**를 통해 띄운다.

### 기본 구조

```
사용자: "이 프로젝트의 보안 취약점을 찾아줘"

메인 Agent의 판단:
  "파일이 많다. SubAgent에게 탐색을 맡기자."

→ Agent 도구 호출:
  - description: "보안 취약점 탐색"
  - prompt: "src/ 폴더의 모든 파일에서 SQL injection,
            XSS, 하드코딩된 비밀번호를 찾아라.
            파일 경로와 줄 번호를 보고해라."
```

### SubAgent 유형

Claude Code에는 **전문화된 SubAgent 유형**이 있다.

| 유형 | 역할 | 사용할 수 있는 도구 | 적합한 작업 |
|---|---|---|---|
| **general-purpose** | 범용 | 모든 도구 (검색, 편집, 실행) | 복잡한 멀티스텝 작업 |
| **Explore** | 코드 탐색 전문 | 탐색 중심 (Read, Grep, Glob, Bash 등 — 편집 도구 제외) | 코드베이스 조사, 파일 찾기 |
| **Plan** | 설계 전문 | 읽기 전용 | 구현 계획, 아키텍처 설계 |

**Explore**가 가장 자주 쓰인다. 코드를 탐색하고 분석하지만 **파일을 편집하지 않는다.** 편집(Edit, Write) 도구가 빠져 있어 안전하고 빠르다.

### 병렬 SubAgent 실전 예시

```
사용자: "코드 리뷰해줘"

메인 Agent의 판단:
  "세 관점에서 동시에 리뷰하자."

→ SubAgent 3개를 동시에 띄움:

  SubAgent 1 (Explore):
    "src/ 폴더에서 에러 핸들링이 빠진 곳을 찾아라"

  SubAgent 2 (Explore):
    "테스트 커버리지가 부족한 파일을 찾아라"

  SubAgent 3 (Explore):
    "하드코딩된 값(매직 넘버)을 찾아라"

→ 3개가 동시에 실행
→ 각자 결과를 메인에게 반환
→ 메인이 합쳐서 정리된 리뷰 보고
```

### SubAgent에게 좋은 지시를 주는 법

SubAgent는 메인 세션의 대화를 **전혀 모른다.** 새로 온 동료에게 일을 시키듯, 필요한 맥락을 모두 알려줘야 한다.

```
❌ 나쁜 지시:
  "아까 말한 버그 찾아줘"
  → SubAgent는 "아까"를 모른다

✅ 좋은 지시:
  "src/auth/login.ts 파일에서 비밀번호 검증 로직을 확인하라.
   validatePassword 함수가 빈 문자열을 통과시키는
   버그가 의심된다. 해당 함수의 코드와
   관련 테스트 파일을 찾아서 보고하라."
  → 파일 경로, 함수명, 의심 증상 모두 포함
```

**SubAgent 지시 작성 원칙:**
1. **구체적 위치**: 파일 경로, 함수명을 명시
2. **명확한 목표**: "찾아라", "분석하라", "비교하라" 등 동사로
3. **결과 형식**: "파일 경로와 줄 번호로 보고", "200자 이내 요약"
4. **독립 실행 가능**: 대화 맥락에 의존하지 않는 자기 완결적 지시

## worktree와 SubAgent의 관계

Ch.16에서 배운 **worktree**를 기억하는가? worktree는 Git의 분리 작업 폴더다.

SubAgent에는 **isolation: "worktree"** 옵션이 있다. 이 옵션을 주면 SubAgent가 **별도의 Git worktree에서 작업**한다.

### 두 가지 격리의 차이

```
SubAgent (기본):
  ┌────────────────────┐
  │  메인 세션          │     ┌─────────────────┐
  │  컨텍스트 A         │ ──→ │ SubAgent         │
  │                    │     │ 컨텍스트 B (분리)  │
  │  같은 파일 시스템   │ ←── │ 같은 파일 시스템   │
  └────────────────────┘     └─────────────────┘
  → 컨텍스트만 분리, 파일은 공유

SubAgent + worktree:
  ┌────────────────────┐
  │  메인 세션          │     ┌─────────────────┐
  │  컨텍스트 A         │ ──→ │ SubAgent         │
  │                    │     │ 컨텍스트 B (분리)  │
  │  원본 폴더         │     │ 별도 checkout (분리)│
  └────────────────────┘     └─────────────────┘
  → 컨텍스트도 분리, 파일도 분리
```

| | SubAgent (기본) | SubAgent + worktree |
|---|---|---|
| **컨텍스트** | 분리 | 분리 |
| **파일 시스템** | 메인과 공유 | 별도 checkout (같은 저장소, 다른 브랜치) |
| **적합한 작업** | 조사, 분석, 리뷰 | 실험적 코드 수정, 리팩터링 시도 |
| **장점** | 빠름, 가벼움 | 메인 코드에 영향 없음 |
| **단점** | 파일 수정 시 충돌 가능 | 느림, 자원 사용 많음 |

### 언제 worktree 격리를 쓰나

```
읽기만 하는 작업 → SubAgent 기본으로 충분
  예: "이 폴더의 구조를 분석해줘"
  예: "보안 취약점을 찾아줘"

파일을 수정하는 실험적 작업 → worktree 격리
  예: "이 함수를 두 가지 방식으로 리팩터링해봐"
  예: "새로운 아키텍처로 변환을 시도해봐"
```

**정리: SubAgent = 컨텍스트 격리. worktree = 파일 시스템 격리. 둘을 결합하면 완전한 격리.**

## SubAgent 운영 원칙

### 1. 쓸 때와 안 쓸 때

```
SubAgent가 필요한 경우:
  ✅ 파일 탐색이 3회 이상 필요한 조사
  ✅ 독립적인 작업 2개 이상을 병렬로
  ✅ 도구 출력이 많아서 메인 컨텍스트가 오염될 때
  ✅ 다른 관점으로 같은 코드를 검토할 때

SubAgent가 불필요한 경우:
  ❌ 파일 1-2개만 읽으면 되는 간단한 작업
  ❌ 결과를 바로 다음 단계에 써야 하는 순차 작업
  ❌ SubAgent에게 메인 대화의 맥락이 전부 필요한 작업
```

### 2. SubAgent를 과용하지 마라

SubAgent도 토큰을 쓴다. 간단한 검색을 굳이 SubAgent에게 시키면 오버헤드만 늘어난다.

```
❌ 과용:
  "config.json 읽어줘" → SubAgent 불필요. 직접 읽으면 된다.

✅ 적절한 사용:
  "200개 파일에서 deprecated API 사용을 전부 찾아줘" → SubAgent 적합
```

### 3. 결과 검증은 메인이 한다

SubAgent의 결과를 그대로 믿지 말고, 메인에서 확인한다.

```
SubAgent 보고: "취약점 3개 발견"
메인 Agent:
  → 보고된 파일 3개를 직접 확인
  → 실제로 취약점인지 판단
  → 사용자에게 최종 보고
```

**SubAgent는 조사원이고, 메인 Agent는 판사다.** 조사 결과를 받아서 최종 판단은 메인이 한다.

## 파이썬 연결: while 루프

agentic loop를 파이썬으로 표현하면 **while 루프**다.

### while 루프 — 조건이 참인 동안 반복

```python
# 기본 while 루프
count = 0
while count < 3:
    print(f"반복 {count}")
    count += 1
# 반복 0
# 반복 1
# 반복 2
```

**while**은 "~하는 동안 반복"이다. for 루프가 "정해진 횟수만큼 반복"이라면, while 루프는 **"조건이 충족될 때까지 반복"**이다.

### agentic loop를 while로 구현

```python
def agentic_loop(task):
    """에이전트 루프: 관찰 → 생각 → 행동 → 반복"""
    done = False
    max_turns = 10  # 무한 루프 방지
    turn = 0
    
    while not done and turn < max_turns:
        # ① 관찰
        state = observe(task)
        print(f"[관찰] {state}")
        
        # ② 생각
        action = think(state)
        print(f"[생각] 다음 행동: {action}")
        
        # ③ 행동
        result = act(action)
        print(f"[행동] 결과: {result}")
        
        # ④ 완료 판단
        done = is_complete(result)
        turn += 1
    
    if done:
        print("✅ 작업 완료!")
    else:
        print("⚠️ 최대 반복 도달, 사용자에게 보고")
```

### SubAgent를 while + 함수로 표현

```python
def subagent(name, task):
    """SubAgent = 독립 함수. 자기만의 변수(컨텍스트)로 작업"""
    context = []  # 독립된 컨텍스트 (메인과 분리)
    
    print(f"[{name}] 작업 시작: {task}")
    
    # 자기만의 agentic loop
    done = False
    while not done:
        result = do_work(task, context)
        context.append(result)  # 이 컨텍스트는 이 함수 안에서만
        done = check_done(result)
    
    # 결과만 반환 (컨텍스트는 반환하지 않음)
    summary = summarize(context)
    print(f"[{name}] 완료. 요약: {summary}")
    return summary


def main_agent(user_request):
    """메인 Agent가 SubAgent 3개를 띄우는 구조"""
    
    # 3개 SubAgent 실행 (개념적 표현 — Claude Code에서는 실제 병렬, 파이썬에서는 순차)
    result_a = subagent("보안검토", "SQL injection 찾기")
    result_b = subagent("성능검토", "N+1 쿼리 찾기")
    result_c = subagent("스타일검토", "변수명 규칙 검사")
    
    # 메인이 합쳐서 최종 보고
    print(f"\n=== 최종 보고 ===")
    print(f"보안: {result_a}")
    print(f"성능: {result_b}")
    print(f"스타일: {result_c}")
```

**SubAgent = 독립 함수.** 자기만의 지역 변수(컨텍스트)로 작업하고, 결과만 반환한다. 메인 함수의 변수(컨텍스트)를 오염시키지 않는다.

### while True와 break — 더 자연스러운 agentic loop

```python
# while True + break 패턴
while True:
    user_input = input(">>> ")
    
    if user_input == "exit":
        break  # 루프 탈출
    
    response = process(user_input)
    print(response)
```

실제 agentic loop는 이 `while True` + `break` 패턴에 가깝다. "언제 끝날지 모르지만, 완료 조건을 만족하면 멈춘다."

## 왜 이렇게 동작하는가

SubAgent의 원리는 컴퓨터 과학의 **프로세스 격리**와 같다.

운영체제에서 프로세스 A가 프로세스 B의 메모리를 건드리지 못하는 것처럼, SubAgent A의 컨텍스트는 SubAgent B와 메인 Agent의 컨텍스트에 영향을 주지 않는다.

```
운영체제:
  프로세스 A → 자기 메모리만 사용 → 결과를 IPC로 전달
  프로세스 B → 자기 메모리만 사용 → 결과를 IPC로 전달
  메인 프로세스 → 두 결과를 받아서 처리

Claude Code:
  SubAgent A → 자기 컨텍스트만 사용 → 결과를 메인에게 반환
  SubAgent B → 자기 컨텍스트만 사용 → 결과를 메인에게 반환
  메인 Agent → 두 결과를 합쳐서 보고
```

이 격리 덕분에:
- **안전하다**: SubAgent가 실수해도 메인 컨텍스트에 영향 없다
- **확장 가능하다**: 필요하면 SubAgent를 더 추가할 수 있다
- **효율적이다**: 각 SubAgent가 자기 역할에만 집중한다

Ch.9에서 배운 **컨텍스트 격리**를 기억하는가? SubAgent는 그 격리를 **자동으로** 구현하는 도구다.

## 다음 챕터로

Agent와 SubAgent로 작업을 분업하는 방법을 배웠다. 하지만 이 모든 작업에는 **비용**이 따른다. Agent가 파일을 읽을 때마다, SubAgent가 각자 작업할 때마다, **토큰**이 소비된다.

Ch.19에서는 **토큰과 비용** — 클로드를 쓸 때 실제로 무엇이 소비되는지, 어떻게 줄이는지를 배운다.

## 이 챕터 핵심 3줄
- **agentic** = 모델이 도구를 부르고 판단하는 방식. 핵심은 **관찰 → 생각 → 행동 → 반복**의 agentic loop
- **SubAgent** = 독립 컨텍스트의 보조 에이전트. **컨텍스트 오염 방지 + 병렬 작업 + 역할 분리** 3가지가 핵심. worktree를 결합하면 파일 시스템까지 격리
- **SubAgent는 함수다** — 자기만의 지역 변수(컨텍스트)로 작업하고 결과만 반환. 메인의 변수를 오염시키지 않는다
