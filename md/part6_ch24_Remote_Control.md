# Ch.24 Remote Control & 파이프라인 설계

> **이 챕터를 마치면**: Remote Control로 다른 기기에서 세션을 이어받을 수 있고, persistent thread의 개념을 이해한다. 트리거 → 실행 → 채널 수신의 파이프라인을 설계할 수 있다.

---

## 배경: 어디서든 클로드와 연결되다

Ch.22에서 **Channels**(외부 이벤트 수신), Ch.23에서 **Scheduled Tasks**(시간 트리거 실행)를 배웠다. 이제 마지막 퍼즐 — **Remote Control**(원격 조종)과 이 모든 것을 연결하는 **파이프라인 설계**다.

Remote Control은 간단한 아이디어다:

> 집 컴퓨터에서 돌고 있는 클로드 세션을, 카페에서 폰으로 조종한다.

코드는 항상 내 컴퓨터에서 실행되고, **채팅 메시지만** 인터넷을 통해 주고받는다.

> **💭 생각의 흐름**
>
> **문제** — 터미널에서 큰 리팩토링을 시작했는데, 퇴근 시간이 됐다. 세션을 닫으면 처음부터 다시 해야 한다
> **질문** — "집에서 계속 돌리면서, 폰으로 진행 상황을 확인하고 지시할 수 없나?"
> **발견** — `claude --remote-control`로 세션을 시작하면, claude.ai/code나 앱에서 원격 연결 가능. 코드는 집에서 돌고, 나는 카페에서 조종
> **결론** — Remote Control = **"내가 어디 있든 세션은 계속된다"**

## Remote Control이란

### 한 줄 정의

**Remote Control = 로컬에서 실행 중인 세션을 다른 기기에서 원격으로 조종하는 기능.**

```
동작 구조:

집 컴퓨터 (로컬)                     카페 (원격)
┌──────────────────┐              ┌──────────────┐
│ Claude Code 세션  │←── TLS ───→│ claude.ai/code│
│ · 코드 실행       │  암호화 중계  │ 또는 모바일 앱 │
│ · 파일 읽기/쓰기  │              │ · 메시지 입력  │
│ · git 작업        │              │ · 결과 확인    │
└──────────────────┘              └──────────────┘

핵심: 코드는 항상 로컬에서 실행. 메시지만 왕복.
```

### 세션 시작 방법

3가지 방법이 있다:

```bash
# 방법 1: 서버 모드 — 터미널을 서버로 유지, 다중 연결 대기
claude remote-control

# 방법 2: 인터랙티브 모드 — 로컬에서도 입력 가능 + 원격 연결
claude --remote-control

# 방법 3: 기존 세션에서 활성화
/remote-control
# 또는 줄여서
/rc
```

**방법 2**가 가장 실용적이다. 로컬에서 작업하다가, 자리를 비울 때 폰으로 이어받을 수 있다.

### 다른 기기에서 연결하는 법

```
방법 A: 터미널에서 스페이스바 → QR코드 표시 → 폰으로 스캔
방법 B: claude.ai/code에서 세션 목록 → 초록 점(온라인) 세션 클릭
방법 C: 세션 URL을 직접 공유
```

### 요구 사항

```
필수:
  · Claude Code v2.1.51 이상
  · claude.ai 로그인 (API 키만으로는 불가)
  · Pro, Max, Team, Enterprise 플랜

Team/Enterprise:
  · 관리자가 admin settings에서 Remote Control 토글 활성화 필요
```

## `claude --teleport` 🆕 — 웹/모바일에서 시작한 세션을 터미널로 끌어오기

Remote Control이 "**로컬 세션을 원격으로 조종**"이라면, Teleport는 **반대 방향**이다.

### 시나리오

```
아침 출근길 (폰):
  claude.ai/code나 iOS 앱에서 작업 시작
  "이 레포에서 마이그레이션 계획 세워줘"
  → 30분간 폰으로 대화

사무실 도착 (터미널):
  $ cd my-repo
  $ claude --teleport
  → 폰에서 하던 세션이 터미널로 그대로 넘어옴
  → 대화 히스토리, 컨텍스트, 계획 모두 유지
```

### 왜 필요한가

전에는 이동 중 웹/앱에서 작업하다가 자리에 앉으면 **세션이 단절**됐다. 터미널에서 처음부터 다시 시작하거나, Remote Control로 원격 연결만 가능했다(코드 실행은 클라우드에서 계속됨).

Teleport는 **클라우드 세션의 실행 권한 자체를 로컬로 옮긴다**:
- 이후 파일 작업은 내 컴퓨터에서 실행 (속도 ↑)
- 로컬 MCP 서버, Skills, Hooks 모두 적용 (능력 ↑)
- 캐시는 그대로 유지 (비용 ↓)

### 사용법

```bash
# 웹/앱에서 작업 중인 세션을 이 터미널로 가져오기
claude --teleport

# 특정 세션 ID 지정
claude --teleport <session-id>
```

세션 목록은 `claude.ai/code`에서 확인할 수 있다.

### Remote Control과의 차이

| | **Remote Control** | **Teleport** 🆕 |
|---|---|---|
| 실행 위치 | 로컬 유지 | **클라우드 → 로컬로 이동** |
| 시작 지점 | 로컬에서 시작 | **웹/앱에서 시작** |
| 용도 | 외출 중에 조종 | 외출 끝나서 자리 앉음 |
| 비유 | 원격 데스크톱 | **순간이동** |

## `/desktop` 🆕 — 터미널 → Desktop 앱 핸드오프

터미널에서 작업 중인데 **시각적 diff가 필요해지는** 순간이 있다. 여러 파일 변경을 한눈에 보고 싶거나, 동시 세션을 나란히 보고 싶거나.

### 동작

```
터미널에서:
  > /desktop

  → Desktop 앱이 자동 실행
  → 현재 세션이 Desktop으로 이어받음
  → 터미널은 자동 종료 (옵션)
```

### 언제 쓰나

- 파일 수정이 5개 이상으로 늘어나서 **시각적 diff 리뷰**가 필요할 때
- 다른 세션과 **병렬로 놓고** 비교하고 싶을 때
- 긴 작업을 **백그라운드로 돌리고** 다른 작업을 시작하고 싶을 때

### 핸드오프 방향 한눈에

```
                    ┌───────────────┐
        /desktop →  │  Desktop 앱   │
                    │ (시각 diff,    │
                    │  다중 세션)    │
                    └───────────────┘
                           ▲
                           │
   ┌───────────────┐      │      ┌───────────────┐
   │  터미널 CLI    │ ─────┼────► │  웹/모바일     │
   │               │ ◄────┘      │ (claude.ai/   │
   │               │   --teleport │   code, iOS)  │
   └───────────────┘              └───────────────┘
         ▲                              │
         │    Remote Control            │
         └──────────────────────────────┘
```

**세션은 하나, Surface는 자유롭게 이동.** 이게 2026년 이후 Claude Code의 기본 설계 철학이다.

## Persistent Thread — 지속 세션

Cowork에서 "진행 중인 작업 상태를 유지"하는 개념이 persistent thread(지속 세션)다.

### Remote Control의 지속성

```
Remote Control 세션의 특성:
  · 네트워크가 끊겨도 머신이 돌아오면 자동 재연결
  · 원격 기기를 바꿔도 (폰 → 태블릿 → PC) 세션은 계속
  · 대화 히스토리가 유지됨

제한:
  · 로컬 터미널을 닫으면 세션 종료
  · 네트워크 끊김 ~10분이면 타임아웃으로 프로세스 종료
```

### Scheduled Tasks의 세션 격리

```
Scheduled Tasks는 지속 세션이 아니다:
  · 각 실행은 별도 세션으로 생성
  · 이전 실행의 대화 히스토리를 공유하지 않음
  · Cloud 작업: 매번 GitHub 레포를 fresh clone

→ 각 실행이 독립적이므로 self-contained 프롬프트가 중요 (Ch.23)
```

### 비유

```
Remote Control: 전화 통화
  → 끊지 않는 한 대화가 이어진다
  → 기기를 바꿔도 같은 대화

Scheduled Tasks: 매일 아침 온 편지
  → 매 편지는 독립적
  → 이전 편지 내용을 자동으로 기억하지 않음
```

## 실전 파이프라인 3가지

Part 6에서 배운 모든 것을 연결하는 파이프라인을 설계해보자.

### 파이프라인 1: git push → 자동 리뷰 → Slack 알림

```
트리거: git push (GitHub)
  │
  ▼
실행: Cloud Scheduled Task 또는 GitHub Actions
  · Claude가 코드 변경사항을 리뷰
  · 보안 이슈, 스타일 위반, 잠재 버그 체크
  │
  ▼
수신: Slack 커넥터를 통해 #eng-reviews 채널에 포스팅
  · "PR #42: 보안 이슈 1건, 스타일 위반 2건"
```

**설정 방법:**

Cloud Scheduled Task 방식:
1. claude.ai/code/scheduled에서 New Task 생성
2. GitHub 레포 연결
3. 프롬프트: "최근 push된 커밋을 리뷰하고 결과를 Slack #eng-reviews에 포스팅"
4. Connectors에서 Slack MCP 추가
5. 빈도: Hourly 또는 push 이벤트 연동

### 파이프라인 2: 경쟁사 모니터링 → 주간 브리프 → 공유

```
트리거: 매주 월요일 오전 9시 (Scheduled Task)
  │
  ▼
실행: Desktop 예약 작업
  · PROJECTS/경쟁사-분석/brief.md 읽기
  · 경쟁사 블로그/뉴스 분석
  · TEMPLATES/weekly-brief-template.md 형식으로 작성
  │
  ▼
저장: CLAUDE-OUTPUTS/YYYY-MM-DD_경쟁사브리프.md
  │
  ▼
공유: 이메일 API로 팀에 발송
  또는 Slack 커넥터로 #competitive-intel 채널에 포스팅
```

**이 파이프라인의 핵심:**
- **Ch.21의 폴더 구조**가 입력 (PROJECTS/brief.md)
- **Ch.23의 self-contained 프롬프트**가 실행을 보장
- **Ch.22의 Channel**이 결과를 외부에 전달

### 파이프라인 3: 고객 요청 → 분류 → 담당자 알림

```
트리거: 고객 메시지가 Telegram으로 도착 (Channel)
  │
  ▼
실행: Claude Code 세션 (Channel 활성화)
  · 메시지를 읽고 분류: 버그 / 기능 요청 / 문의 / 긴급
  · PROJECTS/고객-요청/분류-기준.md 참고
  │
  ▼
수신: 분류 결과에 따라
  · 긴급 → Slack DM으로 담당자에게 즉시 알림
  · 버그 → GitHub Issue 생성
  · 문의 → Telegram으로 자동 답장
```

**이 파이프라인의 핵심:**
- **Channel**(Telegram)이 이벤트를 세션에 push
- **분류 로직**은 프롬프트와 brief.md로 정의
- **결과물**이 여러 채널로 분기

### 파이프라인 설계 원칙

```
좋은 파이프라인의 3가지 원칙:

1. 각 단계가 self-contained
   · 이전 단계의 출력이 다음 단계의 입력
   · 중간에 실패해도 처음부터 다시 시작 가능

2. 실패 시 행동이 정의됨
   · "데이터가 없으면 → 스킵하고 기록"
   · "에러가 나면 → Slack으로 알림"

3. 결과가 관찰 가능
   · 실행 이력이 남음 (사이드바, 로그)
   · /cost로 비용 추적
   · 실패 시 알림
```

## 왜 이렇게 동작하는가

Part 6 전체를 관통하는 원리는 **분리(separation)**와 **연결(connection)**이다.

**분리:**
- 작업 정의(brief.md)와 실행(Claude)이 분리된다 → 같은 정의로 매번 일관된 결과
- 로컬 실행과 원격 조종이 분리된다 → 어디서든 접근 가능
- 이벤트 소스와 반응 로직이 분리된다 → 다양한 트리거를 같은 로직에 연결

**연결:**
- about-me.md + brief.md + template → 컨텍스트 파이프라인
- Channel + Scheduled Task + Remote Control → 이벤트 파이프라인
- 입력 → 처리 → 출력 → 알림 → 피드백 → 개선

이것은 소프트웨어 설계에서 말하는 **관심사의 분리(Separation of Concerns)**와 같다. 각 부분이 자기 역할만 하고, 인터페이스로 연결된다.

## 이 챕터 핵심 3줄
- **Remote Control**: 로컬 세션을 다른 기기에서 원격 조종. `claude --remote-control`로 시작, claude.ai/code나 앱에서 연결. 코드는 항상 로컬, 메시지만 왕복
- **Surface 이동 3종 🆕**: `--teleport`(웹/앱 → 터미널로 세션 이동), `/desktop`(터미널 → Desktop 앱 핸드오프), `--remote-control`(로컬 유지하며 원격 조종). **세션은 하나, Surface는 자유**
- **파이프라인 설계**: 트리거(시간/이벤트/수동) → 실행(Desktop/Cloud/세션) → 결과(파일/Slack/GitHub). 핵심은 분리와 연결
