## 한 줄 판별법

가장 빠른 판단 기준. **"무엇이 필요한가"에 따라 도구가 결정**된다.

| 나는 이런 게 필요하다 | → 선택 |
|---|---|
| "매번 같은 형식으로 만들고 싶다" | **Skill** |
| "Slack, DB 같은 바깥 도구에 닿아야 한다" | **MCP / Connector** |
| "팀 전체가 같은 구성으로 바로 쓰게 하고 싶다" | **Plugin** |
| "실수하면 안 되고 자동 검사가 필요하다" | **Hook** |

### 판별 흐름도

더 정확하게 결정하려면 이 순서로 질문한다:

```
1. 외부 서비스에 접근해야 하나?
   예 → MCP / Connector
   아니오 → 다음

2. 자동 검사/강제가 필요한가?
   예 → Hook
   아니오 → 다음

3. 반복되는 절차/형식인가?
   예 → Skill
   아니오 → 다음

4. 여러 도구를 묶어 팀에 배포해야 하나?
   예 → Plugin
   아니오 → CLAUDE.md에 규칙만 적어도 충분
```

**Hook이 MCP보다 먼저**인 이유: 외부 연결이 필요 없는데 자동 검사만 필요한 경우가 많다. Edit 후 자동 lint, Bash 전 위험 명령 검사 — 이건 Hook만으로 충분하다.

## 5가지 도구 비교표

| | Skill | MCP | Connector | Hook | Plugin |
|---|---|---|---|---|---|
| **한 마디** | 매뉴얼 | 연결 | 케이블 | 자동 검사 | 세트 |
| **해결하는 문제** | 반복 절차 | 외부 접근 | 서비스 연결 | 실수 방지 | 팀 배포 |
| **적용 범위** | 특정 작업 | 특정 서비스 | 특정 서비스 | 모든 작업 | 팀 전체 |
| **토큰 영향** | 트리거 시만 | 큼 (도구 출력) | 큼 (도구 출력) | 없음 | 포함된 것에 따라 |
| **설치** | 폴더 만들기 | `claude mcp add` | MCP 통해 | settings.json | `claude plugin install` |
| **비유** | 레시피 | USB 규격 | USB 케이블 | 안전장치 | 공구함 |

## 같은 문제를 세 방식으로 풀어보기

### 상황: "PR을 올리기 전에 코드 품질을 확인하고 싶다"

#### 방법 1: Hook으로 풀기

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash(git push)",
      "hooks": [{
        "type": "command",
        "command": "npm test && npm run lint"
      }]
    }]
  }
}
```

**장점**: 자동으로 강제. 사용자가 잊어도 실행됨
**단점**: 유연하지 않음. 항상 같은 검사만 실행
**적합**: 항상 동일한 검사를 강제하고 싶을 때

#### 방법 2: Skill로 풀기

```markdown
---
name: pre-pr-review
description: PR 전 코드 품질 체크
triggers: ["PR", "풀리퀘", "push 전"]
---

# PR 전 체크리스트

1. `npm test` 실행 → 전체 통과 확인
2. `npm run lint` → 경고 0개 확인
3. 변경 파일 diff 검토 → 불필요한 변경 없는지
4. 커밋 메시지 컨벤션 확인
5. 결과 요약 보고
```

**장점**: 유연함. 상황에 따라 절차를 조절 가능
**단점**: 자동이 아님. 사용자가 "리뷰해줘"라고 말해야 실행
**적합**: 상황에 따라 다른 절차가 필요할 때

#### 방법 3: Skill + Hook + MCP (Plugin으로 묶기)

```
pr-quality-plugin/
├── skills/pre-pr-review/    # Skill: 리뷰 절차
├── hooks/auto-lint.json     # Hook: push 전 자동 lint
├── mcp/sentry.json          # MCP: Sentry 에러 체크
└── plugin.json              # 메타데이터
```

**장점**: 완전 자동화 + 유연성 + 외부 데이터
**단점**: 설정이 복잡. 팀 배포가 아니면 과함
**적합**: 팀 전체에 배포해야 할 때

### 핵심 교훈

```
Hook만  = 강제는 되지만 유연하지 않음
Skill만 = 유연하지만 강제가 안 됨
둘 다   = 강제 + 유연 → 대부분 이걸로 충분
Plugin  = 팀 배포가 필요할 때만
MCP     = 외부 서비스 접근이 필요할 때만
```

**대부분의 개인 사용자에게는 Skill + Hook 조합이면 충분하다.** MCP와 Plugin은 정말 필요할 때만 추가한다.

## 도입 우선순위: 누가 쓰느냐에 따라 다르다

같은 도구라도 **누가 쓰느냐에 따라** 도입 순서가 달라진다.

### 1단계: 개인 사용자

혼자 쓰는 개발자. 이것부터 시작:

```
✅ CLAUDE.md        — 프로젝트 규칙
✅ rules/           — 경로별 세부 규칙
✅ code-review Skill — 첫 번째 Skill
✅ Hook (lint 자동화) — 첫 번째 Hook

🔲 MCP              — 외부 서비스 필요할 때
🔲 Plugin            — 아직 불필요
```

**핵심**: CLAUDE.md + rules + Skill 하나 + Hook 하나. 이 네 개면 충분히 강력하다.

### 2단계: 비개발 실무자

기획자, PM, 마케터가 Claude(Cowork)을 쓸 때:

```
✅ Cowork 폴더 구조   — 프로젝트 컨텍스트
✅ context files      — 브랜드 가이드, 용어집 등
✅ AskUserQuestion    — 클로드가 물어보게 설정
✅ scheduled tasks    — 주간 보고서 자동 생성 등

🔲 Skill              — 반복 형식이 생기면
🔲 Plugin             — 팀 표준이 필요하면
```

**핵심**: 컨텍스트 파일 + 자동화된 반복 작업. 코딩 지식 불필요.

### 3단계: 팀 리드

팀 전체의 Claude 사용을 관리할 때:

```
✅ CLAUDE.md 공유     — 팀 표준 규칙 (Git에 포함)
✅ path-specific rules — 영역별 규칙
✅ Skill provisioning  — 팀 표준 Skill 배포
✅ Plugin 기준         — 어떤 Plugin을 쓸지 결정
✅ channels           — 팀 커뮤니케이션 채널
✅ audit              — 사용 현황 추적

⚠️ 권한 관리          — 누가 뭘 할 수 있는지
```

**핵심**: 표준화 + 배포 + 감사. Plugin이 이 단계에서 빛난다.

### 도입 우선순위 요약

```
개인     → CLAUDE.md → rules → Skill → Hook → (MCP) → (Plugin)
비개발자 → 컨텍스트 파일 → 자동화 → (Skill) → (Plugin)
팀 리드  → 공유 규칙 → Skill 배포 → Plugin 기준 → 감사
```

**공통 원칙: 간단한 것부터 시작하고, 필요할 때만 다음 단계로.**

## 파이썬 연결: 조건문으로 판단하기

"어떤 도구를 쓸까"라는 판단은 결국 **조건 분기**다. 파이썬의 if/elif/else로 표현하면:

```python
def choose_tool(need):
    if need == "외부 서비스 접근":
        return "MCP / Connector"
    elif need == "자동 검사/강제":
        return "Hook"
    elif need == "반복 절차/형식":
        return "Skill"
    elif need == "팀 배포":
        return "Plugin"
    else:
        return "CLAUDE.md에 규칙만"
```

### 실제 사용 예시

```python
# 여러 조건이 겹칠 수도 있다
needs = ["반복 절차", "자동 검사", "외부 서비스"]

tools = []
for need in needs:
    if need == "외부 서비스":
        tools.append("MCP")
    elif need == "자동 검사":
        tools.append("Hook")
    elif need == "반복 절차":
        tools.append("Skill")

print(f"필요한 도구: {tools}")
# → 필요한 도구: ['Skill', 'Hook', 'MCP']

# 3개 이상이면 Plugin으로 묶는 것도 고려
if len(tools) >= 3:
    print("Plugin으로 묶는 것을 고려하세요")
```

**판단도 코드처럼 구조화**할 수 있다. 조건이 명확하면 선택이 쉬워진다.

## 왜 이렇게 동작하는가

이 다섯 가지 도구가 따로 존재하는 이유는 소프트웨어 엔지니어링의 **단일 책임 원칙(Single Responsibility Principle)**이다.

- Hook = **검증** 책임
- Skill = **절차** 책임
- MCP = **연결** 책임
- Connector = **특정 서비스 연결** 책임
- Plugin = **배포** 책임

하나의 도구가 모든 것을 하면 복잡해지고 깨지기 쉽다. **각자 하나의 책임**만 가지면 조합이 유연해진다.

이건 프로그래밍의 함수 설계와 같다. "한 함수는 한 가지 일만 한다"는 원칙. 함수를 작게 만들면 조합이 자유로워지듯, 도구를 작게 만들면 조합이 자유로워진다.
