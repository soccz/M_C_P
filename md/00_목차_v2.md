## Part 2: 클로드의 기억과 설정

### Ch.5 클로드의 기억
- 왜 매번 같은 말을 반복하게 되는가
- **Memory 시스템**: 글로벌 vs 프로젝트 (user/feedback/project/reference 타입)
- **CLAUDE.md** — 프로젝트 운영 계약서
  - 항상 읽히는 기본 단서, 실전 예제
  - 최소 CLAUDE.md 템플릿 (명령/경계/규칙/검증)
- **.claude/ 폴더 전체 구조**:
  ```
  .claude/
  ├── settings.json
  ├── settings.local.json
  ├── rules/
  │   ├── api.md
  │   ├── testing.md
  │   └── frontend.md
  └── skills/
  ```
- **초보자 필수 파일 8종**:
  about-me.md, working-rules.md, brand-voice.md, glossary.md,
  CLAUDE.md, plan.md, handoff.md, template/*.md
- 🐍 문자열과 f-string — 프롬프트도 결국 텍스트 조합

### Ch.6 클로드의 설정
- **settings.json** 완전 해부:
  - `$schema` — 검증 기준 주소
  - `permissions.allow` / `permissions.deny`
  - 환경변수, 모델 고정
- **설정 우선순위**: Managed > Command Line > Local > Project > User
- **권한 모드 4종 비교**:
  | 모드 | 목적 | 장점 | 주의 |
  |------|------|------|------|
  | default | 기본 승인 포함 | 안전 | 승인 피로 |
  | plan | 계획 우선 | 실수 방지 | 속도 느림 |
  | acceptEdits | 편집 수락 활성화 | 빠름 | 파일 위험 |
  | bypassPermissions | 권한 건너뛰기 | 최고 속도 | 위험 |
- **allowlist vs deny** — 허용 목록이 먼저, 위험한 건 deny로
- **scope(적용 범위)** 개념 — project scope vs organization scope
- 🐍 조건문(if/else) — 권한 관리도 결국 조건 분기

### Ch.7 클로드의 규칙
- **CLAUDE.md vs rules/ vs settings.json** — 세 파일이 왜 따로 존재하는가
  - CLAUDE.md: 프로젝트 운영 계약
  - rules/: 경로별 세부 규칙 (path-specific rules)
  - settings.json: 실제 허용/금지/차단 경계
- **path-specific rules 실전**:
  ```yaml
  ---
  paths:
    - "src/api/**/*.ts"
  ---
  # API 규칙
  - validate all request bodies with zod
  - return { data, error }
  - never expose internal stack traces
  ```
- 규칙을 파일별로 쪼개야 하는 이유
- **파일 네이밍 규칙**: `YYYY-MM-DD_topic_type.md`

### Ch.8 클로드의 습관 — Hooks
- Hook이란: "이런 일이 생기면 이렇게 해"
- **Hook 4가지 타입**:
  - command hook (셸 명령)
  - prompt-based hook (프롬프트 검사)
  - agent-based hook (에이전트 자동 검사)
  - HTTP hook (외부 웹 호출)
- **hooks 구조** (settings.json 내): PreToolUse, PostToolUse, matcher
- 실전: rm 실행 전 확인 훅 만들기
- 실전: Edit 후 자동 lint 훅
- 🐍 함수(def) — Hook도 결국 "이벤트 → 함수 호출"
- Hook과 allowlist의 관계

## Part 3: 대화의 기술 — 3가지 엔지니어링

### Ch.9 Context Engineering
- **프롬프트 엔지니어링보다 컨텍스트 엔지니어링이 더 중요한 이유**
- "무엇을 넣을 것인가"보다 **"무엇을 넣지 않을 것인가"**
- **Attention Budget(주의 예산)** — Claude의 집중력은 유한하다
- **Progressive Disclosure** — 필요한 정보만 필요할 때 보여주기
- **템플릿 vs 보조 자료 파일(supporting file) 분리**:
  - 템플릿: 형식 고정 (빈 구조)
  - 예시: 같은 결과의 감각 제공
  - 보조 자료: 참고 세부 지식
- **tool output이 컨텍스트를 무너뜨린다** — 도구 출력이 토큰을 먹는 문제
- 좋은 context 분리 실전 예시 (everything.md vs 분리된 구조)
- 🐍 딕셔너리 — JSON 구조를 이해하면 Context가 보인다

### Ch.10 Prompt Engineering
- 여전히 중요한 이유 — 컨텍스트의 시작점
- **나쁜 / 보통 / 좋은 프롬프트** 3단 비교 (5가지 상황)
- **제약형 프롬프트 템플릿**:
  ```
  <goal> 무엇을 만들고 싶은가 </goal>
  <context> 어떤 파일과 자료를 읽어야 하는가 </context>
  <constraints> 포함할 것, 피할 것, 길이 제한 </constraints>
  <output_format> 표/prose/JSON/체크리스트 </output_format>
  <done_when> 두꺼을 보면 완료다고 볼 수 있는가 </done_when>
  <if_unsure> 불확실하면 질문할지, 오른다고 말할지 </if_unsure>
  ```
- **프롬프트 9원칙**:
  1. 모호한 활용사보다 조건이 먼저
  2. 긴 컨텍스트에서는 자료를 위에 두고 질문을 아래에
  3. 범용사보다 예시
  4. 200자 제한과 guardrail
  5. 사고 확장(Extended Thinking) 활용
  6. 불확실성 표현 허용
  7. 한 번에 다시 기도 좋지만 나누기가 낫다
  8. 메타적 동보다 구조화 출력(structured output)
  9. JSON 스키마나 출력 형식 No preamble, return only JSON
- **실전 프롬프트 템플릿 5종**: Context-first, 버그수정, Cowork 브리프, 리뷰, 긴문 컨텍스트 분석

### Ch.11 Harness Engineering
- "프롬프트가 대화의 문장이라면, 하네스는 그 문장이 실제로 작동하는 작업장"
- **하네스 = 5가지 최소 정의 묶음**:
  1. CLAUDE.md에 기본 규칙 적기
  2. settings.json에 허용/차단 범위 적기
  3. plan.md로 큰 작업을 먼저 쓰기
  4. hooks로 테스트나 포맷 자동화하기
  5. handoff.md에 세션 끝 상태를 남기기
- **하네스의 3가지 격리**:
  - 컨텍스트 격리 (subagent나 별도 세션)
  - 파일 실체 격리 (worktree나 별도 output 폴더)
  - 승인 격리 (permissions와 hook)
- 시작 프롬프트보다 시작 루틴이 더 중요한 이유
- **Claude Code 운영 7원칙**:
  1. plan-before-edit
  2. 컨텍스트 위치는 항상 확인
  3. 수정 전에 범위, 완료 기준, 검증 명령을 같이
  4. hooks로 테스트나 포맷 자동화
  5. worktree를 분리 작업 폴더로
  6. resume을 의식적으로 — continue/resume 구분
  7. 비용은 토큰 충량이 아니라 cache read/write, tool output, 세션 길이, 테스트 로그 전체를 같이
- **Compact Instructions / Session Memory 템플릿**

## Part 4: 클로드에게 능력을 주다

### Ch.12 기술 가르치기 — Skills
- **Skill이란**: 폴더 하나가 지식 패키지
- **Skill 구조**:
  ```
  your-skill-name/
  ├── SKILL.md              # 필수
  ├── scripts/              # 선택
  ├── references/           # 선택
  └── assets/               # 선택
  ```
- **SKILL.md 작성법**: YAML 프론트매터 + 마크다운 명령어
- **Progressive Disclosure 3단계**: 프론트매터 → 본문 → 연결 파일
- **3가지 사용 사례 카테고리**:
  1. 문서 및 에셋 생성 (frontend-design 등)
  2. 워크플로우 자동화 (skill-creator 등)
  3. MCP 강화 (sentry-code-review 등)
- **성공 기준 정의**: 정량적(90% 트리거, X번 도구 호출) + 정성적(사용자 수정 없이 완료)
- **Skill 테스트 / 반복 / 배포**
- 나만의 첫 Skill 만들기 실전
- 🐍 반복문(for/while) — 반복 작업을 자동화하는 원리
- **Skill + MCP 커넥터 패턴** — 주방 비유(MCP=도구+재료, Skill=레시피)

### Ch.13 도구 연결하기 — MCP
- **MCP(Model Context Protocol)란**: 외부 서비스/도구를 Claude에 연결하는 표준 인터페이스
- MCP 서버 설치하고 연결하기
- `claude mcp serve` / `claude mcp add-json` / `claude mcp remove`
- 실전: 파일시스템 MCP → 로컬 파일 읽기
- 실전: 공개 MCP 레지스트리에서 하나 더 추가
- **MCP가 토큰을 많이 쓰는 이유** — tool output 문제
- MCP는 "많을수록 좋다"가 아니라 **최소 연결, 실제 필요한 것만**

### Ch.14 Plugins & Connectors
- **Plugin이란**: 역할 패키지 — skills, commands, agents, hooks, MCP LSP 등을 묶은 배포 패키지
- **Cowork plugin vs Claude Code plugin** — 출발점/UX/개발 방향 차이
  - Cowork plugin: 비개발자 UX, Customize → plugin marketplace
  - Claude Code plugin: 개발 환경 확장, commands/LSP/hooks/MCP agents
- `claude plugin list/install/update/enable/disable`
- **Connector란**: 서비스 연결 인터페이스
  - **local desktop extension** — 같은 컴퓨터 내 연결
  - **remote web connector** — 클라우드 서비스 연결 (Asana, Linear, Notion 등)
- Plugin과 Connector의 관계: Plugin은 묶음, Connector는 단일 연결

### Ch.15 비교와 선택
- **한 줄 판별법**:
  - "매번 같은 형식으로 하고 만들고 싶다" → **Skill**
  - "Slack, Drive, DB 같은 바깥 도구에 닿아야 한다" → **MCP나 Connector**
  - "팀 전체가 같은 구성으로 바로 쓰게 하고 싶다" → **Plugin**
  - "실수를면 안 되고 승인이나 자동 검사가 필요하다" → **Hook 먼저**
- **같은 문제를 세 방식으로 풀어보기** 실습
- **도입 우선순위**: 개인 → 비개발 실무자 → 팀 리드 단계별
  - 개인: CLAUDE.md + rules + code-review + hookify
  - 비개발: Cowork 폴더 + context files + AskUserQuestion + scheduled tasks
  - 팀: CLAUDE.md 공유 + path-specific rules + skill provisioning + plugin 기준 + channels + audit

## Part 5: 클로드의 작업 방식

### Ch.16 세션 운영
- **세션이란**: Claude와의 하나의 대화 흐름
- **continue / resume** — 같은 컨텍스트를 이어가기
- **worktree** — Git 분리 작업 폴더, 병렬 작업의 핵심
- **/compact** — 컨텍스트 압축, 언제 써야 하나
- **resume hygiene** — 세션 이어받기 정리 습관
- **상태 표시줄(status line)** — 컨텍스트 사용량, 비용, Git 브랜치 실시간 확인
- **세션 운영 체크리스트**:
  - 시작 전: CLAUDE.md, rules, plan.md 확인
  - 수정 전: 범위, 완료 기준, 검증 명령을 같이
  - 완료 시: handoff.md 또는 decision-log.md 갱신

### Ch.17 계획과 인수인계
- **plan.md** — "앞으로 할 일을 줄인다"
  - 목표, 완료기준, 단계, 체크리스트 구조
  - 좋은 plan.md 예시
- **handoff.md** — "지금 어디까지 왔는지 남긴다"
  - 끝난 것, 막힌 점, 다음 단계, 돌아왔을 때 질문
  - handoff.md vs decision-log.md 역할 차이
- **verify.md** — "무엇이 통과해야 관리인지"
- **Plan Mode** — 단계와 실행을 분리한다
- **AskUserQuestion** — 구조화된 질문 수집
- **Approval** — 위험 작업 인간 승인
- **review split** — 구현 세션과 검토 세션을 나누는 방식

### Ch.18 Agent와 SubAgent
- **agentic이란**: 모델이 스스로 도구를 부르고 판단하는 방식
- **Agent vs SubAgent vs Agent Team**:
  | | Agent | SubAgent | Agent Team |
  |---|---|---|---|
  | 정의 | 작업을 수행하는 Claude 인스턴스 | 독립 컨텍스트의 보조 에이전트 | 여러 세션을 팀처럼 운영 |
  | 컨텍스트 | 메인 | 분리 | 각자 분리 |
  | 용도 | 일반 작업 | 병렬/격리 작업 | 역할 분담(planner/builder/reviewer) |
- **agentic loop**: 관찰 → 생각 → 행동 → 반복
- SubAgent를 쓰는 이유: 컨텍스트 오염 방지, 병렬 작업, 역할 분리
- worktree와 SubAgent의 관계
- 🐍 while 루프 — 에이전트 루프도 결국 반복문

### Ch.19 토큰과 비용
- **Token Economy(토큰 경제학)**:
  - input token = Claude가 읽는 모든 것 (프롬프트, 파일, 규칙, 도구 출력)
  - output token = Claude가 쓰는 모든 것 (코드, 요약, 도구 호출)
- **Prompt Caching** — 같은 배경을 반복 보내면 할인
- **Token Budget(토큰 예산)** — 팀 비용 관리
- **비용이 빠지는 5가지 구간**:
  1. 긴 CLAUDE.md와 규칙 파일
  2. tool output (테스트 로그, grep 결과)
  3. 긴 세션의 컨텍스트 누적
  4. agent/subagent의 중복 컨텍스트
  5. 불필요한 MCP 설정

## Part 6: 클로드에게 맡기다 — Cowork

### Ch.20 위임이라는 개념
- **Cowork이란**: "심부름 보내기" — 자료를 장래 두는 시스템이나 결과를 받는 곳
- Chat vs Projects vs Cowork vs Claude Code — **결정 트리**
- Cowork과 Claude Code의 차이는 **위임 방식과 통제 방식**
- **Cowork 폴더 구조**:
  ```
  CLAUDE-COWORK/
  ├── ABOUT-ME/
  │   ├── about-me.md
  │   └── working-rules.md
  ├── PROJECTS/
  │   └── test-project/
  │       └── brief.md
  ├── TEMPLATES/
  │   └── weekly-brief-template.md
  └── CLAUDE-OUTPUTS/
  ```

### Ch.21 Cowork 세팅
- **7일 세팅 로드맵**:
  - Day 1: 폴더와 파일 4개 만든다
  - Day 2: 작은 과제를 한 번 맡겨 본다
  - Day 3: working-rules.md를 수정한다
  - Day 4: 템플릿을 만든다
  - Day 5: 프로젝트 폴더를 실제 업무 기준으로 나눈다
  - Day 6: output 파일 규칙을 정한다
  - Day 7: 반복 작업 하나를 scheduled task로 올린다
- **Cowork 글로벌 지침** 예시
- **시작 템플릿 3종**: about-me.md, working-rules.md, project-brief.md
- 글로벌 지침과 개별 브리프의 차이

### Ch.22 Channels
- **Channel이란**: 외부 이벤트를 실행 중인 세션으로 쏘아주는 경로
- Telegram, Discord, webhook, CI 결과 등
- Channels는 research preview 기능
- 채널 설정법
- **Dispatch vs Remote Control vs Channels vs Claude Code on the web** 4종 비교:
  | | Dispatch | Remote Control | Channels | Code on the web |
  |---|---|---|---|---|
  | 방향 | 새 작업 시작 | 기존 세션 이어받기 | 외부 이벤트 수신 | 원격 VM 작업 |
  | 적합 | 조사/분석 | 장기 실행 관리 | 자동 알림 | GitHub/보안 |

### Ch.23 Scheduled Tasks
- **예약 작업**: "정해진 시간에 같은 작업을 반복 실행"
- Claude Code의 `/loop`과 `CronCreate`/`CronDelete`
- Cowork scheduled tasks vs Claude Desktop scheduled tasks
- **Self-contained 프롬프트 작성법** — 맥락 없이 독립 실행되는 프롬프트
- **Cowork 첫 자동화 7가지**:
  1. 주간 경쟁사 브리프
  2. 아침 우선순위 정리
  3. 미팅 후 후속 정리 초안
  4. 주간 KPI 보고서
  5. 콘텐츠 아이디어 요약
  6. 고객 요청 분류
  7. 매일 아침 캘린더 요약
- 실행 확인과 실패 대처
- 🐍 파일 I/O — 자동화의 기본은 읽고 쓰기

### Ch.24 Remote Control & 파이프라인 설계
- **Remote Control**: 이미 실행 중인 세션을 원격에서 이어받기
  - claude.ai/code나 모바일 앱에서 이어서 보기
- **persistent thread(지속 세션)** — Cowork의 진행 중 상태 유지
- 실전 파이프라인: 트리거 → 실행 → 채널 수신
- 3가지 파이프라인 예시:
  1. git push → Scheduled Task → Slack 알림
  2. 경쟁사 모니터링 → 주간 브리프 → 이메일
  3. 고객 요청 → 분류 → 담당자 Slack DM
