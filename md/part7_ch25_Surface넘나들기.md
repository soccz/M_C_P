# Ch.25 Surface 넘나들기

> **이 챕터를 마치면**: 클로드의 12개 Surface를 각자의 용도별로 구분할 수 있다. 한 작업이 터미널 → 웹 → 모바일을 타고 흘러도 놓치지 않는 감각을 갖는다. 내 업무에 맞는 Surface 조합을 설계할 수 있다.

---

## 배경: "어디서 시작하고 어디로 넘길 것인가"

Ch.1에서 12개 Surface 지도를 처음 봤다. 이 책은 그중 **Claude Code CLI + Cowork**를 중심으로 달려왔다. 이제 나머지를 한 번에 훑는다.

현실은 단일 Surface로 끝나지 않는다:

```
아침 출근길 (iOS)
  → "어제 걸어둔 PR 리뷰 결과 보여줘"
  
사무실 도착 (VS Code 확장)
  → 인라인 diff로 받은 리뷰 적용
  
점심 후 (터미널)
  → claude --teleport 로 모바일 세션을 끌어와 디버깅
  
퇴근 전 (Slack)
  → @Claude 에게 "내일 아침 빌드 확인해줘" 멘션
```

한 작업이 **Surface를 넘어가며 흐른다.** 중요한 건 Surface가 다르다는 게 아니라, **세션이 같다는 것**이다.

> **💭 생각의 흐름**
>
> **문제** — 터미널에서 짜던 코드를 모바일로 이어 보고 싶은데, 복붙으로 옮기기에는 맥락이 너무 많다
> **질문** — "한 작업을 Surface 여러 개가 공유하는 방법이 없나?"
> **발견** — 세션 하나를 여러 Surface가 공유한다. 내가 터미널에서 만든 세션은 Desktop, iOS, Web에서도 이어받을 수 있다
> **확장** — `--teleport`, `/desktop`, Claude Code on the Web — 각 Surface가 서로 세션을 넘겨받는 도구들이 있다
> **결론** — Surface는 **입구**, 세션은 **작업 자체**. 입구가 달라도 같은 작업실로 들어간다

## 12 Surface 다시 보기 — 5가지 분류

Ch.1에서 본 표를 한 번 더, 이번엔 **"실제로 언제 쓰는가"** 관점으로.

| 분류 | Surface | 한 줄 정체 | 대표 사용 시점 |
|---|---|---|---|
| **터미널** | Claude Code CLI | 가장 강력, 가장 낮은 레이어 | 실제 코드 작업 대부분 |
| **IDE** | VS Code / JetBrains / Cursor | 에디터 안에 클로드 | 인라인 diff, @-mention |
| **앱** | Desktop(Mac/Win) | 시각 diff + 다중 세션 보드 | 여러 세션 동시 지켜보기 |
| **앱** | iOS | 이동 중 읽기/간단 지시 | 통근, 미팅 사이 |
| **웹** | Chat (claude.ai) | 일반 대화 | 검색·아이디어 발산 |
| **웹** | Projects | 지식 꾸러미 + 대화 | 문서/브리프 장기 관리 |
| **웹** | **Code on the Web** 🆕 | 브라우저에서 클로드가 GitHub 레포에 직접 작업 | 긴 작업 킥오프, 컴퓨터 없을 때 |
| **협업** | Cowork | 로컬 폴더 + 클로드 자율 | 반복 업무 자동화 |
| **협업** | **Slack 1급 연결** 🆕 | `@Claude` 멘션 | 팀 채널에서 PR 요청 |
| **협업** | **Chrome 확장(beta)** 🆕 | 라이브 웹앱에 붙어서 디버깅 | 프로덕션 UI 버그 추적 |
| **입력** | **Voice dictation** 🆕 | 음성 → 프롬프트 | 타이핑 어려울 때 |
| **외부** | Claude API / Agent SDK | 직접 연동 개발 | 자체 제품에 클로드 내장 |

## Desktop app — 시각 diff와 "세션 보드"

Claude Desktop은 **"세션을 여러 개 켜두고 지켜보는"** 용도가 가장 크다.

```
Desktop의 강점:
  ✅ 시각 diff — 코드 변경을 나란히 보기
  ✅ 다중 세션 보드 — 세션 A, B, C가 동시에 작업 중인 걸 한 화면에
  ✅ /desktop 핸드오프 — 터미널 세션을 Desktop으로 끌어오기
  ✅ Scheduled Tasks 로컬 실행 (Ch.23)
  ✅ Cowork가 이 앱 안에서 돌아감
```

### `/desktop` 🆕 — 터미널 → Desktop 핸드오프

Ch.24에서 본 `/desktop`을 여기서 다시 정리한다.

```bash
# 터미널 세션 안에서
/desktop
```

이 한 줄로 **현재 세션을 Desktop 앱으로 넘긴다**. 터미널은 텍스트만 보이지만, Desktop에 넘기면 **시각 diff + 세션 보드 + 파일 트리 UI**로 바뀐다.

언제 쓰나:
```
✅ diff가 많아져서 터미널로 보기 답답할 때
✅ 세션 여러 개가 동시에 돌고 있고 총괄해야 할 때
✅ 파일 트리를 시각적으로 보면서 판단하고 싶을 때
```

### Desktop ↔ 터미널 역방향

Desktop에서 시작한 세션도 터미널에서 이어받을 수 있다. 그게 `--teleport`다 (다음 절).

## Claude Code on the Web 🆕 — 브라우저에서 GitHub에 바로 작업

**claude.ai/code**. 브라우저 주소창만으로 시작하는 Claude Code다.

```
특징:
  ✅ 컴퓨터가 없어도 됨 (학원 PC, 태블릿, 친구 노트북)
  ✅ GitHub 레포에 직접 작업 → PR로 받음
  ✅ 긴 작업 킥오프에 좋음 (설정 → 떠남 → 결과 수신)
  ❌ 로컬 파일 접근 불가 (모든 작업은 GitHub 레포 안에서)
  ❌ 실시간 diff 확인은 느림
```

### Claude Code on the Web vs 터미널 CLI vs Routines

| | Code on the Web 🆕 | 터미널 CLI | Routines (Ch.23) |
|---|---|---|---|
| **실행 위치** | 클라우드 | 내 컴퓨터 | 클라우드 |
| **트리거** | 사용자가 웹에서 | 사용자가 터미널에서 | 스케줄/이벤트 |
| **세션 지속** | 길게 (닫아도 서버에서 계속) | 세션 열려 있을 때 | 작업 단위로 한 번 |
| **결과 받는 법** | PR 열림 | 즉시 보임 | PR 또는 이메일 |

**쓰는 법 한 줄**: "시간 오래 걸리는 작업을 맡기고 컴퓨터를 덮고 싶다" → Code on the Web.

## `claude --teleport` 🆕 — Surface 간 세션 이동의 핵심

웹/Desktop/iOS에서 시작한 세션을 **터미널로 끌어오는** 명령이다. Ch.24에서 다뤘지만, Surface 이동의 중심 도구이니 다시 요약한다.

```
시나리오:
  ① iOS에서 "로그인 버그 좀 봐줘" 입력
  ② 클로드가 조사 중 — 상세 디버깅은 터미널이 편함
  ③ 데스크 앞에 앉자마자: claude --teleport
  ④ iOS 세션이 터미널에 그대로 이어짐
```

### 세션 이동 3종 정리

| 명령/기능 | 방향 | 용도 |
|---|---|---|
| `claude --teleport` 🆕 | 웹/iOS/Desktop → 터미널 | 모바일에서 시작한 걸 본격 디버깅으로 |
| `/desktop` 🆕 | 터미널 → Desktop | 시각 diff / 세션 보드가 필요할 때 |
| `claude --remote-control` | 터미널 → 원격 트리거 | 외부 이벤트에 반응 (Ch.24) |

**핵심**: 세션은 **하나**, Surface는 **여러 개 입구**. 이 3개 명령이 입구 사이를 잇는 다리.

## IDE 계열 — VS Code / JetBrains / Cursor

에디터 안에 클로드가 들어온 형태다.

### 공통 기능

```
✅ 인라인 diff — 제안을 코드 옆에 바로 표시
✅ @-mention — 특정 파일/심볼을 참조 (@src/auth.ts)
✅ Plan review — 실행 전 계획을 에디터에서 검토
✅ 키바인딩 커스터마이즈 (~/.claude/keybindings.json)
```

### Surface별 차이

| | VS Code | JetBrains | Cursor |
|---|---|---|---|
| **기반** | VS Code 확장 | IDE 플러그인 | Cursor 네이티브 |
| **강점** | 범용, 플러그인 많음 | IntelliJ/PyCharm 심층 연동 | 클로드와의 연동이 1급 |
| **추천 대상** | 웹/JS 개발자 | Java/Kotlin/Python 개발자 | 클로드 중심으로 쓰는 사람 |

### 터미널 CLI와의 관계

IDE 확장은 **결국 터미널 CLI의 얼굴만 바꾼 것**에 가깝다. 내부적으로는 같은 Claude Code 세션을 쓰므로, CLAUDE.md·settings.json·Hooks·Skills가 **그대로 적용**된다.

```
"VS Code 확장에서만 되는 것" 같은 건 거의 없다.
대신 UX가 다르다:
  - 터미널: 텍스트로 빠름
  - IDE: 시각적으로 안정적
```

```
Slack 채널 #engineering:

```

### 주요 사용 패턴

```
✅ 채널에서 바로:
- @Claude PR 리뷰해줘
- @Claude 이번 주 배포 요약
- @Claude 이 스레드 액션 아이템 뽑아줘

✅ DM으로:
- 개인 작업을 슬랙에서도 지시
- 메시지를 Skill 입력으로 사용
```

## Voice dictation 🆕 — 음성으로 프롬프트

iOS/Desktop에서 마이크 버튼을 누르고 말하면 프롬프트가 된다.

```
✅ 적합:
- 길고 맥락 많은 설명 ("이 버그는 어제 김팀장이 공유한 문서의 3번 항목과 관련돼 있는데…")
- 걷거나 운전 중 아이디어 캡처
- 장애인 접근성

❌ 부적합:
- 짧은 명령 (타이핑이 더 빠름)
- 조용한 공간이 필요한 환경
- 코드 정확한 인용이 필요한 입력
```

**주의**: 음성이 프롬프트로 바뀌는 과정에서 문장이 길어지기 쉽다. 길게 말한 뒤 "이 중 핵심만 추려" 한 번 더 요청하는 습관이 좋다.

## Surface 선택 결정 트리

```
내 상황이 뭐지?
│
├── 코드를 실제로 쓰고 있다 (책상 앞, 키보드)
│    ├── 작은 편집 + 인라인 diff가 편함 → IDE 확장 (VS Code/JetBrains)
│    └── 본격 작업 / 자동화 / 스크립트 → 터미널 CLI

├── 여러 세션을 동시에 지켜봐야 한다

│
├── 컴퓨터가 없다
│    ├── 긴 작업 돌리고 결과 기다릴 수 있다 → Claude Code on the Web
│    ├── 지시·확인만 빠르게 → iOS 앱
│    └── 음성이 편하다 → Voice dictation
│
├── 팀과 함께 쓴다
│    └── Slack 1급 연결 (+ Cowork + Routines)
│
├── 프로덕션/라이브 웹앱을 디버깅한다
│    └── Chrome 확장 (beta)
│
└── 제품에 클로드를 내장한다
     └── Claude API / Agent SDK (Ch.28)
```

## "한 세션이 Surface를 타고 흐른다"

이 챕터의 핵심 감각이다. 구체 예시로 마무리:

```
월요일 아침:
  [iOS] "이번 주 우선순위 PR 3개 보여줘"
  → claude.ai에 세션 생성됨 (Claude Code on the Web 내부)
  
월요일 오전:
  [터미널] claude --teleport
  → iOS에서 만든 세션이 터미널로 옴
  → 본격 디버깅 시작
  
월요일 점심:
  [터미널] /desktop
  → 세션이 Desktop 앱으로 이동 (diff 많아서)
  → 시각 diff로 리뷰 진행
  
월요일 오후:
  [Slack] @Claude 내일 아침 8시에 배포 전 최종 테스트 돌려줘
  → Routines 걸림 (Ch.23)
  
화요일 아침:
  [iOS] 배포 결과 확인 → 문제없음 → 머지
```

이 전체가 **한 작업의 흐름**이다. Surface가 5개지만 세션(맥락)은 하나로 이어진다.

## 🐍 파이썬 연결 없음 — 이유

이 챕터는 "각 Surface의 UI·용도"를 다룬다. 파이썬의 특정 개념이 **동형으로 매칭되지 않는다**. 억지로 끼우면 오히려 이해를 방해한다.

프로젝트 규칙("기여도 높은 챕터에만 🐍 배치")에 따라 여기는 **파이썬 없이** 넘어간다. 다음 Ch.26 Computer Use에서는 `subprocess`가 자연스러운 자리에 들어간다.

## 왜 이렇게 동작하는가

Surface는 **"같은 작업을 다른 입구로"** 접근하게 해주는 구조다. 왜 Anthropic이 Surface를 늘리는가? 두 가지 이유가 있다:

1. **업무는 책상에만 있지 않다** — 출근길, 미팅 중, 슬랙 스레드, 라이브 웹앱 — 작업 자체가 이동한다
2. **"맥락 유지 비용"이 핵심 자산** — 복붙으로 세션을 옮기면 토큰·시간·실수가 쌓인다. 세션이 Surface를 따라 흐르면 이 비용이 사라진다

이 구조는 OS의 "윈도우 매니저"와 비슷하다. 같은 프로세스(작업)를 다른 창(Surface)으로 띄우는 것. 창이 달라도 프로세스는 하나.

**Surface = 창, 세션 = 프로세스.** 이 비유를 기억하면 12개 Surface를 12개 앱으로 오해하지 않는다.
