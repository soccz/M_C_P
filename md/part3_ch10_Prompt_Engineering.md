# Ch.10 Prompt Engineering

> **이 챕터를 마치면**: 제약형 프롬프트 템플릿을 자유롭게 쓸 수 있고, 프롬프트 9원칙을 이해한다. 실전에서 바로 복사해서 쓸 수 있는 프롬프트 5종을 갖게 된다.

---

## 배경: Context 위에 올라가는 기술

Ch.9에서 "컨텍스트 엔지니어링이 프롬프트 엔지니어링보다 더 중요하다"고 했다. 그렇다고 프롬프트가 안 중요한 건 아니다.

비유하면:
- **컨텍스트** = 잘 정리된 책상 (좋은 환경)
- **프롬프트** = 책상 위에서 하는 요청 (좋은 지시)

책상이 어지러우면 아무리 요청을 잘 해도 결과가 별로다. 하지만 책상이 깔끔해도 요청이 모호하면 역시 결과가 별로다. **둘 다 잘해야 한다.**

이 챕터에서는 프롬프트 자체를 잘 쓰는 법에 집중한다.

> **💭 생각의 흐름**
>
> **문제** — "README 만들어줘"라고 했더니 원하는 것과 완전 다른 형식으로 나왔다
> **질문** — "내가 뭘 빠뜨린 거지?"
> **발견** — 목표, 형식, 완료 기준을 안 정해줬다. 클로드가 알아서 추측한 것
> **결과** — 제약형 프롬프트 템플릿으로 6가지를 명시하니 한 번에 원하는 결과가 나왔다

## 제약형 프롬프트 템플릿

마스터 가이드에서 가장 실용적인 프롬프트 구조 중 하나다. **6가지 태그**로 요청을 구조화한다.

```
<goal>
무엇을 만들고 싶은가
</goal>

<context>
어떤 파일과 자료를 먼저 읽어야 하는가
</context>

<constraints>
포함할 것, 피할 것, 길이 제한
</constraints>

<output_format>
표 / prose / JSON / 체크리스트 중 무엇인가
</output_format>

<done_when>
무엇을 보면 완료라고 볼 수 있는가
</done_when>

<if_unsure>
불확실하면 질문할지, 모른다고 말할지
</if_unsure>
```

### 왜 이 구조가 좋은가

1. **빠뜨릴 게 없다** — 6가지 칸을 채우다 보면 모호함이 사라진다
2. **클로드가 구조를 인식한다** — XML 태그를 쓰면 각 부분을 명확히 구분한다
3. **재사용할 수 있다** — 같은 틀에 내용만 바꾸면 된다

### 실전 예시: 경쟁사 브리프

```
<goal>
5분 안에 읽을 수 있는 경쟁사 브리프를 만든다.
</goal>

<context>
'projects/competitors/brief.md',
'projects/competitors/source-notes.md',
'templates/weekly-brief-template.md'를 먼저 읽어.
</context>

<constraints>
지난 7일 자료만 쓰고, 출처 충돌은 separate section으로 배.
</constraints>

<output_format>
실단 표 1개, 그 아래 3문단 요약.
</output_format>

<done_when>
'outputs/weekly-brief.md'에 저장되고, 다음 액션이 3개 보인다.
</done_when>

<if_unsure>
근거가 부족하면 추정하지 말고 '확인 필요'로 남겨.
</if_unsure>
```

이 프롬프트는 길어 보이지만, 클로드가 한 번에 정확한 결과를 낸다. "경쟁사 브리프 만들어줘"와 비교하면 재작업이 0에 가깝다.

## 프롬프트 9원칙

마스터 가이드와 Anthropic 공식 문서, 그리고 실전 경험을 종합한 9가지 원칙이다.

### 1. 모호한 활용사보다 조건이 먼저

```
❌ "좋은 코드를 작성해줘"
✅ "함수는 30줄 이내, 매직 넘버 금지, 타입은 모두 명시"
```

"좋은"이 뭔지는 사람마다 다르다. 조건으로 구체화해야 결과가 일관된다.

### 2. 긴 컨텍스트에서는 자료를 위에, 질문을 아래에

```
✅ "'CLAUDE.md'와 'src/features/signup/'을 먼저 읽어.
   그 다음 회원가입 버그를 찾아줘."
```

클로드는 끝부분에 더 집중한다. 자료를 위에 놓고 요청을 아래에 놓으면 지시를 놓칠 확률이 줄어든다.

### 3. 범용사보다 짧은 예시(few-shot)

```
❌ "적절한 형식으로 정리해줘"
✅ "아래 형식대로:
   - 핵심 1줄
   - 근거 2개
   - 다음 액션 1줄"
```

형식을 말로 설명하는 것보다 예시 하나가 10배 정확하다.

### 4. 200자 제한과 guardrail

긴 프롬프트가 항상 좋은 건 아니다. 핵심 지시는 200자 이내로 압축하고, 세부 사항은 컨텍스트 파일(CLAUDE.md, rules/)에 넣는다.

```
❌ (500자 프롬프트에 모든 규칙 나열)
✅ "CLAUDE.md 규칙을 따르면서, 이 함수의 반환 타입을 수정해줘."
```

### 5. Extended Thinking 활용

복잡한 추론이 필요할 때 클로드에게 **"생각할 시간"**을 줄 수 있다.

```
"이 버그의 가능한 원인을 3가지 이상 생각하고,
가장 가능성 높은 것부터 조사해줘."
```

명시적으로 "생각하고"라고 하면 클로드가 더 깊이 추론한다. Anthropic의 Extended Thinking은 이 과정을 모델 레벨에서 지원한다.

### 6. 불확실성 표현 허용

```
✅ "확실하지 않은 부분은 '확인 필요'로 표시하고,
   확실한 부분만 구현해줘."
```

클로드에게 "모르면 모른다고 해도 돼"라고 허용하면 할루시네이션이 줄어든다.

### 7. 한 번에 다 시키기보다 나누기

```
❌ "리팩터링하고 테스트 추가하고 문서도 업데이트해줘"
✅ Step 1: "먼저 리팩터링 계획만 세워줘"
   Step 2: (확인 후) "실행해줘"
   Step 3: "이제 테스트 추가해줘"
```

큰 작업은 나눠서 시키면 중간에 방향을 수정할 수 있다.

### 8. 구조화 출력(Structured Output) 활용

```
✅ "결과를 JSON으로 돌려줘:
   {
     'issue': '설명',
     'root_cause': '원인',
     'files_to_check': ['파일 목록'],
     'confidence': 0-1 사이 숫자
   }"
```

자유 텍스트보다 구조를 지정하면 후속 처리가 쉽다. 특히 자동화에서 유용하다.

### 9. No preamble, return only JSON

자동화나 파이프라인에서 쓸 때:

```
✅ "설명 없이 JSON만 반환해. No preamble."
```

클로드가 "네, 분석해보겠습니다" 같은 서론 없이 바로 결과만 준다.

## 실전 프롬프트 5종

바로 복사해서 쓸 수 있는 프롬프트 모음이다.

### 1. Context-first 프롬프트

```
먼저 이 프로젝트의 'CLAUDE.md', 관련 rules, 그리고 내가 지정한 폴더를 읽어.
그 다음 작업을 시작하기 전에 누락된 전제가 있으면 질문해.
바로 구현하지 말고, 수정 범위와 완료 기준부터 요약해.
```

### 2. 버그 수정 프롬프트

```
이 오류를 바로 고치지 말고 아래 순서로 진행해.
관련 파일을 읽은 뒤 수정 범위와 테스트 계획을 제안해.
바로 코드를 건드리지 마.
1. 재현 경로
2. 원인 가설
3. 수정 범위
4. 테스트 계획
을 먼저 정리하고, 내가 확인한 뒤 수정해.
```

### 3. Cowork 브리프 프롬프트

```
이 폴더의 파일을 먼저 읽어.
그 다음 [주제]를 [성공 기준]에 따라 조사해.
최근 자료를 우선하고, 출처 충돌과 불확실성은 따로 표시해.
가능하면 AskUserQuestion으로 빈칸을 채워.
산출물은 'outputs/'에 저장해.
```

### 4. 리뷰 프롬프트

```
현재 변경분을 correctness, regression risk, missing tests,
security risk 기준으로 검토해.
중요도 순서대로 골라.
스타일 문제보다 동작 문제를 우선해.
각 항목은 아래 형식으로 정리해.
- 문제 요약
- 왜 문제인지
- 어느 파일을 봐야 하는지
- 지금 당장 필요한 수정 또는 테스트
```

### 5. 제약형 프롬프트 (범용)

```
<goal>
[무엇을 만들고 싶은가]
</goal>
<context>
[어떤 파일과 자료를 읽어야 하는가]
</context>
<constraints>
[포함할 것, 피할 것, 길이 제한]
</constraints>
<output_format>
[표/prose/JSON/체크리스트]
</output_format>
<done_when>
[무엇을 보면 완료인가]
</done_when>
<if_unsure>
[불확실하면 질문? 모른다고 표시?]
</if_unsure>
```

## 나쁜 / 보통 / 좋은 — 5가지 상황 비교

### 상황 1: 새 기능 추가

| | 프롬프트 |
|---|---------|
| 나쁜 | "로그인 기능 만들어줘" |
| 보통 | "이메일+비밀번호 로그인 기능 만들어줘. JWT 토큰 방식으로." |
| 좋은 | "src/features/auth/에 이메일+비밀번호 로그인 추가. JWT 토큰 방식. zod로 입력 검증. 테스트도 같이. CLAUDE.md 규칙 따라서." |

### 상황 2: 데이터 분석

| | 프롬프트 |
|---|---------|
| 나쁜 | "이 데이터 분석해줘" |
| 보통 | "매출 데이터를 분석해서 트렌드를 알려줘" |
| 좋은 | "data/sales_2026.csv를 읽고 월별 매출 추이를 표로. 전월 대비 변화율 포함. 이상치가 있으면 '확인 필요'로 표시." |

### 상황 3: 문서 작성

| | 프롬프트 |
|---|---------|
| 나쁜 | "README 써줘" |
| 보통 | "이 프로젝트의 README를 만들어줘. 설치법과 사용법 포함." |
| 좋은 | "README.md를 만들어줘. 프로젝트 이름, 한 줄 설명, 설치(npm install), 실행(npm dev), 테스트(npm test), 라이선스(MIT). 이 순서대로. 200줄 이내." |

### 상황 4: 리팩터링

| | 프롬프트 |
|---|---------|
| 나쁜 | "이거 리팩터링해줘" |
| 보통 | "이 파일을 깔끔하게 정리해줘" |
| 좋은 | "src/utils/helpers.ts에서 calculateTotal 함수만 리팩터링. 순수 함수로 바꾸고 매직 넘버를 상수로. 나머지 함수는 건드리지 마." |

### 상황 5: 코드 리뷰

| | 프롬프트 |
|---|---------|
| 나쁜 | "코드 리뷰해줘" |
| 보통 | "이 PR을 리뷰해줘. 버그 위주로." |
| 좋은 | "최근 변경분을 리뷰해줘. correctness > regression risk > security 순. 각 이슈마다: 문제 요약, 파일 위치, 심각도(P1/P2/P3). 스타일 이슈는 무시." |

## 제품이 바뀌어도 남는 프롬프트 원칙

기능은 업데이트되면 바뀌지만, **원칙은 오래 남는다**. 위 9원칙과 제약형 템플릿은 Claude 3든 4든 5든, 심지어 다른 AI를 쓸 때도 통한다. 이유는:

1. **모호함 감소** — 어떤 모델이든 모호한 입력에는 모호한 출력
2. **구조화** — 기계가 잘 처리하는 형태
3. **검증 가능** — done_when이 있으면 결과를 판단할 수 있다
4. **재현 가능** — 같은 프롬프트를 다시 쓰면 비슷한 결과
