# Ch.25 내 워크플로우 설계하기

> **이 챕터를 마치면**: 자신의 업무를 4가지 과업 모양으로 분류할 수 있고, 각 모양에 맞는 파일 구조와 첫 프롬프트를 설계할 수 있다. 자신의 역할에 맞는 스타터 번들을 골라 즉시 시작할 수 있다.

---

## 배경: "불편함"에서 시작하기

Part 1~6까지 클로드의 기능을 하나씩 배웠다. 기억(Ch.5), 설정(Ch.6), 규칙(Ch.7), 습관(Ch.8), 프롬프트(Ch.10), 스킬(Ch.12), MCP(Ch.13), 세션(Ch.16), 토큰(Ch.19), 위임(Ch.20~24)...

이제 질문이 바뀐다.

> "이 기능들을 **내 업무에** 어떻게 조합하지?"

답은 기능 목록에 없다. 답은 **내 불편함**에 있다.

> **💭 생각의 흐름**
>
> **문제** — 기능은 많이 배웠는데, 실제 업무에서 뭘 먼저 해야 할지 모르겠다. CLAUDE.md도 쓰고, MCP도 연결하고, Skill도 만들었는데... 체계가 없다
> **질문** — "기능이 아니라 **내 업무**에서 출발하면 어떨까?"
> **시도** — 한 주 동안 "클로드 없이 했으면 싫었을 작업"을 적어봤다. 3가지가 나왔다: 주간 보고서 정리, 코드 리뷰 요약, 미팅 후속 정리
> **발견** — 이 3가지는 모두 **같은 패턴**이다: 자료 읽기 → 구조화 → 형식에 맞춰 출력
> **결론** — 기능에서 출발하면 길을 잃고, **불편함에서 출발하면 답이 보인다**

---

## "불편함 → 질문 → 재구성" 사이클

워크플로우 설계는 3단계 사이클이다.

```
1. 불편함 발견
   "이 작업, 매주 같은 걸 반복하는데..."
   "이거 정리하는 데 매번 2시간이 걸려..."
   "형식이 매번 달라서 고쳐야 해..."

         ↓

2. 질문으로 변환
   "이 작업의 입력·출력·반복 주기는?"
   "어떤 부분을 클로드에게 맡길 수 있나?"
   "어떤 파일 구조가 필요한가?"

         ↓

3. 재구성
   "CLAUDE.md에 규칙을 추가하자"
   "템플릿을 만들어 형식을 고정하자"
   "Scheduled Task로 자동화하자"

         ↓

   (결과를 보고 다시 1로 돌아감)
```

핵심: **한 번에 완성하려 하지 말 것.** 작은 불편함 하나를 골라, 질문으로 바꾸고, 재구성하고, 결과를 본다. 그 반복이 워크플로우가 된다.

## 과업의 4가지 모양

모든 업무는 4가지 모양 중 하나에 가깝다. 모양을 알면 파일 구조와 첫 프롬프트가 정해진다.

### 정리형 — "흩어진 것을 모으고 구조화하기"

```
예시:
  · 미팅 노트 정리
  · 주간 보고서 작성
  · 이메일 초안 작성
  · 회의록 요약

특징:
  · 입력: 비정형 자료 (노트, 대화, 메모)
  · 출력: 정형화된 문서 (보고서, 요약, 이메일)
  · 핵심: 형식(template)이 품질을 결정한다
```

**파일 구조:**

```
정리형-작업/
├── brief.md            ← 무엇을 정리하는가
├── template.md         ← 출력 형식
├── input/              ← 정리할 원본 자료
│   ├── meeting-notes.md
│   └── raw-data.csv
└── output/             ← 정리된 결과
```

**첫 프롬프트:**

```
brief.md와 template.md를 읽은 뒤,
input/ 폴더의 자료를 정리해서 template.md 형식으로 작성해줘.
결과를 output/ 폴더에 저장해.
```

### 분석형 — "데이터에서 의미를 뽑기"

```
예시:
  · 경쟁사 동향 분석
  · 고객 피드백 분류
  · 매출 데이터 해석
  · 트렌드 파악

특징:
  · 입력: 데이터, 자료, 외부 정보
  · 출력: 인사이트, 추천, 판단
  · 핵심: 분석 기준(criteria)이 품질을 결정한다
```

**파일 구조:**

```
분석형-작업/
├── brief.md            ← 무엇을 분석하는가, 어떤 기준으로
├── criteria.md         ← 분석 기준과 판단 프레임워크
├── data/               ← 분석할 데이터
│   ├── feedback.csv
│   └── competitor-posts/
└── output/             ← 분석 결과
```

**첫 프롬프트:**

```
brief.md를 읽고 분석 목표를 이해한 뒤,
criteria.md의 기준에 따라 data/ 폴더의 자료를 분석해.
결과를 output/ 폴더에 저장해.
```

### 구현형 — "실제로 만들기"

```
예시:
  · 웹사이트 구축
  · 스크립트 작성
  · 자동화 파이프라인 구축
  · API 연동

특징:
  · 입력: 요구사항, 설계, 참고 코드
  · 출력: 동작하는 코드, 설정 파일
  · 핵심: 완료 기준(done-when)과 검증이 품질을 결정한다
```

**파일 구조:**

```
구현형-작업/
├── plan.md             ← 무엇을 만드는가, 완료 기준
├── reference/          ← 참고 코드, API 문서
├── src/                ← 소스 코드
└── tests/              ← 테스트
```

**첫 프롬프트:**

```
plan.md를 읽고 구현 계획을 세워.
reference/ 폴더의 자료를 참고해.
구현 후 tests/의 테스트를 실행해서 통과 여부를 확인해.
```

### 검토형 — "이미 있는 것을 평가하기"

```
예시:
  · 코드 리뷰
  · 문서 교정
  · 보안 점검
  · 품질 감사

특징:
  · 입력: 검토 대상 + 기준
  · 출력: 피드백, 수정사항, 점수
  · 핵심: 체크리스트가 품질을 결정한다
```

**파일 구조:**

```
검토형-작업/
├── brief.md            ← 무엇을 검토하는가
├── checklist.md        ← 검토 기준 체크리스트
├── target/             ← 검토 대상
└── review/             ← 검토 결과
```

**첫 프롬프트:**

```
brief.md와 checklist.md를 읽은 뒤,
target/ 폴더의 내용을 체크리스트 기준으로 검토해.
결과를 review/ 폴더에 저장해.
```

### 4가지 모양 비교

| | 정리형 | 분석형 | 구현형 | 검토형 |
|---|---|---|---|---|
| **핵심 파일** | template.md | criteria.md | plan.md | checklist.md |
| **품질 결정** | 형식 | 기준 | 완료 조건 | 체크리스트 |
| **적합 도구** | Cowork | Cowork/Code | Claude Code | Claude Code |
| **자동화** | Scheduled Task | Scheduled Task | /loop | Hook |

## 역할별 스타터 번들

"어디서부터 시작하지?"에 대한 답. 자신의 역할에 가장 가까운 번들을 골라, 먼저 만들 파일 3개 → 첫 과제 1개 → 첫 프롬프트 → 멈추는 기준 순서로 시작한다.

### 창업가 / 1인 사업자

```
먼저 만들 파일:
  1. about-me.md — 사업 분야, 의사결정 스타일, 선호 형식
  2. competitor-brief.md — 경쟁사 목록과 모니터링 기준
  3. weekly-report-template.md — 주간 정리 형식

첫 과제:
  "이번 주 경쟁사 블로그 3곳을 읽고 핵심 동향을 정리해줘"

첫 프롬프트:
  competitor-brief.md를 읽고, weekly-report-template.md 형식으로
  이번 주 경쟁사 동향을 정리해줘. output/ 폴더에 저장.

멈추는 기준:
  결과를 읽고 직접 수정하는 시간이 10분 이하면 성공.
  10분 이상 고치고 있으면 brief나 template을 다듬을 때.
```

### 제품 관리자 (PM)

```
먼저 만들 파일:
  1. about-me.md — 담당 제품, 팀 구성, 의사결정 프레임워크
  2. prd-template.md — 제품 요구사항 문서 형식
  3. sprint-review-template.md — 스프린트 리뷰 정리 형식

첫 과제:
  "이번 스프린트 회의록을 읽고 리뷰 문서를 작성해줘"

멈추는 기준:
  팀원에게 공유하기 전 수정이 2곳 이하면 성공.
```

### 마케팅 / 콘텐츠

```
먼저 만들 파일:
  1. about-me.md — 브랜드 톤, 타겟 고객, 금기어
  2. brand-voice.md — 브랜드 목소리 가이드
  3. content-template.md — 콘텐츠 형식 (블로그/SNS/뉴스레터)

첫 과제:
  "이 제품 업데이트 내용을 블로그 포스트로 작성해줘"

멈추는 기준:
  브랜드 톤이 맞고, 팩트 체크만 하면 바로 발행할 수 있으면 성공.
```

### 세일즈 / 고객 성공

```
먼저 만들 파일:
  1. about-me.md — 담당 고객 유형, 제품 특장점, 경쟁 포지셔닝
  2. proposal-template.md — 제안서 형식
  3. objection-handling.md — 자주 받는 반론과 대응

첫 과제:
  "이 고객의 요구사항을 읽고 맞춤 제안서 초안을 작성해줘"

멈추는 기준:
  고객에게 보내기 전 수정이 가격·일정만이면 성공.
```

### 초보 개발자

```
먼저 만들 파일:
  1. CLAUDE.md — 프로젝트 구조, 기술 스택, 코딩 컨벤션
  2. plan.md — 현재 작업의 목표와 완료 기준
  3. rules/code-style.md — 코딩 스타일 규칙

첫 과제:
  "이 함수에 단위 테스트를 추가해줘"

첫 프롬프트:
  plan.md를 읽고, src/utils/calculate.py의
  calculate_total 함수에 대한 단위 테스트를 작성해줘.
  pytest 사용, 엣지 케이스 포함.

멈추는 기준:
  pytest가 통과하고, 코드를 읽었을 때 이해가 되면 성공.
```

### 교육자

```
먼저 만들 파일:
  1. about-me.md — 과목, 학생 수준, 교육 철학
  2. lesson-template.md — 수업 자료 형식
  3. rubric-template.md — 평가 기준표 형식

첫 과제:
  "이 단원의 핵심 개념을 설명하는 워크시트를 만들어줘"

멈추는 기준:
  학생에게 바로 배포할 수 있는 수준이면 성공.
```

### 연구자 / 분석가

```
먼저 만들 파일:
  1. about-me.md — 연구 분야, 분석 프레임워크, 인용 스타일
  2. analysis-template.md — 분석 보고서 형식
  3. criteria.md — 데이터 평가 기준

첫 과제:
  "이 설문 데이터를 읽고 주요 발견사항을 정리해줘"

멈추는 기준:
  수치가 정확하고, 해석이 데이터에 근거하면 성공.
```

### 백오피스 / 운영

```
먼저 만들 파일:
  1. about-me.md — 담당 업무, 사용 시스템, 보고 형식
  2. process-template.md — 프로세스 문서 형식
  3. checklist-template.md — 점검 체크리스트

첫 과제:
  "이번 달 경비 보고서 데이터를 정리해줘"

멈추는 기준:
  수치 합산이 정확하고, 형식이 제출용에 맞으면 성공.
```

## 실제 케이스: Before / After

### 케이스 1: 주간 경쟁사 브리프 (마케팅)

```
Before:
  · 경쟁사 블로그 3곳을 직접 방문
  · 주요 포스트를 읽고 메모
  · 팀 슬랙에 요약 작성
  · 소요: ~90분 / 매주

After:
  · Cowork에 competitor-brief.md + template 세팅
  · Scheduled Task: 매주 월요일 오전 9시 실행
  · 결과 확인 후 미세 수정 (10분)
  · 소요: ~10분 / 매주 (초기 세팅 30분)

절약: 주당 ~80분, 월 ~320분
핵심: template과 brief가 품질을 결정. 한 번 잡으면 매주 일관된 결과.
```

### 케이스 2: 코드 리뷰 요약 (개발자)

```
Before:
  · PR 5~10개를 하나씩 열어봄
  · 변경사항 파악 후 코멘트 작성
  · 팀 회의에서 구두로 요약
  · 소요: ~60분 / 매일

After:
  · CLAUDE.md에 코드 리뷰 규칙 추가
  · Skill: code-review-summary (PR 목록 → 요약 문서)
  · Hook: PR merge 후 자동 실행
  · 소요: 결과 확인 5분 / 매일 (초기 세팅 1시간)

절약: 일당 ~55분
핵심: checklist.md가 리뷰 기준을 고정. 사람이 빠져도 같은 품질.
```

### 케이스 3: 미팅 후속 정리 (PM)

```
Before:
  · 미팅 후 노트를 정리
  · 액션 아이템을 추출
  · 담당자에게 각각 슬랙 DM
  · 소요: ~30분 / 미팅당

After:
  · meeting-template.md에 형식 정의
  · Claude Code에서 노트 붙여넣기 → 정리 + 액션 아이템 추출
  · 수동 확인 후 발송
  · 소요: ~10분 / 미팅당 (세팅 15분)

절약: 미팅당 ~20분
핵심: template이 "어떤 항목을 뽑을지" 결정. 빠뜨리는 항목이 없어진다.
```

## 다른 도구와의 비교

"Claude Code만 써야 하나?" 아니다. 목적에 맞는 도구를 쓰면 된다.

### Claude Code vs Codex (OpenAI)

| | **Claude Code** | **Codex (OpenAI)** |
|---|---|---|
| **실행 환경** | 로컬 터미널 | 클라우드 샌드박스 |
| **파일 접근** | 로컬 전체 | GitHub 레포 (clone) |
| **실시간 상호작용** | 실시간 대화 | 태스크 큐 (비동기) |
| **커스터마이징** | CLAUDE.md, skills, hooks | AGENTS.md |
| **비용 모델** | 토큰 기반 | 태스크 기반 |
| **강점** | 깊은 커스터마이징, 로컬 통합 | GitHub 네이티브, PR 자동화 |

**언제 Claude Code가 나은가:**
- 로컬 파일에 직접 접근해야 할 때
- 실시간으로 대화하며 작업해야 할 때
- MCP, Skill, Hook으로 깊이 커스터마이즈할 때
- Cowork을 통해 비개발 업무까지 확장할 때

**언제 Codex가 나은가:**
- GitHub PR 중심 워크플로우
- 여러 태스크를 동시에 큐에 넣을 때
- 로컬 환경 설정 없이 바로 시작할 때

### 기타 도구들

AI 코딩 에이전트 생태계는 빠르게 변하고 있다. Codex 외에도:

- **Cursor** — AI 기반 코드 에디터. 코드 작성 중 실시간 자동완성과 채팅
- **Windsurf** — 에이전틱 IDE. 파일 수정과 터미널 실행을 자동으로 조합
- **Devin** — 클라우드 기반 자율 에이전트. 독립적으로 태스크 수행
- **OpenClaw** — 오픈소스 개인 AI 어시스턴트. 코딩 에이전트 스킬, GitHub 연동, 크론 작업 지원

이 목록은 이 책을 쓰는 시점의 스냅샷이다. 새 도구가 계속 등장하고, 기존 도구가 빠르게 발전한다.

### 핵심 판단 기준

```
"로컬에서 깊이 통합" → Claude Code
"클라우드에서 가볍게 태스크" → Codex
"비개발 업무 위임" → Cowork
"코드 에디터 안에서 바로" → Cursor, Windsurf
"완전 자율 실행" → Devin
```

이 책에서 배운 Context Engineering, Harness Engineering 원리는 **어떤 도구를 쓰든** 적용된다. 도구는 바뀌어도, 구조화된 컨텍스트와 검증 루프의 가치는 변하지 않는다.

## 왜 이렇게 동작하는가

워크플로우 설계의 핵심 원리는 **분류(classification)**다.

모든 업무를 4가지 모양으로 분류하면:
1. **핵심 파일이 정해진다** — template, criteria, plan, checklist
2. **첫 프롬프트가 정해진다** — 파일을 읽고, 기준에 따라, 형식에 맞춰
3. **적합한 도구가 정해진다** — Cowork vs Claude Code
4. **자동화 방식이 정해진다** — Scheduled Task vs /loop vs Hook

이것은 소프트웨어 설계에서 말하는 **패턴 인식(pattern recognition)**이다. 개별 문제를 하나하나 풀지 않고, 패턴을 인식해서 해결 구조를 재사용한다.

## 변형해보기

1. **쉬운 과제**: 이번 주 업무 5가지를 적고, 각각을 정리형/분석형/구현형/검토형으로 분류해보자. 어떤 모양이 가장 많은가?
2. **어려운 과제**: 가장 많은 모양에서 하나를 골라, 파일 구조를 만들고 첫 프롬프트를 작성해보자. 실제로 클로드에게 맡겨보고, "멈추는 기준"에 도달하는지 확인

## 다음 챕터로

개인 워크플로우를 설계했다. 그런데 혼자 쓰는 것과 **팀이 함께 쓰는 것**은 다르다. 팀 CLAUDE.md는 어떻게 만드는지, 30/60/90일 도입 로드맵은 어떤 순서인지, 보안과 비용 관리는 어떻게 하는지 — Ch.26에서 다룬다.

## 이 챕터 핵심 3줄
- **불편함에서 출발**: "어떤 기능을 쓸까"가 아니라 "어떤 작업이 불편한가"에서 시작. 불편함 → 질문 → 재구성 사이클을 반복
- **4가지 모양**: 정리형(template) / 분석형(criteria) / 구현형(plan) / 검토형(checklist). 모양이 정해지면 파일 구조·프롬프트·도구가 따라온다
- **스타터 번들**: 역할에 맞는 파일 3개 + 첫 과제 1개로 시작. 한 번에 완성하지 말고, 결과를 보고 규칙을 다듬는 반복이 워크플로우
