# Ch.16 세션 운영

> **이 챕터를 마치면**: 세션이 무엇인지, continue/resume/fork-session의 차이를 이해한다. /compact와 worktree를 적절히 쓸 수 있고, 세션 운영 체크리스트를 실천할 수 있다.

---

## 배경: 왜 세션을 신경 써야 하나

Part 4까지 클로드에게 **능력을 주는 법**을 배웠다. 이제부터는 그 능력을 **어떻게 효율적으로 운용하는지** 배운다.

클로드와의 모든 작업은 **세션** 안에서 일어난다. 세션을 잘 관리하면:
- 이전 작업을 잃지 않는다
- 토큰을 아낀다
- 복잡한 작업도 끊기지 않고 이어간다

세션을 신경 쓰지 않으면:
- 매번 처음부터 설명해야 한다
- 컨텍스트가 터져서 앞의 규칙을 잊는다
- "아까 하던 거 이어서 해줘"가 안 된다

> **💭 생각의 흐름**
>
> **문제** — 어제 하던 리팩터링을 오늘 이어서 하고 싶다. 새 세션을 열었더니 클로드가 아무것도 모른다
> **질문** — "어제 대화를 이어가는 방법이 없나?"
> **시도 1** — 어제 했던 내용을 다시 설명했다 → 토큰 낭비, 설명도 부정확
> **시도 2** — `claude --continue`로 이어갔다 → 어제 컨텍스트 그대로!
> **발견** — 세션은 관리하는 것이다. 이어가기, 분기하기, 압축하기 — 각각 용도가 다르다
> **결론** — 세션을 의식적으로 관리하면 작업 연속성과 토큰 효율 둘 다 잡을 수 있다

## 세션이란

**세션 = Claude와의 하나의 대화 흐름**이다.

```
세션 1: "로그인 버그 수정해줘"
  ├── 파일 읽기
  ├── 원인 분석
  ├── 코드 수정
  └── 테스트 확인
  → 세션 종료 (터미널 닫기 또는 /exit)

세션 2: "회원가입 기능 만들어줘"
  ├── 이전 세션과 무관한 새 작업
  └── 새로운 컨텍스트에서 시작
```

세션이 끝나면 **컨텍스트가 저장**된다. 나중에 이어갈 수 있다.

### 세션의 생명주기

```
시작 → 작업 → (압축) → (이어가기) → 종료
 │                │          │
 │                │          └── --continue / --resume
 │                └── /compact
 └── claude 또는 claude --continue
```

## continue / resume / fork-session

세션을 이어가는 세 가지 방법이 있다. 각각 용도가 다르다.

### --continue: 마지막 세션 이어가기

```bash
claude --continue
```

**가장 많이 쓰는 방법.** 마지막으로 작업하던 세션을 그대로 이어간다.

```
어제 세션: "로그인 버그 수정 중 → 원인 파악까지 함"
오늘: claude --continue
→ "어제 원인 파악한 것에 이어서 수정할게요"
```

**언제 쓰나**: 어제(또는 아까) 하던 작업을 그대로 이어갈 때

### --resume: 특정 세션 선택해서 이어가기

```bash
claude --resume
```

여러 세션 중에서 **원하는 것을 골라** 이어간다. 세션 목록이 뜨고, 번호를 선택한다.

```
세션 목록:
  1. [2시간 전] 로그인 버그 수정
  2. [어제] 회원가입 기능
  3. [3일 전] DB 마이그레이션

→ 2번 선택: 회원가입 기능 이어서
```

**언제 쓰나**: 여러 작업을 번갈아가며 할 때, 특정 세션을 골라서 이어갈 때

### --fork-session: 기존 컨텍스트에서 새 방향으로

```bash
claude --fork-session
```

기존 세션의 컨텍스트를 **복사**해서 새 방향으로 간다. 원본 세션은 그대로 남는다.

```
원본 세션: "로그인 버그 수정 → 원인: 비밀번호 검증"

fork:
├── 원본 유지: "비밀번호 검증 수정"
└── 새 방향: "비밀번호 검증은 두고, 리프레시 토큰도 확인"
```

**언제 쓰나**: 같은 맥락에서 다른 방향을 탐색하고 싶을 때. 원본을 건드리지 않고 실험.

### 비교표

| | --continue | --resume | --fork-session |
|---|---|---|---|
| **선택** | 마지막 세션 자동 | 목록에서 선택 | 목록에서 선택 |
| **원본** | 이어감 | 이어감 | 복사 (원본 유지) |
| **용도** | 이어서 작업 | 특정 세션 이어가기 | 다른 방향 탐색 |
| **비유** | 책갈피에서 이어 읽기 | 책장에서 골라 읽기 | 복사본에 메모하기 |

## /compact — 컨텍스트 압축

### 왜 필요한가

세션이 길어지면 컨텍스트가 쌓인다. Ch.5에서 배운 **컨텍스트 윈도우**(약 200K 토큰)에 한계가 있다.

```
세션 초반:
  CLAUDE.md (500) + 대화 (1,000) + 코드 (2,000) = 3,500 토큰 ← 여유

세션 30분 후:
  CLAUDE.md (500) + 대화 (15,000) + 도구 출력 (30,000) = 45,500 토큰 ← 아직 괜찮

세션 2시간 후:
  CLAUDE.md (500) + 대화 (50,000) + 도구 출력 (120,000) = 170,500 토큰 ← 위험!
```

컨텍스트가 가득 차면 **앞부분부터 잊기 시작**한다. CLAUDE.md의 규칙도 잊을 수 있다.

### /compact의 동작

```
/compact 실행 전:
  [CLAUDE.md] [대화1] [도구출력1] [대화2] [도구출력2] ... [대화50] [도구출력50]
  → 170,000 토큰

/compact 실행 후:
  [CLAUDE.md] [압축된 요약: "지금까지 X를 했고, Y가 남았다"]
  → 5,000 토큰
```

과거 대화를 **요약**으로 바꿔서 토큰을 대폭 줄인다.

### 언제 써야 하나

```
✅ /compact를 쓰는 좋은 타이밍:
- /cost로 확인했을 때 토큰이 50% 이상 찼을 때
- 작업 방향이 크게 바뀌어서 이전 내용이 불필요할 때
- 도구 출력이 많이 쌓여서 무거워졌을 때

❌ /compact를 쓰면 안 되는 타이밍:
- 방금 읽은 파일 내용을 아직 쓰고 있을 때
- 진행 중인 디버깅의 중간 결과를 기억해야 할 때
```

### /compact 전에 반드시 할 것

Ch.11에서 배운 **Compact Instructions**를 떠올리자:

```
/compact 전에 확인:
1. 마지막으로 수정한 파일과 이유 — 요약에 포함될까?
2. 통과/실패한 테스트 — 잊으면 안 되는 것
3. 다음에 바로 할 일 — 이어서 뭘 해야 하는지
4. handoff.md에 기록했나? — 더 안전한 대안
```

**handoff.md에 미리 적어두면** /compact로 요약이 부실해도 괜찮다.

## worktree — 병렬 작업 폴더

### worktree란

Ch.11에서 잠깐 언급한 **worktree**를 자세히 알아보자.

worktree는 **Git 저장소의 분리 작업 폴더**다. 같은 저장소를 **두 곳에서 동시에** 작업할 수 있다.

```bash
claude --worktree
```

이 명령으로 시작하면:

```
원본 폴더: my-project/        ← 건드리지 않음
worktree:  my-project-worktree/ ← 여기서 작업
```

### 왜 필요한가

```
❌ worktree 없이:
  "실험해볼게" → 원본 수정 → 실패 → 되돌리기 복잡

✅ worktree로:
  "실험해볼게" → worktree에서 수정 → 실패 → 버리면 끝
                                   → 성공 → 병합
```

### 언제 쓰나

```
✅ worktree가 적합한 경우:
- 리팩터링 실험 (실패하면 버릴 수 있는 작업)
- 두 가지 접근법을 동시에 시도
- PR을 만들면서 동시에 다른 기능 작업
- SubAgent에게 격리된 환경에서 작업 시키기 (Ch.18에서 자세히)

❌ worktree가 불필요한 경우:
- 간단한 버그 수정 (원본에서 바로 해도 됨)
- 파일 하나만 수정하는 작업
```

## 상태 표시줄 (Status Line)

Claude Code는 **작업 중 상태를 실시간으로** 보여준다.

### 확인할 수 있는 것들

```bash
/status     # 현재 상태 확인 — 컨텍스트 사용량, 모델, 권한
/cost       # 토큰과 비용 확인 — 얼마나 썼는지
/memory     # 규칙 파일 확인 — 어떤 CLAUDE.md와 규칙이 읽히는지
```

### 왜 중요한가

```
❌ 상태를 안 보고 작업:
  → 컨텍스트가 꽉 찬 줄 모르고 계속 작업
  → 비용이 예상보다 많이 나감
  → 잘못된 브랜치에서 작업

✅ 상태를 주기적으로 확인:
  → /cost로 토큰 50% 넘으면 /compact 고려
  → /status로 컨텍스트 사용량과 브랜치 확인
  → /memory로 규칙 파일 로딩 상태 확인
```

**습관**: 작업 시작할 때 `/status`와 `/cost` 한 번씩. 30분마다 다시 확인.

## Resume Hygiene — 세션 정리 습관

"resume hygiene"은 **세션을 깔끔하게 이어가는 습관**이다.

### 세션을 끝낼 때

```
1. handoff.md 작성 (또는 업데이트)
   - 끝난 것
   - 막힌 점
   - 다음 단계

2. 필요하면 /compact
   - 컨텍스트가 무거우면 압축

3. 커밋 또는 stash
   - 작업 중인 코드를 저장
```

### 세션을 시작할 때

```
1. 이어갈 것인가, 새로 시작할 것인가?
   - 이어간다 → claude --continue 또는 --resume
   - 새 작업 → claude (새 세션)

2. 이어갔다면:
   - /status로 컨텍스트 상태 확인
   - handoff.md 읽기
   - "어디까지 했지?" 확인 후 작업 시작

3. 새 세션이라면:
   - CLAUDE.md, rules/ 자동 로딩 확인
   - plan.md 있으면 읽기
   - 작업 범위 명시 후 시작
```

### 절대 하지 말 것

```
❌ 아무 생각 없이 새 세션 열기
   → 이전 맥락을 다 잃는다

❌ /compact 없이 2시간 넘게 한 세션
   → 컨텍스트 폭발, 앞부분 규칙 잊음

❌ handoff.md 없이 세션 끝내기
   → "아까 뭐 하고 있었지?" 다음에 반복
```

## 세션 운영 체크리스트

모든 세션에 적용하는 체크리스트:

### 시작 전

```
□ CLAUDE.md, rules/ 확인
□ plan.md 있으면 읽기
□ git status + git log --oneline -5
□ 이어가는 세션이면 handoff.md 확인
```

### 수정 전

```
□ 범위 명시 ("이 파일만", "이 함수만")
□ 완료 기준 ("npm test 통과")
□ 검증 명령 ("npm test -- signup")
```

### 완료 시

```
□ 검증 실행 (테스트, lint, build)
□ handoff.md 갱신 (또는 decision-log.md)
□ /cost 확인 — 다음에 비용 줄일 방법은?
□ 커밋 또는 stash
```

## 파이썬 연결: 파일 입출력

세션은 결국 **상태를 저장하고 불러오는 것**이다. 파이썬에서 이 개념이 바로 **파일 입출력**이다.

### 파일 저장 (세션 끝 = handoff.md 작성)

```python
# 세션 상태를 파일에 저장
session_state = {
    "done": ["로그인 버그 수정", "lint 통과"],
    "blocked": ["리프레시 토큰 미확인"],
    "next": "src/auth/refresh.ts 확인"
}

with open("handoff.md", "w") as f:
    f.write("# 인수인계 메모\n\n")
    f.write("## 끝난 것\n")
    for item in session_state["done"]:
        f.write(f"- {item}\n")
    f.write("\n## 막힌 점\n")
    for item in session_state["blocked"]:
        f.write(f"- {item}\n")
    f.write(f"\n## 다음 단계\n- {session_state['next']}\n")
```

### 파일 불러오기 (세션 시작 = handoff.md 읽기)

```python
# 다음 세션에서 상태 불러오기
with open("handoff.md", "r") as f:
    content = f.read()
    print(content)
```

**세션 관리 = 상태 저장/복원.** 파이썬의 파일 입출력이 하는 일과 같다.

## 왜 이렇게 동작하는가

세션 운영은 소프트웨어 엔지니어링의 **상태 관리(State Management)**다.

웹 앱이 사용자의 로그인 상태를 세션에 저장하듯, Claude Code는 작업 상태를 세션에 저장한다.

- **continue** = 저장된 상태 그대로 복원
- **resume** = 특정 저장점에서 복원
- **fork** = 저장점을 복사해서 새 경로로
- **/compact** = 상태를 요약해서 크기 줄이기
- **handoff.md** = 상태를 사람이 읽을 수 있는 형태로 내보내기

프로그래밍에서 "상태를 잘 관리하면 버그가 줄어든다"는 것처럼, **세션을 잘 관리하면 작업 품질이 올라간다.**

2. **어려운 과제**: 작업 하나를 하면서 세션 운영 체크리스트(시작 전/수정 전/완료 시)를 실제로 따라가보자
