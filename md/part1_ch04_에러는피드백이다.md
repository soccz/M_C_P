# Ch.4 에러는 클로드의 피드백이다

> **이 챕터를 마치면**: 에러 메시지를 보고 당황하지 않고, 읽고, 클로드에게 효과적으로 전달할 수 있다. "안 돼요" 대신 "이렇게 안 됐어요"로 질문하는 습관이 생긴다.

---

## 배경: 에러가 나면 멈추는 사람들

프로그래밍 경험이 없는 사람이 가장 먼저 포기하는 순간이 있다. **빨간 글씨가 나올 때다.**

터미널에서 뭔가를 실행했는데 빨간 글씨가 쭉 나오면, 본능적으로 "내가 뭘 잘못했지"라는 생각이 든다. 그리고 그 빨간 글씨를 읽지 않고 창을 닫는다. 또는 "안 돼요"라고만 말하고 도움을 요청한다.

이건 아깝다. **에러 메시지는 "틀렸어"가 아니라 "여기가 문제야"라는 안내문이기 때문이다.**

의사가 "아프다"고만 하면 진단을 못 한다. "왼쪽 무릎 안쪽이 구부릴 때 아프다"고 하면 바로 검사할 수 있다. 에러 메시지는 컴퓨터가 주는 **증상 설명서**다. 이걸 읽을 줄 알면 절반은 해결된 거다.

> **💭 생각의 흐름**
>
> **문제** — Claude Code에서 파일을 만들라고 했더니 빨간 글씨가 나왔다
> **질문** — "이게 뭐야? 망한 건가?"
> **발견** — 에러 메시지를 읽어보니 "permission denied" — 권한이 없다는 뜻
> **결과** — 다른 폴더에서 다시 시도하니 성공. 에러가 정확히 원인을 알려주고 있었다

## 에러 메시지 읽는 법

에러 메시지는 대부분 이런 구조다:

```
[어디서] [무슨 종류의] 에러: [구체적 설명]
```

### 예시 1: 파일을 못 찾을 때

```
FileNotFoundError: [Errno 2] No such file or directory: 'data.csv'
```

분해하면:
- **FileNotFoundError** — 에러 종류: 파일을 못 찾았다
- **No such file or directory** — 구체적: 그런 파일이나 폴더가 없다
- **'data.csv'** — 어떤 파일: data.csv

**해결**: data.csv가 현재 폴더에 있는지 확인. 경로가 다르거나 파일 이름에 오타가 있을 수 있다.

### 예시 2: 명령어를 못 찾을 때

```
command not found: claude
```

- **command not found** — 그런 명령어가 없다
- **claude** — 어떤 명령어

**해결**: Claude Code가 설치되지 않았거나, 설치 경로가 시스템 PATH에 없다. Ch.2의 설치 과정 다시 확인.

### 예시 3: 권한 문제

```
Error: EACCES: permission denied, open '/etc/config'
```

- **EACCES** — 접근 거부 코드
- **permission denied** — 권한 없음
- **'/etc/config'** — 어떤 파일

**해결**: 이 파일은 관리자 권한이 필요하다. `sudo`를 쓰거나 다른 위치를 사용.

### 예시 4: 네트워크 문제

```
Error: connect ECONNREFUSED 127.0.0.1:3000
```

- **ECONNREFUSED** — 연결 거부
- **127.0.0.1:3000** — 어디에: 내 컴퓨터의 3000번 포트

**해결**: 서버가 실행 중인지 확인. `npm start`나 `python manage.py runserver`를 먼저 실행해야 할 수 있다.

### 예시 5: 파이썬 문법 에러

```
  File "hello.py", line 3
    print("안녕"
          ^
SyntaxError: '(' was never closed
```

- **File "hello.py", line 3** — 어디서: hello.py의 3번째 줄
- **SyntaxError** — 에러 종류: 문법 오류
- **'(' was never closed** — 구체적: 여는 괄호를 닫지 않았다
- **^** — 이 위치에서 발견

**해결**: 3번째 줄에 닫는 괄호 `)` 추가.

## 에러를 읽는 3단계 습관

에러가 나면 이 순서를 따르자:

### 1단계: 마지막 줄부터 읽는다

에러 메시지가 길게 나오면 겁이 난다. 하지만 **가장 중요한 정보는 보통 마지막 줄에 있다.** 파이썬의 경우 "Traceback"이 위에 길게 나오고, 실제 에러 내용은 맨 아래다.

```
Traceback (most recent call last):
  File "app.py", line 45, in main
    result = process_data(raw)
  File "app.py", line 12, in process_data
    return data["key"]
KeyError: 'key'          ← 여기를 먼저 읽는다
```

마지막 줄: `KeyError: 'key'` — 딕셔너리에 'key'라는 키가 없다.

### 2단계: 파일명과 줄 번호를 찾는다

```
File "app.py", line 12
```

app.py의 12번째 줄에서 문제가 발생했다. 이 정보를 클로드에게 전달하면 바로 찾아간다.

### 3단계: 에러 종류로 방향을 잡는다

| 에러 종류 | 의미 | 흔한 원인 |
|----------|------|---------|
| `FileNotFoundError` | 파일 없음 | 경로 오타, 파일 미생성 |
| `SyntaxError` | 문법 오류 | 괄호 안 닫음, 콜론 빠짐 |
| `NameError` | 이름 없음 | 변수명 오타, 미선언 |
| `TypeError` | 타입 불일치 | 숫자+문자열 직접 연산 |
| `KeyError` | 키 없음 | 딕셔너리에 해당 키 없음 |
| `IndexError` | 범위 초과 | 리스트 길이보다 큰 인덱스 |
| `PermissionError` | 권한 없음 | 관리자 권한 필요 |
| `ConnectionError` | 연결 실패 | 서버 미실행, 네트워크 문제 |
| `ModuleNotFoundError` | 모듈 없음 | 패키지 미설치 |
| `command not found` | 명령어 없음 | 프로그램 미설치 |

## "안 돼요" vs "이렇게 안 됐어요"

에러를 클로드에게 전달하는 방법에도 질이 있다.

### 나쁜 전달

```
에러 나요. 안 돼요.
```

클로드: "어떤 에러인가요? 어떤 명령을 실행하셨나요?" — 핑퐁이 시작된다. 시간 낭비.

### 보통 전달

```
파이썬 실행하니까 에러 났어. FileNotFoundError래.
```

클로드: 방향은 잡을 수 있지만, 정확한 파일과 경로를 추가로 물어봐야 한다.

### 좋은 전달

```
python3 app.py를 실행했더니 아래 에러가 나왔어:

Traceback (most recent call last):
  File "app.py", line 12, in process_data
    return data["key"]
KeyError: 'key'

data 변수에 "key" 키가 없는 것 같은데, 원인과 해결법을 알려줘.
```

클로드: 바로 12번째 줄을 보고 `data`의 구조를 확인하고 해결책을 제시한다. **한 번에 끝난다.**

### 에러 전달의 황금 공식

```
[무엇을 했는데] + [전체 에러 메시지 복사] + [내 추측이나 질문]
```

이 세 가지만 있으면 클로드가 거의 한 번에 해결한다.

## 에러를 클로드에게 붙여넣는 습관

Claude Code에서는 더 쉽다. 에러가 나면 **같은 세션에서 바로 보이기 때문이다.**

```
You: python3 app.py 실행해봐
Claude: [실행 결과와 에러가 그대로 보임]
You: 이 에러 원인이 뭐야? 고쳐줘.
```

에러를 복사할 필요도 없다. 클로드가 이미 보고 있다. 이게 Claude Code의 강점이다 — **에러가 대화의 일부가 된다.**

claude.ai Chat에서는 에러를 복사해서 붙여넣어야 하지만, Claude Code에서는 자연스럽게 이어진다.

> **💭 생각의 흐름**
>
> **문제** — 에러가 나면 당황해서 `/clear`로 리셋하곤 했다
> **질문** — "에러가 나면 대화를 리셋해야 하나?"
> **발견** — 아니다. 에러는 맥락의 일부다. 리셋하면 오히려 컨텍스트를 잃는다
> **결과** — 에러가 나면 리셋하지 말고, 그 상태에서 "이거 왜 그래?"라고 이어서 질문하는 습관

## 에러를 두려워하지 않는 마인드셋

### 에러는 실패가 아니다

프로 개발자도 하루에 수십 번 에러를 만난다. 에러 없이 코딩하는 사람은 없다. 에러는 "틀렸다"가 아니라 **"여기를 확인해봐"**라는 신호다.

### 에러를 일부러 만들어보자

두려움을 줄이는 가장 좋은 방법은 **의도적으로 에러를 만들어보는 것**이다. 괄호를 안 닫거나, 따옴표를 빼거나, 0으로 나누거나. 이런 코드를 일부러 실행하면 에러 메시지를 읽는 연습이 된다. (이 챕터 끝의 **실습 1**에서 직접 해본다.)

### 에러 일지 쓰기

처음에는 만나는 에러를 간단히 기록해두면 좋다:

```
날짜: 2026-04-14
에러: ModuleNotFoundError: No module named 'requests'
원인: requests 라이브러리가 설치 안 됨
해결: pip install requests
배운 것: 외부 라이브러리는 따로 설치해야 한다
```

몇 주 지나면 같은 에러가 나와도 "아, 이거 전에 봤지"하고 바로 해결할 수 있다.

## Claude Code에서 자주 만나는 에러 5가지

### 1. Permission denied (파일/도구 접근 거부)

```
Claude wants to use Edit tool on src/config.ts
[Allow / Deny]
```

이건 에러가 아니라 **안전 장치**다. 클로드가 파일을 수정하기 전에 허락을 구하는 것. Allow를 누르면 진행된다. 이게 싫으면 settings.json에서 자동 허용을 설정할 수 있다 (Ch.6).

### 2. Context window 부족

```
Conversation is getting long. Consider using /compact to reduce context.
```

대화가 길어져서 컨텍스트가 꽉 찼다. `/compact`로 압축하거나 `/clear`로 새 대화를 시작하자.

### 3. Tool execution failed

```
Tool execution failed: Command exited with non-zero status
```

클로드가 실행한 명령어가 실패했다. 바로 아래에 실패 원인이 나온다. 대부분 경로 오류, 패키지 미설치, 또는 문법 에러.

### 4. Rate limit

```
Error: Rate limit exceeded. Please wait before making another request.
```

너무 빠르게 너무 많이 요청했다. 잠시 기다리면 풀린다. 플랜에 따라 한도가 다르다.

### 5. Network/Auth 에러

```
Error: Authentication failed. Please run 'claude auth login'.
```

인증이 풀렸다. `claude auth login`으로 다시 로그인.

### 실습 1: 일부러 에러 만들기
터미널에서 한 줄씩 실행해보자:
```bash
python3 -c "print('안녕'"           # SyntaxError
python3 -c "print(hello)"          # NameError
python3 -c "print(1 / 0)"          # ZeroDivisionError
python3 -c "print('나이: ' + 25)"  # TypeError
python3 -c "print([1,2,3][10])"    # IndexError
```

위 에러 중 하나를 Claude Code에서 실행하고, 에러가 나오면 "이 에러 원인이 뭐야? 고쳐줘."라고 이어서 질문해보자.

### 실습 3: 에러 일지 시작하기
오늘 만난 에러 하나를 기록해보자. 형식:
```
에러: [에러 종류]
원인: [왜 났는지]
해결: [어떻게 해결했는지]
배운 것: [한 줄]
```

## 왜 이렇게 동작하는가

에러 메시지가 존재하는 이유는 간단하다: **컴퓨터는 추측하지 않기 때문이다.**

사람은 "print 안녕"이라고 해도 "아, print('안녕') 하려는 거구나"라고 추측할 수 있다. 컴퓨터는 못 한다. 정해진 문법과 다르면 "이건 뭔지 모르겠다"고 정직하게 알려준다. 그게 에러 메시지다.

클로드도 비슷하다. 모호한 질문을 받으면 추측해서 답하는데, 그 추측이 틀리면 원하지 않는 결과가 나온다. 하지만 에러 메시지는 추측하지 않는다. 정확히 어디가 문제인지 알려준다. 그래서 **에러 메시지는 클로드의 답변보다 더 정확한 정보원**일 때가 많다.
