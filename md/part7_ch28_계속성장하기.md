# Ch.28 계속 성장하기

> **이 챕터를 마치면**: 공식 문서와 커뮤니티 자료를 통해 스스로 학습을 이어갈 수 있다. 초보자가 흔히 빠지는 오해를 인식하고, 상황별로 어떤 자료를 찾아봐야 하는지 판단할 수 있다.

---

## 배경: 이 책을 덮은 후

27개 챕터를 읽었다. 기억, 설정, 규칙, 프롬프트, 스킬, MCP, 세션, SubAgent, Cowork, 파이프라인 — 많은 것을 배웠다.

하지만 이 도구는 **계속 변한다.** 이 책을 쓰는 동안에도 새 기능이 추가되고, 기존 기능이 바뀌었다. Channels는 아직 Research Preview이고, Plugin 생태계는 매주 커지고 있다.

이 마지막 챕터는 "더 배울 것"이 아니라 **"어떻게 배울 것인가"**에 대한 것이다.

> **💭 생각의 흐름**
>
> **문제** — 이 책의 내용 중 일부는 이미 바뀌었을 수 있다. 새 기능도 계속 나온다
> **질문** — "최신 정보를 어떻게 따라가지?"
> **발견** — 공식 문서 + Release Notes + 커뮤니티. 이 세 가지를 주기적으로 확인하면 된다. 그리고 가장 좋은 방법은 **클로드에게 직접 물어보는 것**이다
> **결론** — 이 책의 방법론("1인칭 발화")은 책이 끝나도 유효하다. 궁금하면 클로드에게 물어보면 된다

## 공식 학습 경로

TOC에서 정한 학습 순서를 따르면:

```
공식 학습 경로 순서:

1. Release Notes
   → 이 책 이후 뭐가 바뀌었나 먼저 확인

2. "What are Skills?" (Anthropic Help Center)
   → Skill의 최신 정의와 개념 확인

3. "Use Skills in Claude" (Anthropic Help Center)
   → Skill 실제 사용법과 최신 문법

4. Claude Code Docs (docs.anthropic.com)
   → Settings, Skills, Plugins, MCP, Hooks,
     Scheduled Tasks, Remote Control, Channels, Agent Teams

5. Anthropic 공식 PDF 가이드
   → 체계적인 전체 참조 문서
```

### 1단계: Release Notes — 먼저 변경사항 확인

새 버전이 나올 때마다 무엇이 바뀌었는지 확인한다.

```
Release Notes 확인 방법:

1. GitHub Releases
   · github.com/anthropics/claude-code/releases
   · 가장 상세한 변경 이력

2. 공식 문서 Changelog
   · 주요 변경사항 요약

3. Claude Code 내에서
   · /release-notes 명령으로 최근 변경사항 확인
```

**Release Notes를 읽는 법:**

```
모든 항목을 읽지 마라. 이것만 확인:

1. Breaking Changes — 기존에 되던 것이 안 되게 됐나?
2. New Features — 내 워크플로우에 쓸 수 있는 새 기능이 있나?
3. Deprecations — 곧 사라질 기능을 쓰고 있나?

나머지는 필요할 때 찾아보면 된다.
```

### 2단계: 공식 문서

Claude Code의 공식 문서는 가장 신뢰할 수 있는 소스다.

```
공식 문서 구조:

docs.anthropic.com
  └── Claude Code
        ├── Overview          ← 전체 개요
        ├── Getting Started   ← 설치와 첫 시작
        ├── Settings          ← settings.json 상세
        ├── Skills            ← Skill 작성법
        ├── Plugins           ← Plugin 구조
        ├── MCP              ← MCP 서버 연결
        ├── Hooks            ← Hook 설정
        ├── Scheduled Tasks  ← 예약 작업
        ├── Remote Control   ← 원격 조종
        ├── Channels         ← 외부 이벤트 수신
        ├── Agent Teams      ← 에이전트 팀
        └── Changelog        ← 변경 이력
```

**추천 읽기 순서:**

```
이미 이 책을 읽었다면:
  1. Changelog — 이 책 이후 뭐가 바뀌었나
  2. Skills — Skill 작성의 최신 문법
  3. Hooks — Hook 타입과 matcher 패턴
  4. Scheduled Tasks — 예약 작업 최신 옵션

처음 접한다면:
  1. Overview → Getting Started
  2. Settings
  3. Skills → MCP
  4. 나머지는 필요할 때
```

커뮤니티 자료 탐색 사이트:

**주의사항:**

```

  · 버전 차이 주의 (오래된 자료일 수 있음)
  · 서드파티 사이트는 폐쇄·이전될 수 있음

❌ 무작정 복사하지 않기
  · 다른 사람의 CLAUDE.md를 그대로 복사하면 내 상황에 안 맞을 수 있다
  · Skill을 설치하기 전에 SKILL.md를 읽어보자
  · 보안에 민감한 설정(MCP, permissions)은 특히 주의
```

## 초보자 흔한 오해 12가지

이 책을 읽은 후에도 빠지기 쉬운 오해들.

### 오해 1: "CLAUDE.md를 길게 쓸수록 좋다"

```
현실:
  · 길수록 Attention Budget(Ch.9)을 많이 소비
  · 규칙이 많으면 서로 충돌 가능
  · 클로드가 모든 규칙을 같은 비중으로 따르지 않음

해결:
  · CLAUDE.md는 30줄 이하
  · 상세 규칙은 .claude/rules/로 분산
  · 정기적으로 불필요한 규칙 정리
```

### 오해 2: "MCP 서버를 많이 연결할수록 강력하다"

```
현실:
  · 각 MCP 서버가 도구 목록을 컨텍스트에 추가
  · 10개 서버 × 5개 도구 = 50개 도구 설명이 매 턴 로드
  · 토큰 비용 폭발, 도구 선택 혼란

해결:
  · 실제로 쓰는 MCP 서버만 연결 (Ch.13)
  · 프로젝트별로 다른 MCP 구성
  · 안 쓰는 서버는 비활성화
```

### 오해 3: "bypassPermissions가 편하니까 항상 쓰자"

```
현실:
  · 실수로 중요한 파일을 삭제/수정할 수 있음
  · 팀에서는 보안 리스크
  · 디버깅 시 "내가 뭘 허용했더라?" 추적 어려움

해결:
  · 기본은 default 또는 acceptEdits
  · bypassPermissions는 신뢰할 수 있는 자동화에만
  · 새 프로젝트에서는 항상 default로 시작
```

### 오해 4: "한 세션에서 다 하면 효율적이다"

```
현실:
  · 세션이 길어질수록 컨텍스트 누적 → 비용 증가
  · 무관한 작업이 섞이면 클로드가 혼란
  · /compact해도 정보 손실 가능

해결:
  · 한 세션 = 한 작업 (Ch.16)
  · 작업 완료 후 새 세션
  · 이어야 하면 --continue, 다른 작업이면 새 세션
```

### 오해 5: "프롬프트만 잘 쓰면 된다"

```
현실:
  · 프롬프트는 시작점일 뿐 (Ch.10)
  · 컨텍스트(Ch.9)와 하네스(Ch.11)가 더 중요
  · 같은 프롬프트도 CLAUDE.md에 따라 결과가 다름

해결:
  · 프롬프트 < 컨텍스트 < 하네스 순서로 투자
  · CLAUDE.md와 rules/를 먼저 다듬고, 그 다음 프롬프트
```

### 오해 6: "클로드가 틀리면 클로드 탓이다"

```
현실:
  · 대부분 컨텍스트가 부족하거나 모호한 지시 때문
  · "알아서 해줘" → 결과가 원하는 것과 다름
  · 규칙이 충돌하면 예측 불가능한 결과

해결:
  · "이렇게 안 됐어요"로 피드백 (Ch.4)
  · 명확한 완료 기준(done-when) 포함
  · 결과가 다르면 입력(컨텍스트)을 먼저 점검
```

### 오해 7: "Skill을 만들면 무조건 좋다"

```
현실:
  · 한 번 쓸 작업을 Skill로 만들면 과잉 설계
  · Skill 유지보수 비용이 있음
  · 프로젝트가 바뀌면 Skill도 바뀌어야 함

해결:
  · 3번 이상 반복하는 작업만 Skill로 (Ch.12)
  · 처음에는 프롬프트로 해보고, 패턴이 안정되면 Skill화
```

### 오해 8: "SubAgent를 많이 쓰면 빠르다"

```
현실:
  · 각 SubAgent가 별도 컨텍스트 → 메모리 사용 증가
  · SubAgent 간 조율 비용이 있음
  · 단순한 작업을 SubAgent로 나누면 오히려 느림

해결:
  · 독립적이고 병렬 가능한 작업에만 사용 (Ch.18)
  · 2~3개가 적정, 10개는 과잉
```

### 오해 9: "Cowork이면 Claude Code가 필요 없다"

```
현실:
  · Cowork은 위임, Claude Code는 협업 — 용도가 다름 (Ch.20)
  · 코딩, git, 터미널이 필요하면 Claude Code
  · 보고서, 정리, 분석이면 Cowork

해결:
  · 둘 다 쓸 수 있다 — 결정 트리로 판단 (Ch.20)
```

### 오해 10: "자동화할 수 있으면 다 자동화하자"

```
현실:
  · 판단이 필요한 작업은 자동화에 부적합
  · 자동화된 실수는 수동 실수보다 피해가 큼 (빠르게 반복되니까)
  · 자동화 설정·유지보수 비용이 있음

해결:
  · "실패해도 괜찮은 작업"부터 자동화
  · 중요한 결과물은 반드시 사람이 확인
  · 자동화 = "사람의 시간을 아끼는 것"이지 "사람을 빼는 것"이 아님
```

### 오해 11: "한 번에 완벽한 설정을 만들 수 있다"

```
현실:
  · 워크플로우는 반복하면서 다듬는 것 (Ch.25)
  · 처음 설정이 완벽할 수 없음
  · 업무가 바뀌면 설정도 바뀌어야 함

해결:
  · 최소 설정으로 시작 → 결과 보고 다듬기
  · "멈추는 기준" 설정 → 기준에 도달하면 다음 단계
```

### 오해 12: "이 책에 있는 기능이 영원히 유지된다"

```
현실:
  · 기능은 추가, 변경, 폐기될 수 있음
  · Channels는 Research Preview (바뀔 수 있음)
  · 명령어 이름이나 옵션이 변경될 수 있음

해결:
  · Release Notes를 정기적으로 확인
  · "이 기능이 아직 있나?" → 클로드에게 물어보면 된다
  · 원리(Context Engineering, Harness Engineering)는 기능보다 오래 간다
```

## 클로드에게 물어보기 — 이 책의 방법론은 계속된다

Part 0에서 이 책의 철학을 소개했다:

> "클로드에게 클로드를 물어본다."

이 방법론은 이 책이 끝나도 유효하다.

```
새 기능이 궁금할 때:
  "Claude Code에 최근 추가된 기능이 뭐야?"

사용법이 헷갈릴 때:
  "settings.json의 hooks에서 PostToolUse matcher를 어떻게 작성해?"

문제가 생겼을 때:
  "이 에러 메시지가 뭘 의미하는지 설명해줘: [에러 메시지]"

최적화하고 싶을 때:
  "이 CLAUDE.md를 더 효율적으로 만들 수 있어?"
```

다만 Part 0에서 강조했듯이, 클로드의 답변은 **검증이 필요**하다:

```
검증 방법:
  1. 공식 문서와 대조
  2. 실제로 실행해서 확인
  3. "확실해?" 라고 한 번 더 물어보기
  4. 버전 차이 고려 (클로드가 최신 버전을 모를 수 있음)
```

## 마무리: 도구는 바뀌고, 원리는 남는다

이 책에서 배운 것을 두 종류로 나누면:

```
바뀌는 것 (도구):
  · 명령어 이름과 옵션
  · 설정 파일 형식
  · 기능의 추가/변경/폐기
  · MCP 서버 목록
  · Plugin 생태계

남는 것 (원리):
  · Context Engineering — 좋은 입력이 좋은 출력을 만든다
  · Harness Engineering — 구조가 품질을 결정한다
  · Prompt Engineering — 명확한 지시가 명확한 결과를 만든다
  · 관심사의 분리 — 각 부분이 자기 역할만 한다
  · 점진적 개선 — 작게 시작하고, 결과를 보고, 다듬는다
  · 검증 — 결과를 확인하기 전에 완료라고 하지 않는다
```

도구가 Claude Code에서 다른 것으로 바뀌어도, 이 원리는 적용된다. **"어떤 도구를 쓰느냐"보다 "어떻게 구조를 잡느냐"가 중요하다.**

이 책의 진짜 목적은 Claude Code 사용법을 가르치는 것이 아니었다. **"AI 도구와 효과적으로 일하는 사고방식"**을 심는 것이었다.
