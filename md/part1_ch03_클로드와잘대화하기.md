# Ch.3 클로드와 잘 대화하기

> **이 챕터를 마치면**: 같은 질문을 해도 왜 결과가 다른지 이해하고, 좋은 프롬프트를 쓸 수 있다. Slash Commands를 자유롭게 쓸 수 있고, 파이썬의 변수와 자료형이 뭔지 안다.

---

## 배경: 같은 질문인데 왜 결과가 다르지?

클로드를 처음 쓰는 사람은 이런 경험을 한다:

- "코드 짜줘" → 기대와 완전 다른 코드가 나옴
- "요약해줘" → 너무 길거나 너무 짧음
- "파일 수정해줘" → 엉뚱한 파일을 건드림

그래서 "AI가 아직 멍청하네"라고 생각한다. 하지만 대부분의 경우 **문제는 클로드가 아니라 질문에 있다**.

사람한테 일을 시킬 때도 마찬가지다. "그거 좀 해줘"라고 하면 결과가 들쭉날쭉하지만, "A 폴더에 있는 report.csv를 읽어서 매출 상위 10개를 뽑고 표로 정리해줘"라고 하면 원하는 결과가 나온다.

클로드도 같다. **구체적으로 말할수록 결과가 좋아진다.**

> **💭 생각의 흐름**
>
> **문제** — "이 코드 리팩터링해줘"라고 했더니 코드가 완전히 달라졌다
> **질문** — "내가 뭘 잘못 말한 거지?"
> **발견** — "리팩터링"의 범위를 안 정해줬다. 클로드는 전체를 바꿔도 된다고 판단했다
> **결과** — "함수 이름만 snake_case로 바꿔줘. 로직은 건드리지 마."로 다시 시도 → 성공

## 나쁜 질문 vs 보통 질문 vs 좋은 질문

### 상황 1: 버그 수정

**나쁜:**
```
버그 고쳐줘.
```
→ 어떤 버그? 어떤 파일? 클로드가 전체 프로젝트를 뒤지기 시작한다. 시간도 오래 걸리고 엉뚱한 곳을 건드릴 수 있다.

**보통:**
```
로그인 페이지 버그 고쳐줘. 비밀번호가 틀려도 로그인되는 문제야.
```
→ 어디가 문제인지는 안다. 하지만 클로드가 어떤 파일을 봐야 하는지, 어떻게 확인해야 하는지는 모른다.

**좋은:**
```
src/auth/login.ts에서 비밀번호 검증 로직에 버그가 있어.
비밀번호가 틀려도 로그인이 되는 문제야.
바로 고치지 말고 아래 순서로 진행해:
1. 현재 경로 확인
2. 원인 가설
3. 수정 범위
4. 테스트 계획
을 먼저 정리하고, 내가 확인한 뒤 수정해.
```
→ 파일 위치, 증상, 진행 순서, 확인 절차까지. 이런 프롬프트에서 실수가 발생할 여지가 거의 없다.

---

### 상황 2: 요약

**나쁜:**
```
이거 요약해줘.
```

**보통:**
```
이 보고서를 3줄로 요약해줘.
```

**좋은:**
```
이 보고서를 읽고 다음 형식으로 요약해줘:
- 핵심 결론 1줄
- 근거 데이터 2개
- 우리 팀에 영향 있는 부분 1줄
전체 5줄 이내로.
```

---

### 상황 3: 파일 생성

**나쁜:**
```
설정 파일 만들어줘.
```

**보통:**
```
프로젝트 설정 파일을 만들어줘. JSON 형식으로.
```

**좋은:**
```
.claude/settings.json을 만들어줘.
내용:
- Read, Edit, Write, Glob, Grep 허용
- .env 파일 읽기 금지
- Bash(npm test)와 Bash(npm run lint) 허용
다른 권한은 기본값으로.
```

## 좋은 질문의 5가지 패턴

위 예시들을 보면 좋은 질문에 공통점이 있다:

### 1. 구체적 위치를 말한다
```
❌ "코드 수정해줘"
✅ "src/auth/login.ts의 validatePassword 함수를 수정해줘"
```

### 2. 원하는 결과의 형태를 말한다
```
❌ "분석해줘"
✅ "표로 정리해줘. 열은: 이름, 날짜, 금액. 5건 이내로."
```

### 3. 하지 말아야 할 것을 말한다
```
❌ "리팩터링해줘"
✅ "함수 이름만 바꿔줘. 로직은 건드리지 마. 테스트 파일도 건드리지 마."
```

### 4. 진행 순서를 정해준다
```
❌ "테스트 추가해줘"
✅ "먼저 기존 테스트를 확인하고, 빠진 케이스를 나열한 뒤, 하나씩 추가해줘."
```

### 5. 완료 기준을 말한다
```
❌ "될 때까지 해줘"
✅ "npm test가 모두 통과하면 완료야."
```

이 5가지를 다 넣을 필요는 없다. 상황에 따라 1-2개만 추가해도 결과가 크게 달라진다.

## 컨텍스트가 전부다

"컨텍스트"라는 단어가 자주 나올 거다. 이건 **클로드가 현재 알고 있는 모든 정보**를 뜻한다.

사람과 대화할 때를 생각해보자. 처음 만난 사람에게 "그거 어떻게 됐어?"라고 물으면 "뭐가요?"라고 되묻는다. 하지만 어제 같이 프로젝트를 한 동료에게 같은 말을 하면 바로 이해한다. **공유된 맥락(컨텍스트)이 있기 때문이다.**

클로드도 마찬가지다:
- **컨텍스트가 없으면**: 매번 0부터 설명해야 한다
- **컨텍스트가 있으면**: "아, 그 프로젝트 말이지. 지난번에 여기까지 했으니까..."

Claude Code에서 컨텍스트를 제공하는 방법:

| 방법 | 예시 | 지속성 |
|------|------|--------|
| **대화에서 직접 말하기** | "이 프로젝트는 React + TypeScript야" | 이번 세션만 |
| **CLAUDE.md에 적기** | 프로젝트 규칙, 구조 설명 | 영구 |
| **rules/ 파일** | 경로별 세부 규칙 | 영구 |
| **파일을 읽게 하기** | "먼저 README.md를 읽어봐" | 이번 세션 |
| **Memory** | 글로벌 선호도 | 영구 |

Ch.5(클로드의 기억)에서 이걸 자세히 다룬다. 지금은 **"클로드에게 맥락을 주면 결과가 좋아진다"**만 기억하자.

## Slash Commands 실전 가이드

Ch.2에서 5개만 기억하라고 했다. 이제 나머지도 알아보자. 대화 중에 `/`를 입력하면 쓸 수 있는 명령어들이다.

### 세션 관리

| 명령어 | 하는 일 | 언제 쓰나 |
|--------|---------|----------|
| `/clear` | 대화를 비우고 새로 시작 | 주제가 바뀔 때 |
| `/compact [focus]` | 컨텍스트를 압축 (요약 후 축소) | 대화가 길어졌을 때 |

**`/compact`가 특히 중요하다.** 클로드와 오래 대화하면 컨텍스트가 꽉 찬다. 그러면 앞부분을 잊기 시작한다. `/compact`를 쓰면 지금까지 대화를 요약해서 줄인다. 중요한 정보는 남기고 잡담은 버린다.

> **💭 생각의 흐름**
>
> **문제** — 한 시간째 대화했더니 클로드가 처음에 말한 규칙을 잊었다
> **질문** — "왜 까먹었지? 아까 분명히 말했는데"
> **발견** — 컨텍스트 윈도우에 한계가 있다. 오래 대화하면 앞이 밀려난다
> **결과** — `/compact` 습관을 들이고, 중요한 건 CLAUDE.md에 적어두기로 함 → Ch.5

### 정보 확인

| 명령어 | 하는 일 | 언제 쓰나 |
|--------|---------|----------|
| `/memory` | 어떤 memory 파일과 CLAUDE.md가 읽히는지 확인 | 규칙이 적용되는지 확인할 때 |
| `/cost` | 이번 세션의 토큰 사용량과 비용 | 비용 감각 기르기 |
| `/status` | 현재 상태 전반 (컨텍스트 사용량, 모델, 권한 등) | 뭔가 이상하거나 현재 상태 확인할 때 |
| `/permissions` | 현재 권한 설정 | 왜 거부당하는지 확인할 때 |

### 모델과 작업

| 명령어 | 하는 일 | 언제 쓰나 |
|--------|---------|----------|
| `/model` | 모델 전환 (opus/sonnet/haiku) | 복잡한 작업엔 opus, 단순하면 haiku |
| `/init` | CLAUDE.md 초안 자동 생성 | 새 프로젝트 시작할 때 |

### 언제 뭘 쓰나 — 3가지 시나리오

**시나리오 1: 새 프로젝트 시작**
```
/init          ← CLAUDE.md 초안 생성
/model         ← 적절한 모델 선택
/permissions   ← 권한 확인
```

**시나리오 2: 긴 대화 중간**
```
/cost          ← 비용 확인
/status        ← 컨텍스트 사용량 확인
/compact       ← 필요하면 압축
```

**시나리오 3: 작업 마무리**
```
/cost          ← 최종 비용 확인
/clear         ← 다음 작업을 위해 리셋
```

## 🐍 변수와 자료형 — 입력 → 처리 → 출력

클로드에게 질문하면 이런 과정을 거친다:

```
입력(프롬프트) → 처리(클로드의 추론) → 출력(응답)
```

파이썬도 동일하다:

```
입력(데이터) → 처리(코드) → 출력(결과)
```

이 "입력"에 해당하는 것이 **변수**이고, 변수에 담기는 것의 종류가 **자료형**이다.

### 변수: 이름표

변수는 **값에 붙이는 이름표**다. 상자가 아니다.

```python
name = "클로드"
age = 2
is_ai = True
```

- `name`이라는 이름표를 "클로드"라는 글자에 붙였다
- `age`라는 이름표를 2라는 숫자에 붙였다
- `is_ai`라는 이름표를 True(참)라는 값에 붙였다

변수를 쓰는 이유: **같은 값을 반복해서 쓸 때 편하다.**

```python
print("안녕하세요, 클로드님!")
print("클로드님의 나이는 2살입니다.")
print("클로드님은 AI입니다.")
```

이걸 변수로 바꾸면:

```python
name = "클로드"
age = 2

print(f"안녕하세요, {name}님!")
print(f"{name}님의 나이는 {age}살입니다.")
print(f"{name}님은 AI입니다.")
```

`name`을 "제미나이"로 바꾸면 세 줄이 다 바뀐다. 하나하나 고칠 필요 없다.

### 자료형: 값의 종류

파이썬에서 값은 종류가 있다:

| 자료형 | 이름 | 예시 | 쉽게 말하면 |
|--------|------|------|----------|
| `str` | 문자열 | `"안녕"`, `"hello"` | 글자 |
| `int` | 정수 | `42`, `-7`, `0` | 소수점 없는 숫자 |
| `float` | 실수 | `3.14`, `-0.5` | 소수점 있는 숫자 |
| `bool` | 불리언 | `True`, `False` | 참/거짓 |
| `None` | 없음 | `None` | 값이 없다 |

왜 종류가 중요한가? **종류에 따라 할 수 있는 일이 다르기 때문이다.**

```python
# 숫자끼리는 더할 수 있다
print(3 + 5)        # 8

# 문자열끼리는 이어붙일 수 있다
print("안녕" + "하세요")  # 안녕하세요

# 숫자와 문자열은 바로 못 더한다
# print(3 + "살")   # ← 에러!

# 이렇게 변환해야 한다
print(str(3) + "살")  # 3살
```

### type()으로 확인하기

뭔지 모르겠으면 `type()`으로 확인한다:

```python
print(type("안녕"))    # <class 'str'>
print(type(42))        # <class 'int'>
print(type(3.14))      # <class 'float'>
print(type(True))      # <class 'bool'>
print(type(None))      # <class 'NoneType'>
```

### 클로드와의 연결

클로드에게 질문할 때도 "자료형"이 있다고 생각하면 이해가 쉽다:

- **문자열(str)** = 일반 텍스트 질문: `"이 코드 설명해줘"`
- **구조화된 데이터** = JSON 형식: `{"task": "리뷰", "scope": "src/"}`
- **파일** = 컨텍스트 첨부: `CLAUDE.md`, `plan.md` 등

클로드가 받는 입력의 "자료형"이 정확할수록 출력도 정확해진다. "버그 고쳐줘"(모호한 문자열)보다 구체적인 구조를 주면 좋은 이유가 이거다.

### input() — 사용자 입력 받기

```python
name = input("이름이 뭐예요? ")
print(f"안녕하세요, {name}님!")
```

실행하면:
```
이름이 뭐예요? 클로드
안녕하세요, 클로드님!
```

`input()`은 사용자가 키보드로 입력한 값을 변수에 담는다. Claude Code에서 클로드가 "이 파일을 수정해도 될까요?"라고 물어보는 것도 일종의 `input()`이다 — 사용자의 승인을 입력받는 것.

## 잘 안 된다면

| 증상 | 원인 | 해결 |
|------|------|------|
| 클로드가 엉뚱한 파일을 수정 | 파일 경로를 안 말함 | 정확한 경로를 프롬프트에 포함 |
| 결과가 너무 길다 | 길이 제한을 안 줌 | "3줄 이내로", "표로" 등 형식 지정 |
| 클로드가 전체를 바꿈 | 범위를 안 정함 | "이 함수만", "이 줄만" 범위 지정 |
| `/compact` 후 잊어버림 | 중요 정보가 압축됨 | 중요한 건 CLAUDE.md에 적기 |

---

## 왜 이렇게 동작하는가

클로드가 좋은 질문에 더 잘 답하는 이유는 단순하다: **모호함이 줄어들기 때문이다.**

"버그 고쳐줘"에는 가능한 해석이 수백 가지다. 클로드는 그중 하나를 고르는데, 내가 원하는 것과 다를 확률이 높다. "src/auth/login.ts의 validatePassword에서 비밀번호 검증이 빠져있어"에는 가능한 해석이 1-2개다. 맞출 확률이 압도적으로 높다.

이건 나중에 배울 **Context Engineering**(Ch.9)의 핵심이기도 하다. 지금은 "구체적으로 말하자"를 실천하면 충분하다.
