# Ch.7 클로드의 규칙

> **이 챕터를 마치면**: CLAUDE.md, rules/, settings.json이 왜 따로 존재하는지 이해하고, 경로별 세부 규칙(path-specific rules)을 직접 만들 수 있다.

---

## 배경: 왜 규칙 파일이 3개나 필요한가?

Ch.5에서 CLAUDE.md를, Ch.6에서 settings.json을 배웠다. 그런데 `.claude/rules/` 폴더도 있다. 비슷한 파일이 왜 3개나 필요할까?

회사에 비유하면 명확해진다:

- **CLAUDE.md** = 회사 사훈. "우리는 이런 회사다, 이런 원칙을 따른다." 모든 부서가 알아야 하는 공통 규칙.
- **settings.json** = 출입증 시스템. "이 구역은 출입 가능, 저 구역은 금지." 허용과 차단의 물리적 경계.
- **rules/*.md** = 부서별 업무 매뉴얼. "마케팅팀은 이렇게, 개발팀은 저렇게." 경로(폴더)마다 다른 세부 규칙.

CLAUDE.md에 모든 규칙을 다 넣으면 어떻게 될까? 50줄, 100줄로 늘어난다. 클로드는 매 세션마다 이걸 전부 읽는다. **컨텍스트 낭비**다. API 파일을 작업할 때 프론트엔드 규칙까지 읽을 필요가 없다.

rules/로 분리하면 **해당 경로를 작업할 때만** 관련 규칙이 읽힌다. 컨텍스트를 아끼면서 정확한 규칙을 적용하는 방법이다.

> **💭 생각의 흐름**
>
> **문제** — CLAUDE.md가 80줄이 넘었다. API 규칙, 테스트 규칙, 프론트엔드 규칙이 섞여있다
> **질문** — "이걸 한 파일에 다 넣어야 해?"
> **발견** — rules/ 폴더에 경로별로 쪼개면 해당 작업할 때만 읽힌다
> **결과** — CLAUDE.md는 10줄 공통 규칙만, 나머지는 rules/api.md, rules/testing.md 등으로 분리

---

## 세 파일의 역할 비교

| | CLAUDE.md | rules/*.md | settings.json |
|---|-----------|------------|---------------|
| **비유** | 회사 사훈 | 부서별 매뉴얼 | 출입증 시스템 |
| **내용** | 공통 운영 규칙 | 경로별 세부 규칙 | 허용/금지 경계 |
| **읽히는 때** | 매 세션 항상 | 해당 경로 작업 시 | 매 세션 항상 |
| **형식** | Markdown | Markdown (YAML frontmatter) | JSON |
| **위치** | 프로젝트 루트 | `.claude/rules/` | `.claude/` |
| **git에 올림?** | 예 | 예 | 예 (local은 제외) |
| **분리하는 기준** | — | 경로(폴더) 기준 | — |

**핵심 원칙:**
- **공통 규칙** (10줄 이내) → CLAUDE.md
- **경로별 세부 규칙** → rules/
- **허용/금지의 기술적 경계** → settings.json

## path-specific rules 실전

### 기본 구조

rules/ 폴더 안에 `.md` 파일을 만든다. 파일 이름은 자유지만 역할이 드러나는 이름이 좋다.

```
.claude/rules/
├── api.md          ← src/api/ 경로에 적용
├── testing.md      ← tests/ 경로에 적용
├── frontend.md     ← src/components/ 경로에 적용
└── docs.md         ← docs/ 경로에 적용
```

### 규칙 파일의 형식

YAML frontmatter로 적용 경로를 지정하고, 그 아래에 규칙을 쓴다:

```markdown
---
paths:
  - "src/api/**/*.ts"
  - "src/handlers/**/*.ts"
---

# API 규칙

- 모든 요청 본문은 zod로 검증한다
- 응답은 항상 `{ data, error }` 형식을 따른다
- 내부 스택 트레이스는 절대 노출하지 않는다
- 에러 코드는 HTTP 표준을 따른다
- 새 엔드포인트를 추가하면 반드시 테스트도 같이 추가한다
```

**`paths`** 필드가 핵심이다. 여기에 지정한 경로의 파일을 작업할 때만 이 규칙이 클로드에게 읽힌다.

### 경로 패턴 문법

| 패턴 | 의미 | 예시 |
|------|------|------|
| `*` | 아무 파일 | `*.ts` = 모든 .ts 파일 |
| `**` | 아무 깊이의 폴더 | `src/**/*.ts` = src 하위 모든 .ts |
| `?` | 아무 글자 1개 | `file?.txt` = file1.txt, fileA.txt |

### 실전 예시 1: 테스트 규칙

```markdown
---
paths:
  - "tests/**/*"
  - "**/*.test.ts"
  - "**/*.spec.ts"
---

# 테스트 규칙

- 테스트 파일에서는 실제 DB나 외부 API를 호출하지 않는다
- mock은 최소한으로 사용한다. 가능하면 실제 모듈을 사용한다
- 각 테스트는 독립적이어야 한다 (다른 테스트에 의존 금지)
- describe/it 구조로 작성한다
- 테스트 이름은 "어떤 상황에서 어떤 결과가 나와야 한다" 형식으로
```

### 실전 예시 2: 프론트엔드 규칙

```markdown
---
paths:
  - "src/components/**/*"
  - "src/pages/**/*"
---

# 프론트엔드 규칙

- 컴포넌트는 함수형으로만 작성한다
- CSS-in-JS 대신 Tailwind CSS를 사용한다
- 상태 관리는 zustand를 사용한다
- 컴포넌트 파일 하나에 하나의 export만
- Props 타입은 컴포넌트 파일 상단에 정의한다
```

### 실전 예시 3: 문서 규칙

```markdown
---
paths:
  - "docs/**/*"
  - "*.md"
---

# 문서 규칙

- 한국어로 작성한다
- 제목은 #, 소제목은 ##까지만 사용한다
- 코드 예시는 반드시 실행 가능한 코드로
- 각 문서의 첫 줄은 한 줄 요약이어야 한다
```

---

## 왜 규칙을 파일별로 쪼개야 하는가

### 이유 1: 컨텍스트 절약

CLAUDE.md에 API 규칙 + 테스트 규칙 + 프론트 규칙이 다 들어있으면, 프론트 작업할 때 API 규칙도 같이 읽힌다. 토큰 낭비다.

rules/로 분리하면 **필요한 규칙만** 읽힌다. 프론트 작업에는 frontend.md만, API 작업에는 api.md만.

### 이유 2: 유지보수

80줄짜리 CLAUDE.md는 읽기 어렵고 수정할 때 실수하기 쉽다. 규칙 파일이 분리되어 있으면 해당 영역 담당자가 자기 규칙만 수정하면 된다.

### 이유 3: 팀 협업

프론트엔드 개발자와 백엔드 개발자가 같은 CLAUDE.md를 수정하면 충돌이 생긴다. rules/로 분리하면 각자 자기 영역의 규칙 파일만 수정한다.

### 이유 4: 점진적 적용

처음에는 CLAUDE.md 하나로 시작하고, 규칙이 늘어나면 하나씩 rules/로 분리한다. 한 번에 다 만들 필요 없다.

## CLAUDE.md를 깔끔하게 유지하는 법

### Before: 비대한 CLAUDE.md (80줄)

```markdown
# 프로젝트 규칙
## 명령
- test: npm test
- lint: npm run lint
## API 규칙
- zod로 검증
- { data, error } 형식
- 스택 트레이스 노출 금지
## 프론트 규칙
- 함수형 컴포넌트만
- Tailwind CSS
- zustand
## 테스트 규칙
- 실제 DB 호출 금지
- mock 최소화
- describe/it 구조
## 문서 규칙
- 한국어
- 코드 예시 필수
... (50줄 더)
```

### After: 깔끔한 CLAUDE.md (10줄) + rules/

```markdown
# 프로젝트 작업 계약서

## 명령
- test: npm test
- lint: npm run lint
- build: npm run build

## 규칙
- .env 파일 수정 금지
- 테스트 통과 전에는 끝났다고 하지 않는다
- 변경은 작고 검토 가능하게 유지한다
```

나머지는:
- `.claude/rules/api.md` → API 규칙
- `.claude/rules/testing.md` → 테스트 규칙
- `.claude/rules/frontend.md` → 프론트 규칙
- `.claude/rules/docs.md` → 문서 규칙

## 파일 네이밍 규칙

프로젝트에서 파일 이름을 일관성 있게 짓는 것도 중요하다. 마스터 가이드에서 권장하는 규칙:

### 날짜 기반 파일

```
YYYY-MM-DD_topic_type.md
```

예시:
- `2026-04-14_auth-bug_report.md`
- `2026-04-14_weekly-brief_v1.md`

### 버전 기반 파일

```
name_v1.md, name_v2.md
```

예시:
- `client-a_weekly-brief_v1.md`
- `client-a_weekly-brief_v2.md`

### 규칙 파일

이름이 역할을 드러내야 한다:
- `api.md` ← 무엇에 대한 규칙인지 바로 알 수 있다
- `testing.md`
- `frontend.md`
- `architecture.md`

**나쁜 이름:**
- `rules1.md` ← 뭔지 모른다
- `new-rules.md` ← 언제까지 new인가
- `misc.md` ← 잡동사니

### 실습 4: 규칙 적용 확인
rules/testing.md를 만든 뒤, 테스트 파일을 수정하는 작업을 시켜보자. 클로드가 testing.md의 규칙을 따르는지 확인.

## 잘 안 된다면

| 증상 | 원인 | 해결 |
|------|------|------|
| rules/ 파일이 적용 안 됨 | paths 패턴이 잘못됨 | 실제 파일 경로와 패턴을 비교. `**` 잊지 말기 |
| 규칙이 무시됨 | CLAUDE.md와 충돌 | CLAUDE.md의 규칙이 우선. rules/와 겹치는 내용은 CLAUDE.md에서 제거 |
| 어떤 규칙이 적용되는지 모르겠음 | | 클로드에게 "지금 어떤 규칙 파일이 읽히고 있어?"라고 물어본다 |
| YAML frontmatter 에러 | `---` 누락 또는 들여쓰기 | 시작과 끝에 `---` 확인, paths 앞에 공백 2칸 |

## 왜 이렇게 동작하는가

규칙 파일이 분리되는 이유는 **Progressive Disclosure(점진적 공개)** 원칙 때문이다.

사람도 모든 정보를 한 번에 받으면 처리하기 어렵다. "이번 업무에 필요한 것만" 주면 집중할 수 있다. 클로드도 마찬가지다.

컨텍스트 윈도우는 유한하다. 모든 규칙을 항상 읽히게 하면:
1. 토큰이 낭비된다 (비용 증가)
2. 중요한 규칙이 다른 규칙에 묻힌다 (attention 분산)
3. 규칙 간 충돌이 생길 수 있다

rules/로 분리하면:
1. 필요한 것만 읽힌다 (토큰 절약)
2. 해당 영역의 규칙에만 집중한다
3. 충돌 가능성이 줄어든다

이 Progressive Disclosure 개념은 Ch.9(Context Engineering)과 Ch.12(Skills)에서도 핵심적으로 등장한다.

## 변형해보기

1. **쉬운 과제**: 자신의 프로젝트에서 가장 규칙이 많은 영역(API? 프론트? 테스트?)을 하나 골라 rules/ 파일로 분리해보자.
2. **어려운 과제**: rules/ 파일 3개를 만들고, 각 경로에서 작업을 시켜서 해당 규칙이 적용되는지 확인해보자.
