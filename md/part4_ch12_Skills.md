# Ch.12 기술 가르치기 — Skills

> **이 챕터를 마치면**: Skill이 무엇인지, 왜 필요한지, 어떻게 만드는지 이해한다. SKILL.md의 Progressive Disclosure 3단계를 알고, 직접 첫 번째 Skill을 만들 수 있다.

---

## Skill이 뭔데?

지금까지 우리는 CLAUDE.md에 규칙을 적고, settings.json에 권한을 정하고, hooks로 자동화를 걸었다. 이걸로 클로드가 "어떻게 일하는지"는 정했다.

그런데 **클로드가 "무엇을 할 줄 아는지"**는 어떻게 정할까?

예를 들어:
- "블로그 글을 항상 이 형식으로 써줘" — 매번 설명해야 하나?
- "코드 리뷰할 때 이 체크리스트를 따라줘" — 매번 복붙해야 하나?
- "프론트엔드 컴포넌트를 만들 때 이 패턴을 써" — 매번 알려줘야 하나?

**Skill**은 이런 반복되는 지식과 절차를 **폴더 하나에 담아두는 것**이다.

비유하면:
- CLAUDE.md = 회사 규칙 (모든 직원이 따르는 것)
- rules/ = 부서별 규칙 (특정 팀만 따르는 것)
- **Skill = 업무 매뉴얼** (특정 작업을 할 때 꺼내 보는 것)

> **💭 생각의 흐름**
>
> **문제** — 클로드에게 "블로그 글 써줘"라고 하면 매번 다른 형식으로 나온다
> **질문** — "어떻게 하면 항상 같은 형식으로 나올까?"
> **시도 1** — CLAUDE.md에 블로그 형식을 적었다 → 다른 작업할 때도 이 규칙이 불필요하게 로딩된다
> **시도 2** — 매번 프롬프트에 형식을 복붙했다 → 귀찮고 실수하기 쉽다
> **발견** — Skill로 만들면 "블로그 글 써줘"라고만 해도 자동으로 형식이 적용된다
> **결론** — 반복되는 작업은 Skill로 패키징한다

## Skill의 구조

Skill은 폴더 하나다. 최소한으로는 파일 하나만 있으면 된다.

```
my-skill/
├── SKILL.md              # 필수 — 이것만 있어도 Skill이다
├── scripts/              # 선택 — 실행할 스크립트
├── references/           # 선택 — 참고 문서
└── assets/               # 선택 — 이미지, 템플릿 등
```

### SKILL.md가 핵심

SKILL.md는 두 부분으로 나뉜다:

**1. YAML 프론트매터** — 메타데이터 (언제 이 Skill을 쓸지)
**2. 마크다운 본문** — 실제 지시사항 (어떻게 할지)

```markdown
---
name: blog-writer
description: 프로젝트 블로그 글을 표준 형식으로 작성한다
triggers:
  - "블로그" 
  - "blog"
  - "포스트"
---

# Blog Writer

## 형식
1. 제목: 한 줄, 40자 이내
2. 요약: 2-3문장
3. 본문: H2로 섹션 구분, 각 섹션 3-5문단
4. 마무리: 핵심 요약 + 다음 글 예고

## 톤
- 친근하지만 전문적
- "~입니다" 존댓말 통일
- 코드 예시는 최소화, 개념 설명 우선

## 참고 파일
- references/style-guide.md 참조
- assets/template.md를 기본 틀로 사용
```

## Progressive Disclosure 3단계

Skill의 핵심 설계 원리는 **Progressive Disclosure(점진적 공개)**다. Ch.9에서 배운 컨텍스트 엔지니어링의 원칙이 여기서도 적용된다.

### 1단계: 프론트매터만 읽는다

클로드가 처음 세션을 시작하면, **모든 Skill의 프론트매터만** 훑어본다.

```yaml
---
name: blog-writer
description: 프로젝트 블로그 글을 표준 형식으로 작성한다
triggers: ["블로그", "blog", "포스트"]
---
```

이 단계에서는 토큰을 거의 안 쓴다. "이런 Skill이 있다"는 것만 안다.

### 2단계: 트리거되면 본문을 읽는다

사용자가 "블로그 글 써줘"라고 하면, `triggers`에 "블로그"가 있으니 이 Skill이 활성화된다. 이때 SKILL.md의 **마크다운 본문**을 읽는다.

### 3단계: 필요하면 연결 파일을 읽는다

본문에서 `references/style-guide.md 참조`라고 써 있으면, 그때서야 그 파일을 읽는다.

```
1단계: 프론트매터만 (모든 Skill)     → 토큰 최소
2단계: 본문 읽기 (트리거된 Skill만)  → 필요한 만큼
3단계: 참조 파일 (본문이 가리킬 때)  → 정말 필요할 때만
```

이게 왜 중요한가? **Attention Budget(주의 예산)**을 아끼기 위해서다. 10개의 Skill을 등록해도, 매 세션에서 10개의 전체 내용을 다 읽지 않는다. 프론트매터만 훑고, 필요한 것만 깊이 읽는다.

## Skill을 어디에 두는가

Skill 폴더는 두 곳에 둘 수 있다:

### 프로젝트 안에 두기 (권장)

```
my-project/
├── .claude/
│   ├── CLAUDE.md
│   └── skills/           ← 여기
│       ├── blog-writer/
│       │   └── SKILL.md
│       └── code-review/
│           └── SKILL.md
└── src/
```

프로젝트와 함께 Git으로 관리된다. 팀원과 공유하기 좋다.

### 글로벌 위치에 두기

```
~/.claude/skills/         ← 여기
├── blog-writer/
│   └── SKILL.md
└── code-review/
    └── SKILL.md
```

모든 프로젝트에서 쓸 수 있다. 개인용 Skill에 적합.

## 3가지 사용 사례

Skill은 크게 세 가지로 나뉜다.

### 1. 문서 및 에셋 생성

"항상 이 형식으로 만들어줘" 류의 작업.

```yaml
---
name: frontend-design
description: React 컴포넌트를 디자인 시스템 규칙에 맞게 생성
triggers: ["컴포넌트 만들어", "component", "UI"]
---
```

예: 블로그 작성, 컴포넌트 생성, 이메일 템플릿, 보고서 형식

### 2. 워크플로우 자동화

"이 절차대로 해줘" 류의 작업.

```yaml
---
name: deploy-checklist
description: 배포 전 체크리스트를 자동으로 실행
triggers: ["배포", "deploy", "release"]
---

# 배포 체크리스트

1. `npm test` 실행 — 전체 통과 확인
2. `npm run build` — 빌드 성공 확인
3. CHANGELOG.md 업데이트 확인
4. 버전 번호 확인
5. 결과 요약 보고
```

예: 배포 체크리스트, PR 리뷰 절차, 리팩터링 순서

### 3. MCP 강화

외부 도구와 연결된 Skill. Ch.13에서 MCP를 배운 뒤 이해할 수 있다.

```yaml
---
name: sentry-review
description: Sentry 에러를 분석하고 코드 수정 제안
triggers: ["sentry", "에러 분석"]
tools: ["mcp__sentry__get_issues"]
---

# Sentry 에러 리뷰

1. Sentry에서 최근 에러 가져오기
2. 스택 트레이스 분석
3. 관련 코드 찾기
4. 수정안 제시
```

지금은 "이런 것도 가능하다" 정도만 알면 된다.

## Skill과 CLAUDE.md의 차이

둘 다 "클로드에게 알려주는 것"인데, 뭐가 다를까?

| | CLAUDE.md | Skill |
|---|---|---|
| **범위** | 항상 적용 | 트리거될 때만 적용 |
| **비유** | 회사 규칙 | 업무 매뉴얼 |
| **토큰** | 매번 읽힘 | 필요할 때만 읽힘 |
| **내용** | 프로젝트 전반 규칙 | 특정 작업의 절차/형식 |
| **예시** | "테스트 통과 전 완료 금지" | "블로그는 이 형식으로" |

**규칙 = CLAUDE.md, 절차 = Skill**로 구분하면 깔끔하다.

핵심 판단 기준: **"이 내용이 모든 작업에 필요한가, 특정 작업에만 필요한가?"**
- 모든 작업 → CLAUDE.md
- 특정 작업 → Skill

## 성공 기준 정의

Skill을 만들었으면, **잘 동작하는지 어떻게 알까?**

### 정량적 기준

- **트리거 정확도**: 관련 키워드를 말했을 때 80% 이상 자동 활성화
- **도구 호출 횟수**: 예상한 도구를 예상한 횟수만큼 호출하는가
- **완료율**: Skill대로 끝까지 수행하는가, 중간에 이탈하는가

### 정성적 기준

- **수정 없이 사용 가능**: 결과물을 사용자가 거의 수정 없이 쓸 수 있는가
- **일관성**: 같은 Skill을 10번 실행하면 비슷한 품질이 나오는가
- **이탈 없음**: Skill의 절차를 건너뛰거나 자기 방식으로 바꾸지 않는가

### 테스트 방법

```
1. Skill을 만든다
2. 트리거 키워드로 3-5번 실행해본다
3. 결과를 비교한다
4. 이탈하거나 품질이 낮으면 SKILL.md를 수정한다
5. 다시 3-5번 실행한다
```

이 반복이 **Skill 개발**이다. 한 번에 완벽하게 만들 수 없다. 프롬프트와 마찬가지로 반복 개선한다.

### 배포

완성된 Skill을 팀이나 다른 프로젝트에서 쓰려면:

| 방법 | 적합한 경우 |
|------|----------|
| **프로젝트 내** `.claude/skills/` | 이 프로젝트에서만 쓸 때. Git에 커밋하면 팀원도 자동 사용 |
| **글로벌** `~/.claude/skills/` | 내 모든 프로젝트에서 쓸 때 |
| **Git 저장소 공유** | 팀 전체가 같은 Skill을 쓸 때. 저장소를 클론 후 `.claude/skills/`에 배치 |

배포 후 확인: Claude Code에서 트리거 키워드를 입력해서 Skill이 정상 동작하는지 테스트한다.

## Skill + MCP 커넥터 패턴

Skill과 MCP는 함께 쓰면 강력하다. 비유로 이해하자.

```
주방 비유:

MCP  = 도구 + 재료
       (칼, 도마, 오븐 + 밀가루, 달걀, 버터)

Skill = 레시피
       ("이 순서로 이 재료를 이 도구로 조리해")
```

MCP가 Sentry에서 에러를 가져오는 **도구**를 제공하고,
Skill이 그 도구를 **어떤 순서로 어떻게 쓰는지** 정의한다.

도구만 있으면 클로드가 아무렇게나 쓴다. 레시피만 있으면 도구가 없어서 못 쓴다. **둘 다 있어야** 일관된 결과가 나온다.

이 패턴은 Ch.13(MCP)을 배운 뒤에 직접 만들어볼 수 있다.

## 파이썬 연결: 반복문 (for/while)

Skill의 핵심은 **반복되는 작업을 한 번 정의해서 계속 쓰는 것**이다. 파이썬에서 이 개념이 바로 **반복문**이다.

### for — "이 목록의 각각에 대해"

```python
# 체크리스트의 각 항목을 확인
checklist = ["변수명", "에러 처리", "테스트", "보안"]

for item in checklist:
    print(f"확인 중: {item}")
```

```
확인 중: 변수명
확인 중: 에러 처리
확인 중: 테스트
확인 중: 보안
```

Skill의 체크리스트를 순서대로 실행하는 것과 같다.

### while — "조건이 맞는 동안 계속"

```python
# Skill 테스트를 3번 반복
test_count = 0
all_passed = False

while test_count < 3 and not all_passed:
    test_count += 1
    result = run_skill_test()  # 가상의 함수
    if result == "pass":
        all_passed = True
    print(f"테스트 {test_count}: {result}")
```

Skill을 만들고 테스트하고 개선하는 반복 과정과 같다.

### range — "N번 반복"

```python
# 5개 파일에 대해 리뷰 Skill 실행
files = ["login.ts", "signup.ts", "auth.ts", "token.ts", "session.ts"]

for i, file in enumerate(files, 1):
    print(f"[{i}/5] 리뷰 중: {file}")
```

**반복문이 프로그래밍에서 하는 일**을 Skill이 클로드에게 한다: 같은 절차를 매번 일관되게 실행.

---

## 왜 이렇게 동작하는가

Skill은 **지식의 캡슐화(encapsulation)**다. 소프트웨어 엔지니어링에서:

- 함수 = 코드의 캡슐화 (같은 로직을 반복 호출)
- 클래스 = 데이터와 로직의 캡슐화
- Skill = **작업 지식의 캡슐화** (같은 절차를 반복 실행)

프로그래밍 경험이 없어도, "자주 하는 일을 매뉴얼로 만들어두면 누가 해도 같은 결과가 나온다"는 건 직관적으로 이해할 수 있다.

## 변형해보기

1. **쉬운 과제**: 자기가 자주 시키는 작업을 하나 골라 SKILL.md를 작성해보자
2. **어려운 과제**: 트리거 키워드를 한국어/영어 둘 다 넣고, 실제로 3번 실행하여 일관성을 확인해보자

## 다음 챕터로

Skill은 "클로드가 할 줄 아는 것"을 정의한다. 하지만 Skill만으로는 **외부 세계에 닿을 수 없다**. Slack에 메시지를 보내거나, 데이터베이스를 조회하거나, Sentry에서 에러를 가져오려면 **연결**이 필요하다.

Ch.13에서는 이 연결을 만드는 **MCP(Model Context Protocol)**를 배운다.

## 이 챕터 핵심 3줄
